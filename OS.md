# 개발 OS (Development OS)

개발 작업을 **일관된 절차**로 수행하기 위한 표준 운영 문서.
Claude는 이 프로젝트에서 개발 요청을 받으면 아래 파이프라인을 순서대로 따른다.

### 목표
- 기존 프로젝트의 구조·컨벤션을 정리한 문서를 만들거나, 있다면 확인 후 갱신한다.
- 요구사항을 분석해 **재사용성**을 고려한 기능 코드와 테스트 코드(단위+통합)를 작성한다.
- 테스트가 실패하면 원인을 분석 후 코드를 수정하고, 모두 통과할 때까지 반복한다.
- 모든 테스트가 검증되면 docs 및 HTTP 문서를 추가하거나 갱신한다.

## 핵심 원칙
- **문서가 먼저(Docs-first)**: 코드를 짜기 전에 구조·컨벤션을 파악/정리한다.
- **재사용 우선(Reuse-first)**: 새로 만들기 전에 기존 코드·패턴을 찾아 재사용한다.
- **테스트로 증명(Test-proven)**: 모든 기능은 단위·통합 테스트로 동작을 증명한다.
- **초록불에서만 전진(Green-gate)**: 테스트가 모두 통과해야 다음 단계로 넘어간다.
- **흔적을 남김(Traceable)**: 각 단계의 산출물(문서·코드·테스트)을 정해진 위치에 남긴다.

## 설계도안 — 전체 파이프라인

```
[1] 코드베이스 파악      [2] 분석 & 개발         [3] 테스트 검증 루프      [4] 문서화
 구조·컨벤션 정리   →    재사용 설계 +      →    실행→실패시 수정→   →   docs + HTTP
 (없으면 생성)           단위+통합 테스트        재실행 (전부 통과)        문서 추가/갱신
      │                      │                       │  ▲                    │
      └──────────────────────┴───────────────────────┘  └─ 실패 시 반복 ──────┘
```

각 단계는 **이전 단계의 완료조건(DoD)을 충족해야** 시작한다.

---

## [1단계] 코드베이스 구조 · 컨벤션 정리

**목적**: 작업 전에 프로젝트의 구조와 규칙을 이해하고, 이를 문서로 고정한다.

**입력**: 현재 저장소 전체.

**작업**
- 구조/컨벤션 문서(`docs/CONVENTIONS.md`)의 존재를 먼저 확인한다.
  - **없으면**: 디렉토리 구조, 네이밍 규칙, 레이어 구분, 테스트 위치/명명, 의존성·빌드 도구, 코드 스타일을 조사해 새로 작성한다.
  - **있으면**: 실제 코드와 대조해 **달라진 부분만** 업데이트한다(불필요한 전면 재작성 금지).
- 기존 패턴(유틸, 공통 모듈, 에러 처리, 테스트 헬퍼)을 목록화해 2단계의 재사용 후보로 남긴다.

**산출물**: `docs/CONVENTIONS.md` (생성 또는 갱신)

**완료조건(DoD)**
- [ ] 컨벤션 문서가 현재 코드 상태와 일치한다.
- [ ] 재사용 가능한 기존 자산이 식별되어 있다.

---

## [2단계] 요구사항 분석 · 개발 · 테스트 작성

**목적**: 요구사항을 재사용성 있게 구현하고, 동작을 증명할 테스트를 함께 작성한다.

**입력**: 사용자 요구사항 + 1단계의 컨벤션/재사용 목록.

**작업**
1. **요구사항 분석**: 입력/출력, 경계 조건, 예외 케이스를 한 문장씩 정의한다. 모호하면 진행 전에 사용자에게 질문한다.
2. **재사용 고려 설계**: 1단계에서 찾은 기존 코드·패턴을 우선 활용한다. 중복 로직은 공통화하고, 새 코드도 다른 곳에서 재사용 가능하도록 분리·일반화한다.
3. **구현**: 1단계 컨벤션을 따른다(네이밍·구조·스타일 일관성).
4. **테스트 작성 (둘 다 필수)**
   - **단위 테스트**: 함수/모듈 단위 로직, 정상·경계·예외 케이스.
   - **통합 테스트**: 모듈 간 연동, 실제 흐름(예: API 요청→응답, DB 연동) 검증.

**산출물**: 기능 코드 + 단위 테스트 + 통합 테스트.

**완료조건(DoD)**
- [ ] 모든 요구사항이 코드로 구현됐다.
- [ ] 각 요구사항에 대응하는 단위·통합 테스트가 존재한다.
- [ ] 기존 자산을 재사용했거나, 재사용 불가 사유가 분명하다.

---

## [3단계] 테스트 검증 루프

**목적**: 작성한 테스트가 **모두 통과**할 때까지 수정-재실행을 반복한다.

**입력**: 2단계의 코드와 테스트.

**작업 (반복)**
1. 전체 테스트를 실행한다.
2. **실패하면**: 로그를 읽고 **원인을 먼저 분석**한다(테스트가 틀렸는지, 코드가 틀렸는지 구분).
3. 원인에 맞게 코드(또는 테스트)를 수정한다.
4. 다시 1번으로 돌아가 재실행한다.
- 같은 실패가 반복되면 추측 수정 대신 가설을 세워 검증한다(로그 추가·격리 실행).

**산출물**: 전부 통과한 테스트 실행 결과.

**완료조건(DoD)**
- [ ] 모든 단위·통합 테스트가 통과(green)한다.
- [ ] 무시(skip)되거나 가짜로 통과하는 테스트가 없다.

> ⚠️ 이 단계의 DoD를 충족하지 못하면 4단계로 넘어가지 않는다.

---

## [4단계] 문서화 (docs + HTTP 문서)

**목적**: 검증된 기능을 사용·유지보수할 수 있도록 문서화한다.

**입력**: 3단계를 통과한 기능.

**작업**
- **docs**: 기능 설명, 사용법, 주요 결정/제약을 `docs/`에 추가하거나 기존 문서를 갱신한다.
- **HTTP 문서**: API가 추가/변경됐다면 엔드포인트, 메서드, 요청/응답 예시, 상태 코드를 문서화한다(예: `docs/http/*.http` 또는 OpenAPI 파일). 이미 있으면 변경분만 업데이트한다.

**산출물**: `docs/` 문서, HTTP/API 문서 (생성 또는 갱신).

**완료조건(DoD)**
- [ ] 새 기능/변경점이 문서에 반영됐다.
- [ ] API 변경이 HTTP 문서에 반영됐다.

---

## 산출물 위치 규약
| 종류 | 위치 |
|---|---|
| 구조·컨벤션 문서 | `docs/CONVENTIONS.md` |
| 일반 문서 | `docs/` |
| HTTP/API 문서 | `docs/http/` (또는 OpenAPI 파일) |
| 테스트 | 프로젝트 컨벤션을 따름(1단계에서 확인) |

## 운영 메모
- 단계 간 전이는 **DoD 체크리스트 통과**가 기준이다. 충족 못 하면 이전 단계로 되돌아간다.
- 각 단계 완료 시 커밋을 권장한다(`/commit` 스킬 활용). 단계별 원자적 커밋이 추적성을 높인다.
- 이 문서(OS.md)와 실제 작업 방식이 어긋나면, 문서를 현실에 맞게 갱신한다.
