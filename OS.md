# OS

> 내 클로드 OS가 **무엇을 할 수 있어야 하는가**를 정의한다. 문서는 명세일 뿐,
> 각 항목은 `.claude/`에 구현해야 동작한다.
>
> 최상위는 두 갈래다.
> **A. 개발 지원** — OS가 *나의 개발을 돕는* 것.
> **B. OS 관리** — *내가 OS를 돌보는* 것(메타).

---

# A. 개발 지원 (coding)

OS가 실제 개발 작업을 돕는 영역. 작업 종류별 **모드**가 중심이고, 모드 무관
**공통 스킬**과 항상 켜진 **기반**이 받친다.

## A-1. 모드 (작업 종류별 워크플로)

### 유지보수 모드 — 만드는 중
기존 코드를 고칠 때. **조사→분석→설계→구현→QA** 순서로 지휘.

| 단계 | 구현 | 종류 | 산출물 | 상태 |
|---|---|---|---|---|
| 0. 지휘 | `maintenance-mode` | skill | 전체 흐름 진행 | ✅ |
| 1. 조사 | `investigate` | agent | 조사 노트 (지라·git 히스토리) | ✅ |
| 2. 분석 | `analyze` | agent | 영향 범위·비기능·테스트 공백 | ✅ |
| 3. 설계 | `design-doc` | skill | 수정 설계 문서 (합의 지점) | ✅ |
| 4. 계획 | `maintenance-mode` | skill | 구현 계획 `docs/plan/<티켓>.md` (순서 있는 단계) | ✅ |
| 5. 구현 | `code-writer` | agent | 계획대로 단계별 TDD | ✅ |
| 6. QA | `code-reviewer` | agent | 통과/보류 판정 (↔5 반복) | ✅ |

> `code-writer`·`code-reviewer`는 모드 중립 공유 에이전트 — 신규 기능 모드도 재사용.

### 신규 기능 모드 — 예정
새 기능을 만들 때. **요구사항→설계→구현→테스트** (세부는 추후 정의).
`interview`는 기술·구현 택1 등 모호 지점에 단독으로 먼저 꽂아 쓴다.

## A-2. 공통 스킬 (모드 무관)

### 코드 작업
| 스킬 | 기능 | 상태 |
|---|---|---|
| `git-commit` | 좋은 커밋 원칙대로 커밋 | ✅ |
| `code-review` | "리뷰해줘" 하면 변경 코드 점검 (단독 요청용) | ✅ |

### 의도·결정
| 스킬 | 기능 | 상태 |
|---|---|---|
| `interview` | 모호한 결정·범위를 하나씩 물어 구체화 (기술·구현 택1, 유지보수 범위 모호 등) | ✅ |

## A-3. 기반 (항상 켜진 것)
| 항목 | 무엇을 | 적용 | 상태 |
|---|---|---|---|
| 프로젝트 규칙 준수 | 코드 프로젝트의 CLAUDE.md에 컨벤션·명령을 두고 그대로 따름 | 프로젝트 CLAUDE.md | ⬜ |
| 커밋 전 자동 점검 | 커밋 직전 포맷팅/린트 자동 실행 | Hook | ⬜ |

> **프로젝트 CLAUDE.md에 반드시 둘 것 — 테스트 명령.**
> `code-writer`(4단계)는 TDD(레드→그린)로 짜지만, *무슨 명령으로 테스트를 돌리는지*는
> 모른다. 그 명령은 언어·빌드도구마다 달라 모드 중립 공용 에이전트에 박을 수 없고,
> 각 코드 프로젝트의 CLAUDE.md가 알려줘야 한다. 이게 없으면 4단계 TDD가 안 돈다.
>
> | 스택 | TDD 루프 명령 (단일 테스트) |
> |---|---|
> | Java + Gradle | `./gradlew test --tests "<클래스명>"` |
> | Java + Maven | `./mvnw test -Dtest=<클래스명>` |
> | JS/TS (Vitest) | `npx vitest run <파일>` |
>
> 전체 실행이 아니라 **단일 테스트 필터**가 핵심 — 루프는 방금 쓴 테스트 하나만
> 빠르게 돌린다. 전체 실행은 커밋 전 마지막 확인용.

# B. OS 관리 (메타)

OS 자신을 운영·측정·유지하는 영역.

## B-1. 운영 스킬
| 스킬 | 기능 | 상태 |
|---|---|---|
| `skill-stat` | 스킬 호출 횟수·랭킹 통계 (OS 자신을 측정) | ✅ |

## B-2. 기반
| 항목 | 무엇을 | 적용 | 상태 |
|---|---|---|---|
| 스킬 호출 카운터 | 스킬 쓸 때마다 횟수 누적 | Hook | ✅ |

## B-3. 구성요소 작성 규칙
만들 때 공통으로 지키는 기준. 핵심 원리: **파일마다 컨텍스트에 올라오는 시점이
다르다. 토큰 비용은 거기서 갈린다.** 항상 로딩되는 곳은 가볍게, 특수한 건
필요할 때만 로딩되게 배치한다(점진적 공개).

| 파일 | 언제 로딩 | 규칙 |
|---|---|---|
| CLAUDE.md | 매 턴 항상 | 제일 비싼 자리. 늘 필요한 규칙만, 짧게. 나머진 스킬로 |
| SKILL.md description | 항상(매칭용) | 트리거를 실제 발화로 구체적으로 — 라우팅 정확도 핵심 |
| SKILL.md 본문 | 부를 때만 | 60줄 이내. 깊은 내용은 REFERENCE.md로 분리 |
| agent 정의 | 서브에이전트 안 | 메인 컨텍스트 안 먹음. 읽기 많은 일은 여기로 |
| hook | 안 올라옴 | 컨텍스트 비용 0 |

### SKILL.md
- **간결하게 — 60줄 이내 목표.** 트리거 시 통째로 로딩되므로, 길면 핵심 지시가
  희석되고 토큰을 매번 먹는다.
- 남길 것: 트리거, 행동을 바꾸는 원칙, 절차, 깨면 안 되는 규칙.
- 뺄 것: 장황한 "왜" 설명, 반복, 당연한 말.
- 깊은 내용은 별도 `REFERENCE.md`로 빼서 필요할 때만 읽게 한다.

### 공통
- "왜"는 사람이 보는 문서(OS.md)·대화에, "어떻게 행동할지"는 실행 파일에.
- 한 사실은 한 곳에만. 복붙 말고 이름으로 참조한다(토큰·정합성).
- 읽기 많은 작업은 agent로 격리해 결론만 메인에 들인다.
- 정확도는 분량이 아니라 구체성에서 나온다. 짧고 구체적인 게 토큰·정확도 둘 다 이긴다.
