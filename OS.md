# 시각 검증 OS — 코드는 사실을 재고, AI 눈이 상황을 보고 동적으로 판정한다

여러 곳에서 쓰이는 **공통 컴포넌트**의 변형들을 한 화면(갤러리)에 모아 렌더하고,
**역할을 둘로 나눠** 검증한다.

- **코드** = 반박 불가한 *사실*을 잰다. 내용 넘침(잘림), 글자 대비 수치, 색 hex, opacity, 크기.
  사실만 재고 **판정은 (넘침 같은 맥락 무관 깨짐을 빼면) 하지 않는다.**
- **AI 눈** = 스크린샷을 직접 보고, 그 사실 수치까지 참고해 **상황에 맞춰 동적으로 판정**한다.
  같은 "대비 1.97" 이라도 *비활성 버튼이면 정상, 본문 글자면 깨짐* 으로 맥락에 따라 다르게 본다.

## 왜 이렇게 가르나

"이게 깨진 거냐"의 기준은 컴포넌트·상황마다 다르다. 고정 임계값(예: 대비 4.5 미만이면 경고)은
숫자만 보고 기계적으로 찍어서 *비활성 흐림*과 *안 읽히는 본문*을 구분 못 한다.
그래서 **판정의 무게중심을 고정 규칙(코드) → 화면을 보는 AI 로 옮긴다.**

- 코드는 정밀 수치를 댄다 (AI 눈이 못 보는 미세 색차·1px 차이까지).
- AI 는 그 수치 + 화면을 보고 맥락으로 판정한다.
- **기준을 주면 그 기준대로**(규칙 검사), **안 주면 알아서**(자유 판단) — 같은 엔진이 둘 다 한다.

| 판정 방식 | 장점 | 약점 |
|---|---|---|
| 고정 규칙(코드) | 항상 똑같음(재현성) | 경직됨, 맥락·컴포넌트 차이 못 봄 |
| 동적 판정(AI 눈) | 맥락·컴포넌트에 적응 | 재현성 다소 낮음 |
| → **둘 다 남긴다** | 코드=흔들리지 않는 사실 / AI=그 위의 맥락 판정 | |

## AI 눈의 판정 경계 — "깨짐"은 절대, "값이 맞나"는 상대

실험으로 확인한 AI 눈의 작동 원리이자 한계. 판정은 두 종류로 갈린다.

| 판정 종류 | 기준(프레임 내 비교 대상) | 잡는 것 |
|---|---|---|
| **절대 — "깨졌나"** | 불필요 (혼자 봐도 자명) | 글자 잘림·넘침, 요소 겹침, 아예 안 읽힘 |
| **상대 — "값이 맞나"** | **필요** | 색, 간격, 크기, 폰트 굵기, 모서리, 그림자 |

- "잘린 글자"는 옆을 안 봐도 잘렸다. 하지만 "이 색·간격이 맞나"는 **비교 대상이 있어야** 답이 나온다.
- 텍스트는 색 기준이 못 된다 — 어떤 배경이든 검정/회색이 당연하라 *배경이 무슨 색이어야 하는지*를 말해주지 않는다. placeholder·배지처럼 **배경과 한 팔레트로 조율돼야 하는 요소**만 기준이 된다.

**진짜 사각지대:** 어긋난 값이 **고립**되고(프레임에 비교 대상 0) **균일**하게 바뀌면 → AI 는 의도된 디자인으로 받아들이고 **못 잡는다.** (예: 미디어·배지·이웃 다 없는 크림빛 카드를 진짜 흰색 카드와 똑같이 "정상" 판정. 1px·미세색이 안 잡히는 것과 같은 결.)

**상대 판정의 구현 = before/after 비교(시각 회귀).** "값이 맞나"는 비교 대상이 필요하므로, **코드 변경 전(기준선)과 후를 다시 촬영해 나란히** 댄다. 이걸 `visual-comparator`(2장 비교 눈)와 `visual-regress`(오케스트레이션 스킬)가 한다 — "의도한 변경 말고 딸려서 바뀐 게 있나"를 잡는다. 여기서도 역할은 둘로 갈린다:
- **픽셀 diff (WHAT)** — `pixelmatch` 로 무엇이 바뀌었는지 정밀하게(1px·미세색) 재고, 바뀐 변형만 추린다. 판정은 안 한다.
- **AI 눈 (MATTER)** — `visual-comparator` 가 그 변화가 *의도 외냐*를 맥락으로 판정한다. 픽셀이 못 보던 맥락을 AI가, AI가 놓친 고립·미세 변화를 픽셀이 메운다(서로 안전망).

**그래서 "변형 모아보기"가 핵심이다.** 실무 회귀는 보통 일부 요소·이웃이 안 건드려져 기준이 남고, 변형을 한 화면에 모으는 행위 자체가 **그 기준을 프레임에 공급**해 사각지대를 메운다. 남는 고립·균일 회귀는 AI 눈이 아니라 **코드/토큰 측정**의 몫(→ 7단계).

**산 증거(probe 변형, `variants/card.ts`):** 같은 크림빛 어긋남(`bgTint`)을 두고 —
`tone-b`(placeholder·배지 있음)는 톤 충돌로 **잡고**, `tone-c`(`hideMedia`로 고립)는 비교 대상이 없어 흰색 대조군 `tone-d`와 똑같이 "정상" 판정해 **못 잡는다.** 갤러리에서 `tone-c`만 정답(주의)과 불일치하는 빨간 칸으로 드러난다.
판정은 손댄 걸 모르는 **별도 블라인드 에이전트**로 받아 편향을 배제하고, 그 결과를 `ai-notes.json`에 기록해야 갤러리에 반영된다.

## 멀티 컴포넌트 (동적 적용)

- 컴포넌트마다 규칙을 따로 짜지 않는다. **레지스트리(`registry.ts`)에 컴포넌트+변형 목록만 등록**하면 된다.
- 갤러리는 `/gallery?c=<키>` 로 대상 컴포넌트를 골라 **범용 렌더**한다.
- 촬영·측정·갤러리 스크립트는 컴포넌트를 모른다 — DOM 의 `[data-variant-id]` 만 보고 동작한다.
- AI 눈 검수는 스크린샷만 보면 되므로 **어떤 컴포넌트든 그대로** 적용된다.

**새 컴포넌트 추가:** ① 변형 목록 작성 → ② 레지스트리 한 줄 등록 → ③ `npm run capture -- <키> && npm run gallery -- <키>`

## 활용하는 클로드 코드 기능

- **Playwright** — 변형별 요소 스크린샷, computed style 측정(`evaluate`)
- **이미지 Read** — 클로드가 스크린샷을 직접 보고 동적 판정 (AI 눈, 핵심)
- **CLAUDE.md** — 프로젝트 컨텍스트와 검증 규칙 정의
- **스킬 / 서브에이전트** — 워크플로우를 역할별로 분리·재사용:
  - 스킬(오케스트레이션): `visual-add`(등록)·`visual-remove`(해제)·`visual-list`(목록·진입)·`visual-check`(전체 검증·1장 절대 판정)·`visual-confidence`(판정 N회 반복·안정성)·`visual-regress`(before/after 시각 회귀)
  - 서브에이전트(전문 일꾼·재사용 "눈") — 둘로 갈림:
    - `visual-judge` — 스샷 **1장** → ok/warn/error **절대** 판정("깨졌나"). **`visual-check`·`visual-confidence` 두 스킬이 공유.**
    - `visual-comparator` — **2장**(before/after) → same/expected/unexpected **상대** 판정("의도 외로 바뀌었나"). `visual-regress` 가 호출. judge 의 짝.
- **훅(hook)** — 자동화:
  - `record-skill-usage.py` (PreToolUse·Skill) — 스킬 호출 통계 기록.
  - `stop-dev-server.sh` (SessionEnd) — 시각 검증이 *자동 기동한* dev 서버만 소유권(PID 마커) 추적해 세션 끝에 정리. 사용자가 직접 띄운 서버는 안 건드린다.

## 진행 단계

> 각 단계는 그 자체로 완성품이다. **어디서 멈춰도 그 시점까지가 동작하는 결과물**이 되도록 만든다.

| 단계 | 목표 | 결과물 |
|------|------|--------|
| 1 | 검증 대상 만들기 | 공통 컴포넌트 + 정상·함정 변형들을 모은 `/gallery` 화면 |
| 2 | 변형 전부 촬영 | 스크립트가 모든 변형을 스샷으로 떨군다 (컴포넌트 무관) |
| 3 | 코드 측정 (사실) | 넘침·대비 수치·색·opacity 측정. 넘침만 코드 깨짐, 나머지는 수치로만 |
| 4 | AI 눈 동적 판정 | 스샷 + 수치를 보고 **맥락으로** 깨짐 식별 (기준 주면 규칙 검사) |
| 5 | 갤러리 보고서 + 자체 채점 | 코드 사실 · AI 판정 · 정답 대비 점수를 한 페이지에 |
| 6 | 멀티 컴포넌트 | 레지스트리로 임의 컴포넌트에 동적 적용 (Card·Button …) |
| 7 | (확장) 기준 도입 | **[구현됨] 기준선 before/after 비교** — `visual-regress`+`visual-comparator`+픽셀 diff. / (남음) Figma 디자인 정답지 비교 |
