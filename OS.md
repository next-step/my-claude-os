# 개발 OS 청사진 (Blueprint)

> 이 문서는 "나만의 클로드 코드 OS"의 **설계도**다.
> 핵심은 **"기능 하나를 개발할 때 항상 따르는 개발 흐름(프로세스)"** 이다.
> AI(클로드)와 협업해 개발할 때, 매번 즉흥적으로 하지 않고 이 흐름을 따른다.
> 살아있는 문서로, 단계가 진행될 때마다 갱신한다.

---

## 1. 비전 — 무엇을 만드는가

같은 개발을 매번 다르게 하면 품질이 들쭉날쭉해진다. **개발 OS는 "기능을 개발하는 방식"을
하나의 정해진 흐름으로 굳히는 개인 운영체제**다. 무엇을 만들지(PRD)부터 어떻게 만들지(Plan),
사람의 검토, 구현, 테스트까지 — 한 기능이 거치는 길을 미리 정해둔다.

지향하는 것:
- **검토 후 구현** — 코드를 짜기 전에 "무엇을/어떻게"를 먼저 정하고 사람이 확인한다.
- **테스트는 필수** — 구현에는 단위/통합 테스트가 반드시 따라온다.
- **재활용 우선** — 새로 짜기 전에 기존 코드를 먼저 찾아 활용한다.
- **안전성** — 되돌리기 어려운 일(푸시·머지)은 명시적 동의가 있을 때만.
- **학습** — 이 OS를 만드는 과정 자체가 "AI와 협업하는 법"을 배우는 실습이다.

---

## 2. 핵심 — 기능 개발 흐름 (The Flow)

기능 하나를 개발할 때 **항상** 아래 순서를 따른다.

```
① PRD 작성  →  ② Plan 작성  →  ③ 사람 검토(게이트)  →  ④ 구현
                                       │                    ├─ 단위/통합 테스트 (필수)
                                  통과해야 진행            └─ 기존 코드 재활용 우선
                                                              ↓
                                                        ⑤ 마무리 (PR → 리뷰 → merge)
```

### ① PRD 작성 — "무엇을, 왜"
- 만들려는 기능이 **무엇을 해결하는지, 왜 필요한지, 어떤 동작을 기대하는지**를 먼저 적는다.
- 코드 얘기 이전에 "문제와 요구사항"에 합의하는 단계다.
- 결과물: PRD 문서 (저장소에 파일로 남긴다 → `docs/prd/` *제안*).

### ② Plan 작성 — "어떻게"
- PRD를 어떻게 구현할지 **설계와 단계**를 적는다: 손댈 파일, 데이터 흐름, 테스트 전략, 위험 요소.
- 이때 **기존 코드를 먼저 조사**해서, 재활용할 수 있는 부분을 Plan에 명시한다. (원칙 3.2)
- 결과물: Plan 문서 (저장소에 파일로 남긴다 → `docs/plan/` *제안*).

### ③ 사람 검토 — 게이트(Gate)
- PRD와 Plan을 **사람이 검토**한다. 이 단계는 **멈춤 지점**이다.
- 문제가 있으면 ①·②로 돌아가 고친다. **통과(승인)하기 전에는 구현을 시작하지 않는다.**
- AI는 여기서 임의로 구현으로 넘어가지 않는다. (가장 중요한 안전장치)

### ④ 구현 — 테스트와 함께, 재활용 우선
- 승인된 Plan을 따라 구현한다.
- **단위 테스트 / 통합 테스트를 반드시 작성한다.** 테스트 없는 구현은 이 OS에서 "미완료"다.
  - 기본 방침: **구현과 함께** 테스트를 작성한다(나란히). *(TDD로 바꾸고 싶으면 이 줄을 수정)*
- **기존 코드 재활용을 우선**한다. 비슷한 기능·유틸이 이미 있으면 새로 만들지 말고 재사용/확장한다.

### ⑤ 마무리 — PR → 리뷰 → merge
- 구현·테스트가 끝나면 커밋(`/commit`)하고, 본인 브랜치에 푸시(`/push`) 후 **PR로 리뷰 요청**한다.
- 리뷰 피드백을 반영해 다시 푸시하고, 승인되면 merge한다. (README의 미션 진행 방식)

> **불변 규칙:** ③ 검토를 통과하지 않으면 ④로 못 넘어간다. ④에는 테스트가 반드시 포함된다.

## 3. 설계 원칙 (Principles)

새 작업을 할 때 이 원칙에 비춰 판단한다.

1. **검토 후 구현.** 무엇을(PRD)·어떻게(Plan)를 정하고 사람이 승인하기 전에는 코드를 쓰지 않는다.
2. **재활용 우선.** 새로 짜기 전에 기존 코드를 먼저 찾는다. 중복 구현을 만들지 않는다.
3. **테스트는 구현의 일부.** 단위/통합 테스트 없이 "구현 완료"라고 하지 않는다.
4. **로컬은 가볍게, 외부는 무겁게.** 커밋은 진행하되 푸시·머지처럼 되돌리기 어려운 일은 확인 후.
5. **모든 산출물은 프로젝트 안에.** PRD·Plan·문서·스킬·훅은 전부 이 저장소 안에 둔다. (CLAUDE.md 규칙 1)
6. **만들면서 가르친다.** 구현뿐 아니라 "왜 이렇게 했는가"를 설명한다. (CLAUDE.md 규칙 2)

## 4. 흐름을 받쳐주는 도구 (Supporting Tooling)

위 개발 흐름을 매끄럽게 돌리기 위한 보조 장치들. (도구는 흐름을 위한 것이지, 그 반대가 아니다.)

| 도구 | 흐름에서의 역할 | 상태 |
|------|----------------|------|
| `/prd` 스킬 | ① 요구사항 — 요구사항 md → PRD 생성 → 검토 루프(확정 게이트) | ✅ |
| `/plan` 스킬 | ② Plan — 확정 PRD → 재활용 조사(1차 스캔 → 갭만 병렬 Explore) → 설계 → 게이트 | ✅ |
| `/feature` 스킬 (런처) | 흐름 안내 — 현재 단계를 스캔해 지도를 보여주고 다음 단계 스킬을 실행 | ✅ |
| `/commit` 스킬 | ⑤ 마무리 — 의미 단위 로컬 커밋 | ✅ |
| `/push` 스킬 | ⑤ 마무리 — 원격 푸시(확인 후) | ✅ |
| `/skill-stat` 스킬 | OS를 어떻게 쓰는지 관측 | ✅ |
| 사용 통계 훅 | 스킬 호출 자동 기록 | ✅ |
| 구현 스킬 (`/impl`?) | ④ 구현 — 승인된 Plan 기반 코드+테스트 | ⬜ 예정 |
| 테스트 가드 | ④ 테스트 누락 방지 | ⬜ 예정 |

> **단계 = 스킬, 흐름 = 런처/규칙.** 흐름의 한 단계는 자체 게이트를 가지므로 단계별 스킬
> (`/prd`, `/plan`)로 쪼갠다 — 책임이 또렷해지고 단독 실행도 된다.
> 흐름을 굴리는 역할은 둘로 나뉜다: **CLAUDE.md 규칙**이 "PRD→Plan→게이트를 따르라"고 항상
> 강제하고, **`/feature` 런처**는 "지금 어디고 다음이 뭔지" 안내하며 다음 단계 스킬을 실행한다.
> `/feature`는 직접 PRD·Plan을 만들지 않는다(런처는 위임만 한다).
> 단계 사이에 진짜 연결 로직(정합성 검사·자동 핸드오프 등)이 생기면 그때 런처를 키운다.

## 5. 로드맵 (Roadmap)

- **1단계: 흐름 정의 + 기반 도구** — 개발 흐름 청사진화, 커밋/푸시/통계 도구. ✅
- **2단계 (현재): 흐름 도구화 (단계별 스킬 분리)** — `/prd`(①)·`/plan`(②)을 전담 스킬로 분리, `/feature`는 단계를 안내·실행하는 **런처**로 축소. ✅
- **3단계: 구현 단계 + 강제 장치** — 구현(④)을 `/impl` 스킬로, 기존 코드 조사 자동화, 테스트 누락 가드(훅).
- **4단계: 자기 개선** — 통계·이력을 근거로 흐름의 병목을 찾아 개선.

---

## 6. 컨벤션 (Conventions)

- **언어:** 한국어. 커밋 메시지도 한국어 + 접두사(`feat : `, `fix : `, `docs : `, 콜론 앞뒤 공백).
- **커밋 서명:** 메시지 끝에 `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`.
- **브랜치/리뷰:** 본인 GitHub 아이디 브랜치 → PR로 리뷰 요청 → 피드백 반영 후 merge (README).
- **파일 위치:** 모든 산출물은 저장소 안. 개인 사용 로그(`skill-usage.log`)는 `.gitignore` 대상.

---

## 7. 용어 (Glossary)

- **PRD (Product Requirements Document):** 만들 기능의 "무엇을·왜". 요구사항 합의 문서.
- **Plan:** PRD를 "어떻게" 구현할지의 설계·단계. 기존 코드 재활용 계획 포함.
- **게이트(Gate):** 다음 단계로 넘어가려면 통과해야 하는 멈춤 지점. (여기서는 사람의 검토.)
- **단위 테스트 / 통합 테스트:** 구현의 정확성을 검증하는 필수 산출물.

---

> 📌 *제안* 표시(예: `docs/prd/` 경로, 테스트 작성 시점)는 기본값일 뿐이다.
> 실제 선호에 맞게 자유롭게 고치고, 고치면 이 청사진도 함께 갱신한다.
