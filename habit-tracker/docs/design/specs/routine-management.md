# 루틴 관리 화면 명세

> 작성일: 2026-06-27 (검토 기반 신규 작성)
> 연관 정책: docs/planning/POLICY.md §4-2, §4-3, §5(루틴 삭제: Soft delete만 허용)

---

## 목적

자신의 루틴을 추가하고 관리하는 화면. 추가는 가볍게, 삭제는 신중하게, 성공 피드백은 즉각적으로.

---

## 현재 구현 검토 — 발견된 문제

### 문제 1: 브라우저 confirm 다이얼로그 — 디자인 시스템 통제 밖

**현재:** `onsubmit="return confirm('이 루틴을 삭제할까요?')"`

- 브라우저 네이티브 다이얼로그는 각 OS/브라우저마다 UI가 다르다 (Chrome, Safari, Firefox 모두 다름)
- 디자인 시스템의 색상, 폰트, 모서리 규칙이 일절 적용되지 않는다
- 토스, 당근 등 모바일 우선 앱에서 절대 사용하지 않는 패턴
- POLICY §4-3에서 "삭제 → 그만하기"로 용어 변경을 명시

**개선:** 인라인 2단계 확인 패턴

### 문제 2: 삭제 버튼 레이블 — POLICY §4-3 위반

POLICY §4-3 네이밍/톤 정책: "삭제" → "그만하기"

현재 버튼 내 아이콘(`bi-trash`)만 있고 텍스트는 없지만,
`aria-label`, tooltip, 확인 문구에서 "삭제"라는 표현을 쓰지 않아야 한다.

### 문제 3: 추가 성공 피드백 없음

루틴 추가 후 PRG redirect로 페이지가 새로고침되는데, 어떤 루틴이 방금 추가됐는지 피드백이 없다.
사용자는 "추가가 됐나?" 하고 목록을 직접 확인해야 한다.

### 문제 4: 빈 상태 메시지 톤 — POLICY §4-2 위반

**현재:** `"아직 루틴이 없습니다. 위에서 추가해보세요!"`

POLICY §4-2: "따뜻한 메시지 + 행동 유도 버튼 필수"
POLICY §4-3 예시: `"아직 루틴이 없어요. 첫 루틴을 만들어볼까요? 🌱"`

현재 메시지는 "위에서 추가해보세요"로 UI 내 위치(폼이 위에 있음)를 안내하는데,
이것은 레이아웃에 의존한 안내로 접근성·반응형 변경 시 의미가 깨진다.

## 사용자 플로우

### 플로우 A — 루틴 추가

```
루틴 관리 진입
  → 폼 영역에서 루틴 이름 입력
  → 설명 입력 (선택)
  → 요일 선택 (버튼)
  → "추가하기" 버튼 탭
  → POST /routines → redirect
  → 페이지 재로드: 새 루틴이 목록 맨 위 or 맨 아래 + 플래시 성공 메시지 표시
```

### 플로우 B — 루틴 삭제 (인라인 2단계)

```
루틴 아이템의 "그만하기" 버튼 탭
  → 해당 아이템: "정말 그만할까요?" + [취소] [그만하기] 버튼으로 인라인 전환
  → [취소] 탭 → 원래 상태 복귀
  → [그만하기] 탭 → POST /routines/{id}/delete → redirect
  → 목록에서 해당 루틴 사라짐
```

### 플로우 C — 유효성 오류

```
루틴 이름 없이 "추가하기" 탭
  → 서버 유효성 검사 실패
  → 오류 메시지: "루틴 이름을 입력해주세요" (현재 Thymeleaf th:errors 방식 유지)
  → 폼 포커스 유지
```

### 화면 2 — 인라인 삭제 확인 (2단계)

```
기본 상태:
┌──────────────────────────────────────────┐
│  아침 스트레칭              [그만하기]    │
│  월 화 수 목 금                          │
└──────────────────────────────────────────┘

"그만하기" 탭 후:
┌──────────────────────────────────────────┐  ← 배경: --color-danger-light (#FEF2F2)
│  아침 스트레칭                           │
│  월 화 수 목 금                          │
│  정말 그만할까요?      [취소] [그만하기] │  ← 하단에 확인 행 추가
└──────────────────────────────────────────┘
```

**구현 방식 (JS 인라인, 서버 폼 유지):**

```html
<!-- 목록 아이템 -->
<div class="list-group-item routine-item" id="routine-${r.id}">
  <div class="routine-item-main d-flex align-items-center">
    <!-- 루틴 정보 -->
    <div class="flex-grow-1">...</div>
    <!-- 그만하기 버튼 -->
    <button type="button"
            class="btn btn-outline-secondary btn-sm"
            onclick="showDeleteConfirm(${r.id})"
            aria-label="${r.name} 그만하기">
      그만하기
    </button>
  </div>
  <!-- 인라인 확인 영역 (초기 hidden) -->
  <div class="routine-delete-confirm d-none" id="confirm-${r.id}">
    <span class="text-danger me-2" style="font-size: 14px;">정말 그만할까요?</span>
    <button type="button" class="btn btn-sm btn-outline-secondary me-1"
            onclick="hideDeleteConfirm(${r.id})">취소</button>
    <form th:action="@{/routines/{id}/delete(id=${r.id})}" method="post" class="d-inline">
      <button type="submit" class="btn btn-sm btn-danger">그만하기</button>
    </form>
  </div>
</div>
```

- `showDeleteConfirm(id)`: `.routine-delete-confirm.d-none` 제거, 배경색 --danger-light 추가
- `hideDeleteConfirm(id)`: 원복
- 최종 제출은 서버 폼 → PRG 패턴 유지 (JS disable 없어도 동작)

- 3초 후 JS로 자동 제거 (또는 닫기 버튼)

## 텍스트 가이드

### 삭제 관련

- 버튼: `"그만하기"` (POLICY §4-3, "삭제" 금지)
- 확인 문구: `"정말 그만할까요?"` (confirm 다이얼로그의 "이 루틴을 삭제할까요?" 대체)
- 최종 확인 버튼: `"그만하기"` (위험 액션이므로 btn-danger)
- 취소 버튼: `"취소"` (btn-outline-secondary)

### 성공 메시지

- 형식: `"[루틴 이름] 루틴이 추가됐어요!"`
- 예시: `"아침 스트레칭 루틴이 추가됐어요!"`

### 유효성 오류 (현행 유지)

- 이름 미입력: Thymeleaf `th:errors="*{name}"` (Bean Validation 메시지)

### 빈 상태

- 메시지: `"아직 루틴이 없어요. 첫 루틴을 만들어볼까요?"`

## 반응형 규칙

### 모바일 (< 576px)

- 요일 버튼: 현행 `d-flex flex-wrap gap-2` 유지 — 줄바꿈 허용
- "그만하기" 버튼: 텍스트 유지 (아이콘만으로 대체 금지 — 의도 불명확)
- 삭제 확인 행: 루틴 아이템 카드 내 하단에 full-width로 배치
- 성공 플래시 메시지: 풀 너비, container 안에서

### 데스크탑 (> 768px)

- 폼과 목록을 2열로 분리하는 방안은 Phase 2 이후 검토 (현재는 단일 컬럼 유지)
