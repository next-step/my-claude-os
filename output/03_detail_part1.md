# 상세 해설: LiveEdit — AR-지향 마스크 캐시(AR-Oriented Mask Cache) 메커니즘

> 이 문서는 LiveEdit 논문의 핵심 개념 중 **"AR-지향 마스크 캐시" 단 하나**만 떼어 내어, 고등학생도 따라올 수 있게 직관 → 비유 → 단계별 전개 → 예시 순으로 풀어 쓴 상세 해설(part 1)입니다. (3단계 증류 파이프라인 등 다른 개념은 별도 part에서 다룹니다.)

---

## 사전 지식 (꼭 알아야 할 개념)

- **스트리밍 비디오 편집**: 영상 전체를 다 받아놓고 한꺼번에 고치는 게 아니라, 프레임이 들어오는 즉시 실시간으로 고쳐서 내보내는 방식. "라이브 방송에 필터를 거는 것"이라 보면 됩니다.
- **토큰(token)**: 확산 트랜스포머(Diffusion Transformer)는 이미지/프레임을 작은 조각(패치)들로 잘라 각 조각을 하나의 "토큰"이라는 벡터로 다룹니다. 즉 한 프레임 = 여러 개의 토큰 묶음.
- **latent(잠재 표현)**: 원본 픽셀을 그대로 쓰지 않고, 인코더로 압축한 작은 표현. LiveEdit은 latent를 3프레임씩 묶어 한 덩어리(chunk)로 처리합니다.
- **Self-Attention(자기-어텐션)**: 토큰들이 서로를 "참조"하며 정보를 주고받는 트랜스포머의 핵심 연산. 토큰 수가 N이면 비용이 대략 N×N에 비례해서, 토큰이 많으면 매우 비쌉니다.
- **FFN(Feed-Forward Network)**: 어텐션 뒤에 붙어 각 토큰을 독립적으로 한 번 더 변환하는 작은 신경망.
- **캐시(cache)**: 한 번 계산한 결과를 저장해 두었다가, 똑같은 게 또 필요하면 다시 계산하지 않고 꺼내 쓰는 것. (브라우저 캐시와 같은 개념)
- **L₂ 거리**: 두 벡터의 차이를 제곱해 더한 뒤 제곱근을 씌운 값. 쉽게 말해 "두 점 사이의 직선 거리". 값이 크면 많이 달라졌다는 뜻.
- **τ(타우, threshold)**: "이 정도 이상 달라지면 편집된 걸로 본다"를 정하는 기준선(임계값).
- **AR(Auto-Regressive, 자기회귀) / 인과적(causal)**: 과거와 현재만 보고 미래는 보지 않으며, 한 덩어리씩 순서대로 생성해 나가는 방식. 스트리밍에 꼭 필요한 성질.

## 핵심 개념 — AR-지향 마스크 캐시 (AR-Oriented Mask Cache)

### 직관 (결국 무엇을 하려는가)

영상 편집에서 사용자가 바꾸고 싶은 부분은 보통 화면의 **일부**입니다. 예를 들어 "강아지를 고양이로 바꿔줘"라고 하면 강아지가 있는 영역만 바뀌면 되고, 뒤의 하늘·건물·바닥 같은 **배경은 그대로** 두어야 합니다. 그런데 일반 모델은 매 프레임마다 화면의 **모든 토큰**을 똑같이 비싼 연산으로 다시 계산합니다. 안 바뀌는 배경까지 매번 처음부터 새로 그리는 셈이라 (1) 느리고, (2) 매번 미세하게 다르게 그려져 배경이 깜빡이거나(flickering) 구조가 무너집니다.

AR-지향 마스크 캐시의 핵심 아이디어는 간단합니다. **"실제로 편집이 일어나는 영역만 새로 계산하고, 변하지 않는 영역은 직전에 계산해 둔 결과(캐시)를 그대로 재사용하자."** 어디가 편집 영역인지는 직전 chunk에서 "원본과 편집본이 얼마나 달라졌나"를 L₂ 거리로 재서 자동으로 판별합니다. 이렇게 하면 약 70%의 중복 토큰 계산을 건너뛰어 **빨라지고(12.66 FPS)**, 배경은 캐시값을 그대로 쓰니 **흔들리지 않아 일관성도 좋아집니다.** "AR-지향"이라는 말은, 이 캐시가 과거 chunk의 결과를 다음 chunk가 이어받는 자기회귀(스트리밍) 구조에 맞춰 설계되었다는 뜻입니다.

### 비유

> 아래는 직관적 이해를 위한 비유이며, 논문 본문에 그대로 나오는 표현이 아닙니다.

벽화를 그리는 화가를 떠올려 보세요. 의뢰인이 "벽 가운데 그려진 강아지만 고양이로 바꿔주세요"라고 했습니다. 어리석은 화가는 매번 벽 전체(하늘, 나무, 울타리까지)를 처음부터 다시 칠합니다 — 느리고, 다시 칠할 때마다 하늘색이 조금씩 달라져 얼룩집니다. 똑똑한 화가는 **강아지 부분만 마스킹 테이프로 표시**해 그 영역만 새로 칠하고, 나머지는 어제 칠해 굳은 그대로 둡니다. 빠르고, 배경은 전혀 흔들리지 않습니다.

여기서 "마스킹 테이프로 표시한 영역"이 곧 **편집 마스크**, "어제 칠해 굳은 부분을 그대로 두는 것"이 **캐시 재사용**입니다. 그리고 어디에 테이프를 붙일지는 "어제 그림과 오늘 요청이 얼마나 다른가"(L₂ 거리)를 보고 정합니다.

### 단계별 전개

여기서 기호 정의:
- `z_src^{k-1}` : 직전(k−1번째) chunk의 **원본(source) latent** — 편집하기 전 영상.
- `z_edit^{k-1}` : 직전 chunk의 **편집된(edited) latent** — 모델이 고쳐 놓은 결과.
- `‖ · ‖₂` : L₂ 거리(앞서 설명한 "차이의 크기").
- `τ` : 임계값. 이보다 차이가 크면 "편집됨"으로 판정.
- `M` : 토큰별 이진(0/1) 마스크. 1 = 편집 영역(새로 계산), 0 = 정적 영역(캐시 재사용).

1. **차이 측정**: 직전 chunk에서 각 토큰 위치마다 원본과 편집본의 차이를 잰다 —
   `d = ‖ z_edit^{k-1} − z_src^{k-1} ‖₂`.
   → 이 토큰이 직전에 실제로 "고쳐졌는지"를 수치로 본다.

2. **마스크 만들기(이진화)**: 차이가 임계값을 넘으면 편집 영역으로 표시한다 —
   `M = 1 (편집)  if d > τ ,  else  M = 0 (정적)`.
   → 화면을 "고쳐야 할 곳"과 "그대로 둘 곳"으로 가른다.

3. **토큰 가지치기(prune)**: 다음 chunk를 처리할 때 `M = 0`인 토큰(약 70%)은 **Self-Attention 연산에서 제외**한다.
   → 비싼 N×N 어텐션에 들어가는 토큰 수 N이 확 줄어 계산이 빨라진다.

4. **캐시 재사용**: `M = 0` 토큰의 출력은 새로 계산하지 않고, **직전에 저장해 둔 중간 feature(캐시값)를 그대로 가져다 채운다.**
   → 배경이 매번 새로 그려지지 않으므로 깜빡임이 사라지고 시간적 일관성이 유지된다.

5. **적용 위치 제한**: 이 캐시는 **Self-Attention 층에만** 적용하고, **FFN에는 적용하지 않는다.**
   → 논문 ablation에 따르면 FFN까지 캐시하면 고주파(미세 디테일)에 민감해 심한 열화가 생기기 때문. 그래서 일부러 self-attention에만 건다.

결과적으로 프레임당 약 79ms, 12.66 FPS 실시간 처리를 달성한다. (`d`, `M` 계산 자체는 가볍고, 절약되는 어텐션 비용이 훨씬 크다.)

### 예시 (작은 토이 입력으로 따라가기)

화면을 아주 단순하게 **토큰 6개짜리 1차원 줄**이라고 합시다. (실제로는 한 프레임에 수백~수천 개지만, 이해를 위해 6개로 축소.)

직전 chunk에서 측정한 원본/편집본 latent 값(설명용 가상의 숫자):

| 토큰 위치 | z_src (원본) | z_edit (편집본) | 차이 d = ‖edit−src‖₂ |
|:--:|:--:|:--:|:--:|
| 1 (하늘) | 2.0 | 2.0 | 0.0 |
| 2 (하늘) | 2.1 | 2.1 | 0.0 |
| 3 (강아지→고양이) | 5.0 | 9.0 | 4.0 |
| 4 (강아지→고양이) | 4.0 | 8.5 | 4.5 |
| 5 (바닥) | 1.0 | 1.05 | 0.05 |
| 6 (바닥) | 1.0 | 1.0 | 0.0 |

임계값을 `τ = 0.5`로 두면:

- 토큰 3, 4 → d(4.0, 4.5) > 0.5 → **M = 1 (편집 영역)** → 다음 chunk에서 새로 계산.
- 토큰 1, 2, 5, 6 → d(0.0~0.05) ≤ 0.5 → **M = 0 (정적 영역)** → 캐시값 재사용.

마스크: `M = [0, 0, 1, 1, 0, 0]`

→ 6개 중 새로 계산하는 토큰은 2개뿐. **4/6 ≈ 67%를 가지치기**(논문에서 말한 "약 70% pruning"과 같은 맥락). Self-Attention은 이제 6개가 아니라 사실상 편집 토큰 중심으로만 돌아 비용이 크게 줄고, 하늘·바닥(배경)은 직전 값 그대로라 다음 프레임에서도 흔들리지 않습니다.

만약 `τ`를 너무 크게(예: 5.0) 잡으면 토큰 3,4의 차이(4.0, 4.5)마저 임계값을 못 넘어 **편집해야 할 영역까지 캐시로 덮어버려** 편집이 안 먹습니다. 반대로 너무 작게(예: 0.01) 잡으면 토큰 5의 미세한 0.05까지 편집으로 잡아 **불필요한 계산이 늘어** 느려집니다. 그래서 논문도 "콘텐츠에 따라 τ의 세심한 튜닝이 필요하다"고 한계로 언급합니다.

## 전체 흐름 한눈에 보기 (텍스트 파이프라인)

```
직전 chunk (k-1)
   z_src^{k-1}  ─┐
                 ├─▶ d = ‖z_edit − z_src‖₂  (토큰별 차이 측정)
   z_edit^{k-1} ─┘
                       │
                       ▼
              d > τ ?  (임계값 비교)
              ┌────────┴─────────┐
         예(편집,M=1)        아니오(정적,M=0)
              │                    │
              ▼                    ▼
   다음 chunk(k)에서       직전 캐시 feature
   Self-Attention 새로 계산   그대로 재사용 (계산 skip)
              │                    │
              └────────┬───────────┘
                       ▼
        합쳐서 다음 chunk 출력 구성
   (≈70% 토큰 prune → 79ms/프레임, 12.66 FPS)
   ※ 캐시는 Self-Attention 층에만, FFN에는 미적용
```

## 자주 하는 오해 (FAQ)

- **Q. 마스크를 "지금 프레임"에서 계산하나요?**
  아니요. **직전 chunk(k−1)**의 원본/편집본 차이로 마스크를 만들어 **다음 chunk(k)**에 적용합니다. 미래를 보지 않는 인과적(AR) 구조라서 이렇게 한 박자 이전 정보를 씁니다.

- **Q. 캐시를 쓰면 편집 품질이 떨어지지 않나요?**
  편집이 일어나는 영역(M=1)은 캐시를 쓰지 않고 항상 새로 계산하므로 편집 자체는 정상 반영됩니다. 캐시는 "안 바뀌는 배경"에만 적용되어 오히려 배경 일관성(BC 0.956)을 **높여** 줍니다.

- **Q. 왜 FFN에는 캐시를 안 거나요?**
  논문 ablation 결과, FFN까지 캐시하면 고주파(미세 디테일)에 민감해 화질이 심하게 열화됩니다. 그래서 Self-Attention 층에만 적용하는 것이 최적입니다.

- **Q. 70% pruning이면 화면의 70%가 안 바뀐다는 뜻인가요?**
  대략 그런 의미입니다. 일반적인 편집에서는 화면 대부분이 배경(정적)이고 실제 편집 영역은 일부이므로, 평균적으로 약 70%의 토큰이 캐시 재사용 대상이 되어 연산을 건너뛸 수 있다는 것입니다. 콘텐츠에 따라 이 비율은 달라집니다.

- **Q. "AR-지향"이 정확히 무슨 뜻인가요?**
  과거→현재로 한 chunk씩 순서대로 생성하는 자기회귀(스트리밍) 추론 구조에 맞춰, 직전 결과를 다음에 이어받도록 캐시를 설계했다는 의미입니다. 미래 프레임을 미리 보는 양방향 방식에는 쓸 수 없는, 실시간 스트리밍 전용 설계입니다.
