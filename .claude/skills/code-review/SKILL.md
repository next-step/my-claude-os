# Code-review 스킬

변경한 코드를 사람 리뷰어처럼 점검한다. 새로 추가/수정한 부분만 보고,
근거 없이 트집 잡지 않는다.

## 원칙

- **변경분만 본다.** `git diff`에 나온 것만. 기존 코드 전체를 감사하지 않는다.
- **근거를 댄다.** "왜 문제인지"와 "어떻게 고칠지"를 같이 적는다. 막연한 지적 금지.
- **심각도를 나눈다.** 꼭 고칠 것과 취향 차이를 섞지 않는다.

## 절차

1. 변경분을 가져온다.
   ```bash
   git diff HEAD            # staged + unstaged
   git diff <base>...HEAD   # 브랜치 전체를 볼 땐 base 지정
   ```
2. 아래 관점으로 훑는다.
   - 정확성: 버그, 빠진 예외 처리, 경계값, null
   - 설계·가독성: 중복, 과한 복잡도, 이름, 책임 분리
   - 비기능: 성능(반복문 내 비용, N+1), 보안(입력 검증, 비밀값 노출)
   - 테스트: 변경에 맞는 테스트가 있나
3. 지적마다 `파일:라인`을 단다.

## 산출물

```
## 리뷰 결과

### 꼭 고칠 것
- 파일:라인 — (문제) → (고칠 방향)

### 고려해볼 것
- 파일:라인 — (제안)

### 좋은 점
- (잘한 부분 짧게)
```

- 지적이 없으면 "문제 없음"이라고 분명히 말한다. 억지로 만들지 않는다.
- 유지보수 QA로 호출됐으면, 설계 문서의 의도대로 고쳐졌는지도 함께 확인한다.
