# Maintenance Mode (오케스트레이터)

유지보수는 여러 단계의 묶음이다. 이 스킬은 직접 다 하지 않고 단계마다 알맞은
도구에 맡긴다. 읽기·쓰기·검증이 많은 단계는 **서브에이전트**에 위임해 중간
출력으로 메인 컨텍스트가 더러워지지 않게 한다. 구현·리뷰 agent는 모드 중립이라
신규 기능 모드와도 공유한다.

## 원칙

- 이해가 수정보다 먼저. 조사·분석 없이 코드를 건드리지 않는다.
- 격리 이득이 큰 단계(조사·분석·구현·QA)는 agent, 사용자 합의가 필요한 설계는 skill.
- 설계 문서를 사용자에게 보여주고 합의받기 전엔 구현하지 않는다.
- 비자명한 구현은 계획(`docs/plan/<티켓>.md`)을 먼저 세우고 단계별로. 계획 없이 구현하지 않는다.

## 흐름

| 단계 | 위임 대상 | 종류 | 산출물 |
|---|---|---|---|
| 1. 조사 | `investigate` | agent | 조사 노트 |
| 2. 분석 | `analyze` | agent | 분석 노트 |
| 3. 설계 | `design-doc` | skill | 설계 문서 (합의 지점) |
| 4. 계획 | (이 스킬) | — | 구현 계획 `docs/plan/<티켓>.md` (순서 있는 단계) |
| 5. 구현 | `code-writer` | agent | 계획대로 단계별 TDD |
| 6. QA | `code-reviewer` | agent | 통과/보류 판정 (↔5 반복) |

## 진행 방식

1. Agent 도구로 `investigate`를 띄워 조사 노트를 받는다(티켓 → 커밋 히스토리로 범위를 좁힘).
   - 남은 "미해결 의문"이 **범위를 크게 가르면** `interview`로 사용자와 범위를 확정하고 다음으로.
   - **사소하면** 최소 범위로 가정하고, 그 가정을 명시한 채 진행한다.
2. 조사 노트를 넘겨 `analyze` 서브에이전트를 띄워 분석 노트를 받는다.
3. 두 노트를 입력으로 `design-doc` 스킬을 실행해 설계 문서를 쓴다. **여기서
   멈추고 사용자 합의를 받는다.**
4. 합의되면 **구현 계획**을 세운다: 설계를 순서 있는 작은 단계(구조→구현→연결)로 쪼개
   `docs/plan/<티켓>.md`에 쓴다. **계획 없이 구현으로 넘어가지 않는다.** 단, 비자명한
   작업만 — 한 파일 국소 수정이면 설계가 곧 계획이라 생략한다.
5. 계획의 단계를 **순서대로 한 단계씩** `code-writer`로 구현한다(설계·노트·해당 단계를
   프롬프트에 실어 TDD). 단계마다 테스트 초록을 확인하고 계획을 체크한 뒤 다음 단계로.
   (단계=커밋 하나로 두면 커밋 순서가 계획 순서와 맞는다)
6. 구현 노트를 넘겨 `code-reviewer` 서브에이전트로 QA 판정을 받는다.
   - **보류**면 리뷰 지적을 붙여 `code-writer`를 다시 띄운다. 통과까지 5·6 반복.
   - 같은 지적이 안 풀리면 멈추고 사용자에 알린다(무한 루프 방지).
7. 커밋은 사용자 요청 시 `git-commit` 스킬로 넘긴다.

> 각 서브에이전트는 문맥이 없다. 호출할 때 이전 단계 노트를 프롬프트에 실어 넘긴다.
> 각 단계 산출물은 다음 단계의 입력이다. 단계를 건너뛰지 않는다.
