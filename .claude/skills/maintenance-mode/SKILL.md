# Maintenance Mode (오케스트레이터)

유지보수는 여러 단계의 묶음이다. 이 스킬은 직접 다 하지 않고 단계마다 알맞은
도구에 맡긴다. 특히 읽기 많은 조사·분석은 **서브에이전트**에 위임해 중간
출력으로 메인 컨텍스트가 더러워지지 않게 한다.

## 원칙

- 이해가 수정보다 먼저. 조사·분석 없이 코드를 건드리지 않는다.
- 읽기 많은 단계는 agent, 판단·작성 단계는 skill.
- 설계 문서를 사용자에게 보여주고 합의받기 전엔 구현하지 않는다.

## 흐름

| 단계 | 위임 대상 | 종류 | 산출물 |
|---|---|---|---|
| 1. 조사 | `investigate` | agent | 조사 노트 |
| 2. 분석 | `analyze` | agent | 분석 노트 |
| 3. 설계 | `design-doc` | skill | 설계 문서 (합의 지점) |
| 4. 구현 | (직접) → `git-commit` | skill | 최소 수정 + 테스트 |
| 5. QA | `code-review` + 테스트 실행 | skill | 검증 |

## 진행 방식

1. Agent 도구로 `investigate` 서브에이전트를 띄워 조사 노트를 받는다. 티켓이
   모호하면 멈추고 사용자에 확인.
2. 조사 노트를 넘겨 `analyze` 서브에이전트를 띄워 분석 노트를 받는다.
3. 두 노트를 입력으로 `design-doc` 스킬을 실행해 설계 문서를 쓴다. **여기서
   멈추고 사용자 합의를 받는다.**
4. 합의되면 최소 수정 + 테스트 작성 → 검증 → `code-review` 스킬로 QA.
5. 커밋은 사용자 요청 시 `git-commit` 스킬로 넘긴다.

> 각 단계 산출물은 다음 단계의 입력이다. 단계를 건너뛰지 않는다.
