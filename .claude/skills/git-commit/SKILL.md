# Git Commit 스킬

변경사항을 **좋은 커밋(Good Commit)** 원칙에 맞춰 커밋한다.

## 이 스킬의 범위

이 스킬은 **"잘 커밋하는 일"만** 한다.

- **빌드/테스트 검증은 이 스킬의 책임이 아니다.** 변경사항은 이미 검증된(빌드·테스트 통과) 상태라고 **가정한다.** 검증은 코드를 작성·수정한 앞선 작업 단계나 CI가 책임진다.
- 따라서 이 스킬은 빌드를 돌리거나 `git stash`로 working tree를 휘젓지 않는다. 깨끗하게 스테이징하고 커밋하는 데 집중한다.
- 만약 변경이 명백히 미완성/깨진 상태로 보이면(예: 빈 함수, TODO만 가득) 커밋하지 말고 사용자에게 먼저 알린다.

## 핵심 규칙 (반드시 지킬 것)

1. **Co-authored-by에 AI/Claude 이름을 절대 넣지 않는다.** 커밋 메시지 끝에 어떤 `Co-Authored-By:` 트레일러도 추가하지 않는다.
2. **타입 접두사를 쓰지 않는다.** `feat:`, `fix:`, `chore:`, `docs:` 같은 Conventional Commits 접두사를 붙이지 않는다.
3. **사용자가 명시적으로 요청할 때만 커밋한다.** 묻지 않았는데 먼저 커밋하지 않는다.

## 커밋 절차

### 1단계: 현재 상태 파악 (병렬로 실행)

```bash
git status          # 어떤 파일이 변경/추가/삭제됐는지
git diff HEAD       # 실제 변경 내용 (staged + unstaged)
git log --oneline -10   # 기존 커밋 메시지 스타일/언어 참고
```

- 기존 커밋들이 한국어면 한국어로, 영어면 영어로 메시지를 맞춘다.
- 현재 브랜치가 `main`/`master`이고 보호된 작업이면, 새 브랜치를 만들지 검토한다.

### 2단계: 변경사항을 논리적 단위로 묶기

- 서로 다른 목적의 변경이 섞여 있으면 나눠서 여러 번 커밋한다 (atomic commit).
- 관련된 변경만 골라 `git add <파일들>`로 스테이징한다. 무의미한 임시파일/빌드산출물은 제외한다.
- `git add .`로 무차별 스테이징하지 말고, `git status`로 의도한 파일만 들어갔는지 확인한다.

> 검증(빌드/테스트)은 하지 않는다 — 이미 통과한 상태라고 가정한다. (위 "이 스킬의 범위" 참고)

### 3단계: 좋은 커밋 메시지 작성

**제목 (첫 줄)**
- 50자 내외, 무엇을 했는지 명령형/요약형으로.
- 끝에 마침표 없음.
- "왜"가 제목만으로 안 드러나면 본문에서 설명.

**본문 (필요할 때만)**
- 제목과 한 줄 띄우고 작성.
- **무엇을 바꿨는지가 아니라 "왜" 바꿨는지**를 설명한다 (변경 내용은 diff가 이미 보여준다).
- 한 줄 72자 내외로 줄바꿈.

커밋은 heredoc으로 작성한다:

```bash
git commit -m "$(cat <<'EOF'
README에 설치 방법과 사용 예시 추가

신규 사용자가 클론 직후 막히는 일이 잦아 진입 장벽을 낮춤.
EOF
)"
```

> 위 메시지에 `Co-Authored-By` 트레일러를 **붙이지 않는다.**

### 4단계: 결과 확인

```bash
git log -1 --stat   # 방금 만든 커밋 확인
```

푸시는 사용자가 요청했을 때만 한다:

```bash
git push            # 또는 git push -u origin <branch>
```

## 좋은 커밋 메시지 예시

좋은 예 ✅
```
을지로 점심 맛집 데이터 스키마 정의

추천 정렬에 거리·가격대·대기시간이 필요해 필드를 미리 확정함.
```

나쁜 예 ❌
```
feat: update            # 타입 접두사 + 무엇을 했는지 불명확
수정함                   # 너무 모호함
asdf                    # 의미 없음
```

## 주의

- `git config user.name`/`user.email`을 임의로 바꾸지 않는다. (작성자는 사용자 본인)
- `git add .`로 무차별 스테이징하기 전에 `git status`로 의도치 않은 파일이 포함되는지 확인한다.
- 커밋을 되돌리기 어려운 작업(force push, history 변경)은 반드시 사용자에게 먼저 확인한다.
