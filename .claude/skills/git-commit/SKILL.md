# GitHub 커밋 스킬

변경 사항을 살펴보고 의미 있는 커밋 메시지로 GitHub에 커밋·푸시한다.

## 사용 시점
- "커밋해줘", "깃에 올려줘", "푸시해줘", "GitHub에 반영해줘" 같은 요청을 받았을 때

## 절차

### 1. 현재 상태 파악 (먼저 병렬로 확인)
- `git status` — 무엇이 변경/추가/삭제됐는지
- `git diff` (스테이징 안 된 변경) 와 `git diff --staged` (스테이징된 변경) — 실제 변경 내용
- `git branch --show-current` — 현재 브랜치 확인
- `git log --oneline -5` — 이 저장소의 기존 커밋 메시지 스타일 파악

> 💡 **왜?** 커밋 메시지는 "무엇을, 왜 바꿨는지"를 담아야 하므로, 실제 diff를 읽고 작성해야 한다. 또 기존 커밋 스타일(예: `docs:`, `feat:` 같은 한글 conventional commit)을 따라야 일관성이 유지된다.

### 2. 변경 사항 검토
- 변경 내용이 하나의 논리적 작업 단위인지 확인한다.
- 커밋하면 안 되는 파일(비밀키, `.env`, 임시 파일, IDE 설정 등)이 섞여 있지 않은지 점검한다.
  - 이 저장소에는 `.idea/`(IntelliJ 설정)가 있으니, 사용자가 명시적으로 원하지 않는 한 커밋에 포함하지 않는다.
- 의도와 다른 파일이 섞여 있으면 사용자에게 먼저 알린다.

### 3. 스테이징
- 관련된 파일만 선택적으로 추가한다: `git add <파일경로>`
- 사용자가 "전부 올려줘"라고 명시한 경우에만 `git add -A` 또는 `git add .` 사용을 고려한다.

### 4. 커밋
- 이 저장소의 스타일(한글 conventional commit)에 맞춰 메시지를 작성한다.
  - 형식: `<type>: <한글 요약>`
  - type 예시: `feat`(기능), `fix`(버그수정), `docs`(문서), `refactor`(리팩터링), `chore`(잡일), `test`(테스트)
- 제목은 50자 내외로 간결하게, 명령형/요약형으로 작성한다.
- 변경 이유나 맥락이 중요하면 본문에 한 줄 비우고 추가한다.
- 커밋 메시지 끝에 다음 서명을 포함한다:

  ```
  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  ```

  HEREDOC을 사용해 커밋한다:
  ```bash
  git commit -m "$(cat <<'EOF'
  docs: 예시 커밋 메시지

  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  EOF
  )"
  ```

### 5. 푸시
- 현재 브랜치를 원격에 푸시한다: `git push`
- 원격에 추적 브랜치가 없으면(`git push`가 실패하면): `git push -u origin <현재브랜치>`
- main/master 브랜치라면 푸시 전에 사용자에게 한 번 더 확인한다. (이 미션은 개인 브랜치에서 작업 후 PR을 보내는 방식이므로 보통 개인 브랜치에 푸시한다.)

### 6. 결과 보고
- 커밋 해시와 메시지, 푸시된 브랜치를 사용자에게 알린다.
- 필요하면 PR 생성 명령(`gh pr create`)을 안내한다. (이 저장소는 PR 기반 코드 리뷰 흐름을 따른다.)

## 주의사항
- **사용자가 요청하지 않은 변경까지 커밋하지 않는다.** 무엇을 커밋할지 불명확하면 먼저 묻는다.
- 비밀정보(토큰, 비밀번호, 키)가 diff에 보이면 커밋을 멈추고 사용자에게 알린다.
- `git config` 사용자 정보를 임의로 바꾸지 않는다.
- 강제 푸시(`--force`)는 사용자가 명시적으로 요청할 때만, 위험성을 알린 뒤 사용한다.
