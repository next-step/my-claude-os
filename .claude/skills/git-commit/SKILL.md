# Git Commit 스킬

변경 사항을 분석해 **접두사 규칙에 맞는 커밋 메시지**를 작성하고 git 커밋까지 수행한다. 푸시는 하지 않는다.

## 접두사(Prefix) 규칙

커밋 메시지와 PR 제목은 반드시 아래 접두사 중 하나로 시작한다. 형식은 `<접두사>: <설명>`.

| 접두사 | 사용 시점 |
|--------|-----------|
| `feature`  | 새로운 기능 추가 |
| `refactor` | 동작은 그대로 두고 가독성·유지보수성 향상 (함수 분리, 변수명 개선, 중복 제거 등) |
| `test`     | 테스트 코드 추가 |
| `fix`      | 버그 수정 |
| `hotfix`   | 긴급 수정 |
| `infra`    | 시스템 기본 구조·운영 환경 설정 |
| `docs`     | 문서 수정 |
| `style`    | 코드 변경 없는 포맷팅 (줄바꿈, 공백 정리 등) |
| `chore`    | 프로젝트 구조 변경, 빌드/패키지 매니저 수정 등 |
| `merge`    | 브랜치 병합, 병합 충돌 해결 |

## 메시지 작성 규칙

- 제목: `<접두사>: <한 줄 요약>` — 명령형, 한국어, 50자 이내 권장.
- 변경이 여러 갈래면 본문에 `-` 불릿으로 핵심을 정리한다.
- **무엇을** 바꿨는지보다 **왜** 바꿨는지가 드러나면 더 좋다.
- 변경 성격이 여러 접두사에 걸치면, 논리적으로 **여러 개의 커밋으로 분리**하는 것을 우선 고려한다.

## 실행 절차

1. **현재 상태 파악** — 아래를 병렬로 실행한다.
   - `git status` — 변경/추적되지 않은 파일 확인
   - `git diff` (스테이징 안 된 변경) 와 `git diff --staged` (스테이징된 변경)
   - `git log --oneline -10` — 기존 커밋 메시지 스타일 확인
   - `git branch --show-current` — 현재 브랜치 확인

2. **변경 분석 & 접두사 결정** — diff 내용을 보고 위 표에서 가장 알맞은 접두사를 고른다. 성격이 다른 변경이 섞여 있으면 커밋을 나눈다.

3. **스테이징** — 의도한 파일만 `git add` 한다. 관련 없는 파일을 한꺼번에 올리지 않는다.

4. **커밋** — 메시지는 HEREDOC으로 작성한다.
   ```bash
   git commit -m "$(cat <<'EOF'
   feature: 사용자 로그인 기능 추가

   - JWT 기반 인증 미들웨어 구현
   - 로그인 실패 시 에러 응답 처리
   EOF
   )"
   ```

5. **결과 보고** — 커밋 해시와 메시지를 사용자에게 알린다. 푸시는 하지 않으며, 필요하면 사용자가 직접 `git push` 하도록 안내한다.

## 주의사항

- 커밋은 사용자가 명시적으로 요청했을 때만 한다. 코드만 수정한 단계에서 임의로 커밋하지 않는다.
- 이 스킬은 **커밋까지만** 수행한다. `git push`는 실행하지 않는다.
- 커밋 전 `git status`로 실제 스테이징된 내용을 반드시 확인한다.
- 비밀키·토큰·`.env` 등 민감 파일이 포함되지 않았는지 점검한다.
- 사용자가 별도 지시를 하지 않는 한 `git config`(사용자 이름/이메일)는 변경하지 않는다.
