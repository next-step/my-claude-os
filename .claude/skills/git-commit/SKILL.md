현재 작업 트리의 변경사항을 **prefix가 붙은 atomic commit**으로 만들어줘.

추가 맥락: $ARGUMENTS

## 핵심 원칙 (반드시 지킬 것)

1. **모든 커밋은 타입 prefix로 시작한다.** prefix 없는 커밋은 만들지 않는다.
2. **한 커밋 = 한 가지 기능 단위.** 성격이 다른 변경이 섞여 있으면 단위별로 **나눠서 여러 커밋**을 만든다.
3. **`git push`는 하지 않는다.** 로컬 commit 까지만 수행한다.

## 진행 순서

### 1. 변경 상태 파악
- `git status` 로 변경된 파일 목록 확인
- `git diff HEAD` (unstaged + 이미 staged 모두) 와 `git diff --staged` 로 실제 변경 내용을 읽는다
- 변경이 하나도 없으면 그 사실을 알리고 중단한다

### 2. 변경을 "기능 단위"로 분류
파일과 헝크(hunk)를 의미 단위로 묶는다.

- 서로 **목적이 다르면** 분리한다 (예: 버그 수정 vs 새 기능 vs 문서 수정이 한 번에 들어온 경우 → 별도 커밋)
- **같은 기능**을 위한 여러 파일 변경은 한 커밋으로 묶는다
- 한 파일 안에 성격이 다른 변경이 섞여 있으면 `git add -p` 로 헝크 단위로 나눈다

분류가 끝나면 **커밋을 만들기 전에** 그룹별 계획을 사용자에게 먼저 보여준다:

```
커밋 계획:
1. feat: 로그인 폼 유효성 검증 추가   →  src/login.tsx, src/validators.ts
2. docs: README 설치 가이드 보강       →  README.md
```

### 3. Prefix 규칙
커밋 메시지 형식: `<type>: <요약>` (Conventional Commits)

| prefix | 용도 |
|--------|------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변화 없는 구조 개선 |
| `docs` | 문서 변경 |
| `chore` | 빌드·설정·패키지 등 잡무 |
| `test` | 테스트 추가·수정 |
| `style` | 포맷·세미콜론 등 동작 무관 스타일 |
| `perf` | 성능 개선 |

- 요약은 무엇을 했는지 간결하게. 어떤 prefix가 맞을지 모호하면 추측하지 말고 사용자에게 확인한다.

### 4. Atomic commit 실행 (자동 분리)
- 그룹이 **1개**면 → 해당 파일만 `git add <파일들>` 후 커밋
- 그룹이 **여러 개**면 → 그룹마다 `git add <해당 파일들>` (필요 시 `git add -p`) → 커밋, 을 순서대로 반복
- **`git add .` / `git add -A` / `git commit -am` 처럼 전체를 한 번에 담는 명령은 쓰지 않는다** (단위 섞임 방지)
- 각 커밋 후 `git log --oneline -n <개수>` 로 결과를 보여준다

### 5. 커밋 직전 점검 (보안)
스테이징한 diff에 시크릿(API key, token, password, 비밀번호, DB 인증정보, private key 등)이나 실제 고객·운영 데이터가 포함됐는지 가볍게 확인한다. 의심되면 **커밋하지 말고** 먼저 사용자에게 알린다.

### 6. 하지 않는 것
- `git push` (commit 까지만)
- 요청 범위 밖 파일을 임의로 스테이징하는 것
- 기존 인증·권한·로깅·암호화 로직을 함께 건드리는 것

## 메시지 작성 규칙
- 제목 줄은 간결하게(50자 내외 권장). 필요하면 본문에 "왜" 바꿨는지 적는다.
- 메시지는 한국어 기본, 기술 용어·라이브러리명·코드는 원문(영어) 그대로 둔다.
