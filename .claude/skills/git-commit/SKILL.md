# Git Commit 스킬

변경사항을 **좋은 커밋(Good Commit)** 원칙에 맞춰 커밋한다. 커밋은 리뷰어에게
이정표다 — 잘 정리된 커밋은 리뷰를 돕고, 아니면 장애물이 된다.

## 이 스킬의 범위

**"잘 커밋하는 일"만** 한다.

- **빌드/테스트 검증은 이 스킬 책임이 아니다.** 변경은 이미 검증됐다고 가정한다(앞선 작업·CI 책임).
- 그래서 빌드를 돌리거나 `git stash`로 working tree를 휘젓지 않는다.
- 변경이 명백히 미완성/깨진 상태면(빈 함수, TODO만) 커밋하지 말고 먼저 알린다.

## 핵심 규칙 (반드시 지킬 것)

1. **Co-authored-by에 AI/Claude 이름을 절대 넣지 않는다.**
2. **타입 접두사를 쓰지 않는다** (`feat:`, `fix:`, `chore:`, `docs:` 등).
3. **사용자가 명시적으로 요청할 때만 커밋한다.**
4. **기존 이력의 언어를 따른다** (이 레포는 한국어).

## 좋은 커밋의 3원칙

1. **작게, 하나의 작업 단위.** 한 커밋엔 한 목적만. 크기 기준은 "제목 한 줄로
   포괄되는 규모" — 넘으면 나눈다.
2. **순서를 준다.** 논리적으로 쌓이는 순서로 커밋한다(구조·파일 추가 → 구현 →
   연결·바인딩). 앞 커밋을 뒤에서 다시 고치지 말고, 필요하면 순서를 정리한다.
3. **메시지에 의미(왜)를 담는다.** 코드를 그대로 옮긴 제목은 정보가 없다.
   무엇을 왜 했는지 의도를 담는다.

## 커밋 절차

### 1단계: 상태 파악 (병렬)

```bash
git status
git diff HEAD
git log --oneline -10   # 기존 메시지 스타일·언어 참고
```

### 2단계: 논리적 단위로 나누기 (원칙 1·2)

- 목적이 다른 변경이 섞였으면 나눈다. 관련 파일만 `git add <파일>`.
- 여러 커밋이면 쌓이는 순서로 올린다.
- `git add .` 금지. `git status`로 의도한 파일만 확인. 비밀값·생성물·임시파일은 제외.

### 3단계: 메시지 작성 (원칙 3)

**제목** — 명사 종결형 요약("~추가", "~수정"). 한국어는 ~30자. 마침표·접두사 없음.
코드를 그대로 옮기지 말고 의도를 담는다.

**본문(필요할 때만)** — 제목과 한 줄 띄우고, 무엇이 아니라 **"왜"** 를 쓴다.
판단·트레이드오프·배경이 있으면 반드시. 72자 안쪽 줄바꿈.

```bash
git commit -m "$(cat <<'EOF'
스크롤 확정 후 인덱스로 선택 상태 갱신

didSelectItemAt:에서 바로 select하면 스크롤 전 인덱스라 어긋나, deselect만 처리.
EOF
)"
```

### 4단계: 결과 확인

```bash
git log -1 --stat
```

푸시는 사용자가 요청할 때만 한다.

## 예시

좋은 예 — 의도가 드러남
```
스크롤 확정 후 인덱스로 선택 상태 갱신
```

나쁜 예
```
deselectItem 함수 호출 추가    코드를 그대로 옮겨 의도가 없음
feat: update                  접두사 + 불명확
수정함                        무엇을 왜 했는지 없음
```

## 주의

- `git config user.name`/`email`을 임의로 바꾸지 않는다(작성자는 사용자 본인).
- 커밋 순서 정리(rebase 등 history 변경)·force push는 되돌리기 어렵다. 아직 push 안 한
  로컬 커밋에 한해, 사용자에게 먼저 확인한다. (CLAUDE.md 가드레일)
