# retrospect — 회고 스킬

OS 루프의 **피드백 단계**다. `recommend-stocks`(선정 기록)와 `analyze-company`(분석 기록)가
쌓아둔 과거 예측을 **현재가와 대조**해 무엇이 맞고 틀렸는지 가리고, 다음 추천·분석에 쓸 기준을 개선한다.

**핵심 설계 — 사실은 스크립트, 해석은 토론.**
"목표 도달했나/손절 닿았나"는 현재가와 진입/목표/손절을 비교하는 *산수*라 스크립트가 결정론적으로 확정한다.
"왜 그랬나 / 기준을 어떻게 바꾸나"는 정답이 없는 *해석*이라 전문가들이 수렴할 때까지 토론한다.
(OS 철학: 정성 해석=서브에이전트, 정밀 수치=결정론적 스크립트. analyze-company 와 동일한 분리.)

```
A. [스크립트] evaluate_records.py  → 종목별 status·실현수익률 사실 확정 (현재가는 quote.py 재사용)
   ↓
A'. [수집] kr-macro-researcher 1회 호출 → 분석 구간의 거시 사실판 확정 (analyze-company 와 공유하는 에이전트)
   ↓
B. [토론 루프] 전문가 4인 동적 토론 (최대 5R, 수렴하면 조기 종료) — 사실표 + 거시 사실판을 공통 근거로
   ↓
C. [종합] update_status.py 로 원본 status 갱신 + save_retro.py 로 회고 리포트 + 튜닝안 제시
```

> **거시 사실 수집을 토론 밖으로 뺀 이유:** 예전엔 거시 전문가가 라운드마다 직접 웹검색했다. 그러면 ① 라운드마다 중복 검색이고 ② 4인이 서로 다른 시장 사실 위에서 다툰다. 이제 메인이 `kr-macro-researcher`로 **한 번** 수집해(=`evaluate_records.py`가 종목 사실을 한 번 확정하듯) 4인에게 똑같이 공유한다. "사실은 수집, 해석은 토론" 분리를 거시에도 그대로 적용한 것. 이 에이전트는 analyze-company 가 현재 환경을 볼 때도 쓰는 **공유 부품**이다.

## 실행 절차

### 1단계 — 사실 판정 (결정론적 스크립트) ⚠️ 토론보다 먼저
현재가를 끌어와 과거 예측의 결과를 **사실로 확정**한다. 이 수치가 이후 토론의 공통 근거(사실표)다.
```
python3 .claude/skills/retrospect/scripts/evaluate_records.py
```
- `data/analyses/*.md` 전체를 읽어 각 종목의 현재가(analyze-company `quote.py` 재사용)와
  진입/목표/손절을 대조해 `status`(hit_target | stopped | watching | open)·실현수익률·경과일을 산출한다.
- **터미널 상태(hit_target/stopped)는 sticky** — 한 번 도달/손절이면 되돌리지 않는다(예측의 성패는 일어난 사건).
- 추천 기록의 **탈락 표본**도 현재가만 맥락으로 끌어온다(빠뜨린 종목이 올랐나 — 단 탈락 당시 가격 미기록이라 정성 점검만).
- **기록이 없으면**(`evaluated: []`) 회고할 게 없다 — 사용자에게 "먼저 analyze-company/recommend-stocks 로 기록을 쌓아야 한다"고 알리고 종료한다.
- 출력 JSON 전체를 보관한다. 이게 2단계 전문가들에게 넘길 **사실표**다.

### 1.5단계 — 거시 사실판 수집 (수집 전용 에이전트, 토론 전 1회) ⚠️ 토론 시작 전
토론에 들어가기 전에 **거시 시장 환경을 한 번만** 조사해 4인의 공통 근거로 만든다.
(거시 전문가가 라운드마다 직접 검색하지 않는다 — `retro-macro-analyst`는 토론 전담으로 슬림화됐다.)
```
Agent(kr-macro-researcher) ← "기간: <가장 이른 분석일> ~ <기준일(오늘)> (구간 비교 모드).
    이 구간 동안 한국 시장이 전반적으로 올랐나/빠졌나(코스피 지수 변동)·금리·환율·업종 흐름을 조사해줘."
```
- 기간은 1단계 사실표에서 **가장 이른 `분석일`부터 기준일(오늘)**까지로 잡는다(평가 대상 전체를 덮는 구간).
- 돌려받은 거시 요약(지수 변동·금리·환율·업종·출처)을 **거시 사실판**으로 보관한다. 2단계에서 **4인 모두에게** 사실표와 함께 넘긴다.
- `kr-macro-researcher`는 `analyze-company`가 현재 환경을 볼 때도 쓰는 **공유 수집 에이전트**다(용도=기간만 다름).

### 2단계 — 전문가 토론 (동적 루프, 최대 5라운드) ⚠️ 이 스킬의 핵심
서브에이전트는 서로 직접 대화하지 못한다. **네(메인)가 사회자가 되어 라운드로 중재**한다.
참여 전문가 4인: `retro-technical-analyst` · `retro-fundamental-analyst` · `retro-macro-analyst` · `retro-skeptic`.

**라운드 1 — 개회 주장 (4인 동시 소집)**
한 번의 응답에서 Agent 도구를 **4번 호출**해 동시에 띄운다(병렬). 각자에게 넘긴다:
- 1단계 **사실표**(전체 또는 관련 슬라이스),
- 1.5단계 **거시 사실판**(kr-macro-researcher 결과 — 특히 거시 전문가가 자기 시장 맥락의 단일 출처로 쓴다. 직접 검색 금지),
- 각 분석 기록의 `axes`(그때 어떤 근거로 봤나 — 파일에서 읽어 함께 전달),
- "라운드 1이다. 다른 전문가 입장은 아직 없다. 네 관점의 개회 입장을 정해진 출력 형식으로 내라."

각 전문가는 `### 합의 여부` 줄에 `AGREE` 또는 `DISSENT: <쟁점>` 토큰을 넣어 돌려준다.

**수렴 판정 (매 라운드 후 네가 한다)** — 다음을 모두 만족하면 **수렴 → 토론 종료**:
1. `DISSENT` 를 낸 전문가가 없다(전원 AGREE), **또는**
2. 직전 라운드 대비 **새로운 실질 반박이 안 나온다**(같은 쟁점 반복이면 더 돌 의미 없음).
- 아직 `DISSENT` 가 있고 새 논점이 있으면 → 다음 라운드.

**라운드 2~5 — 반박/수정 (이의자만 재소집)**
- 비용 절약: **직전에 `DISSENT` 했거나, 남의 주장에 반응할 게 있는 전문가만** 다시 부른다(전원 AGREE한 사람은 재소집 안 함).
- 각 전문가에게 **다른 전문가들의 직전 입장 전문**을 함께 넘기고 "여기에 반박/동의하고, 합의 여부를 갱신하라"고 한다.
- 매 라운드 후 다시 수렴 판정. **최대 5라운드**까지만.

**5라운드까지 미합의면** — 네가 다수·근거 우위로 결론을 내리되, **회의론자가 끝까지 고수한 쟁점은 억지로 합의시키지 말고** "미해결 쟁점(다음 회고로 이월)"으로 남긴다. (회의론자를 넣은 이유 — 섣부른 기준 변경 방지.)

> **사회자로서 네 자세:** 전문가 말을 그대로 받아쓰지 말고, 충돌을 **연결·중재**하라. 예: 기술적은 "진입 적중"이라는데 거시는 "그냥 장이 좋았다"면 → 회의론자에게 표본·베타 분리를 묻고, 셋의 말을 종합해 "실력인지 운인지" 결론을 내려라.

### 3단계 — 종합 & 기록 (방식 A)
토론 결과를 사용자에게 보여주고, 두 가지를 기록한다.

**(a) 원본 status 갱신** — 1단계에서 status 가 바뀐 종목만(`status_changes`):
```
echo '{"updates":[{"file":"data/analyses/...","status":"hit_target"}, ...]}' | python3 .claude/skills/retrospect/scripts/update_status.py
```
- 원본 분석파일의 **frontmatter `status:` 줄만** 바뀐다. 진입/목표/손절·근거·본문은 **건드리지 않는다**(예측 박제).
- 이로써 `recommend-stocks` 추천표가 분석 기록 status 를 역참조할 때 회고 결과가 자동 반영된다.

**(b) 회고 리포트 저장** — 토론 경위·종목별 평가·튜닝안을 조립해 넘긴다(스키마는 `scripts/save_retro.py` 상단):
```
echo '<조립한 JSON>' | python3 .claude/skills/retrospect/scripts/save_retro.py
```
- `data/retros/YYYY-MM-DD-retro.md` 로 append-only 저장(같은 날 재실행은 `-2`,`-3`…).
- `rounds`·`converged`·`open_issues` 에 토론이 몇 라운드 돌았고 수렴했는지, 미해결 쟁점이 뭔지 정확히 담는다.

### 4단계 — 사용자에게 출력
아래 골격으로 보여준다.
```
# 📒 회고 리포트  (기준일: YYYY-MM-DD)

## 성적표 (사실)
- 평가 N건: 목표달성 x · 손절 y · 관찰중 z · 평균 실현수익률 ±%
- (종목별 status·실현수익률 표)

## 전문가 토론 요약
- 총 R라운드 (수렴 종료 / 5R 미합의)
- 핵심 쟁점과 어떻게 결론났나 (실력 vs 운 등)

## 합의된 결론
- ...

## 미해결 쟁점 (있으면 — 다음 회고로 이월)
- ...

## 튜닝 제안 (다음 추천/분석 반영 — ⚠️ 실제 수정은 사람 승인)
- [대상 스크립트/문서] 무엇을 어떻게 — 왜

## ⚠️ 유의
- 회고는 과거 예측을 평가할 뿐, 원본 분석의 가격대·근거는 수정하지 않는다.
- 튜닝안은 제안이며, 스크립트 상수/OS.md 의 실제 변경은 사람이 승인 후 반영한다.
```

### 5단계 — 튜닝 반영은 사람 승인 후
도출된 튜닝안(예: "부채비율 컷 150%→130%")은 **제안까지**가 이 스킬의 역할이다.
사용자가 승인하면 그때 해당 스크립트 상수와 `docs/OS.md`(+`docs/OS-log.md` 로그)를 함께 고친다.
*왜 자동 반영 안 하나:* OS 가드레일 — 기준 변경은 영향이 크고, 회고가 자기 판단으로 시스템을 바꾸면 통제 불능. 사람이 최종 결정.

## 원칙
- **사실은 스크립트, 해석은 토론.** status·수익률 같은 수치는 `evaluate_records.py` 가 확정한 값만 쓴다(전문가가 재검색·재계산 금지).
- **append-only 존중(방식 A).** 원본 분석 기록은 `status:` 한 줄만 갱신하고 예측·근거는 박제. 회고 본문은 별도 `data/retros/` 에 누적.
- **억지 합의 금지.** 표본이 적으면 회의론자의 `DISSENT` 를 존중해 "미해결 쟁점"으로 남긴다. 과적합 방지가 합의보다 중요.
- **이 스킬은 평가·제안까지만.** 실제 기준 변경·매매는 사람 몫(OS 가드레일).
- 트리거는 **수동**(주 1회 권장). 자동 스케줄은 미래 로드맵.
