# task-impl

ticket-start 브리핑을 받아 개발 단위(태스크)로 분해하고,
태스크 하나씩 구현 → 확인 → 커밋을 반복해 작업을 완료한다.

`/ticket-start` → **/task-impl** → `/dev-loop` 흐름의 중간 단계.

---

## 0단계: 입력 파악

아래 순서대로 입력을 찾는다:

1. **`docs/ticket-briefing.md` 파일**: 먼저 Read로 파일 존재 여부를 확인한다. 있으면 해당 내용을 브리핑으로 사용한다.
2. **직전 대화의 ticket-start 브리핑**: "## 티켓 브리핑:" 형식으로 시작하는 텍스트
3. **자유 형식 설명**: 사용자가 직접 작업 범위를 설명한 텍스트

셋 다 없으면 다음을 묻는다:

```
작업 범위를 설명해 주세요.
ticket-start 브리핑 텍스트를 붙여넣거나, 할 일을 직접 설명해도 됩니다.
```

입력을 확보한 뒤, 아래 파일들을 순서대로 확인해 **커밋 컨벤션**을 감지한다:

```
commitlint.config.js / .commitlintrc / .commitlintrc.json
CONTRIBUTING.md
.github/COMMIT_CONVENTION.md
docs/commit-convention.md
```

- **감지 시**: 타입 목록·언어·스코프 필수 여부 등 규칙 요약을 메모하고 `[컨벤션 감지됨: <파일명>]` 출력 후 1단계로 이동
- **미감지 시**: 기본 Conventional Commits 규칙 사용 (아래 C단계 참조)

---

## 1단계: 태스크 분해

입력에서 다음을 추출한다:

- 요구사항 요약
- 영향받는 파일 / 컴포넌트 목록
- 사이드이펙트 체크리스트 (있는 경우)

그 다음 **개발 단위(태스크)** 로 쪼갠다.

### 분해 기준

| 기준          | 설명                                                                         |
| ------------- | ---------------------------------------------------------------------------- |
| 논리적 완결성 | 하나의 태스크 = 하나의 커밋으로 설명 가능한 변경                             |
| 의존성 분리   | A가 완료돼야 B를 시작할 수 있는 경우 순서를 명시                             |
| 파일 응집도   | 같은 파일/컴포넌트를 건드리는 변경은 하나의 태스크로 묶음                    |
| 크기 제한     | 태스크 하나가 너무 크면 재분할 (기준: 파일 3개 초과 또는 200줄 초과 예상 시) |

---

## 2단계: 태스크 목록 확인

분해 결과를 아래 형식으로 출력한다:

```
## 개발 태스크 목록

총 N개 태스크 | 예상 커밋 N개

┌───┬──────────────────────────────┬─────────────────────────┬──────┐
│ # │ 태스크 제목                     │ 대상 파일                 │ 의존 │
├───┼──────────────────────────────┼─────────────────────────┼──────┤
│ 1 │ [제목]                        │ [파일명, 파일명]           │  -   │
│ 2 │ [제목]                        │ [파일명]                  │  1   │
│ 3 │ [제목]                        │ [파일명, 파일명]           │  1   │
└───┴──────────────────────────────┴─────────────────────────┴──────┘

각 태스크 상세:

**태스크 1: [제목]**
- 대상: [파일명]
- 변경 내용: [무엇을 어떻게]
- 커밋 메시지 예상: `feat(xxx): [요약]`

**태스크 2: [제목]**
...

이 순서대로 진행할까요?
수정하려면 알려주세요. (예: "2와 3 순서 바꿔줘", "1번 빼줘", "4번 추가: ...")
문제없으면 "시작"이라고 입력해 주세요.
```

사용자가 수정을 요청하면 목록을 업데이트하고 다시 보여준 뒤 재확인한다.
"시작" 또는 승인 의사를 받으면 3단계로 이동한다.

---

## 3단계: 태스크 구현 루프

확정된 태스크 목록을 순서대로 처리한다.

### 사전 평가: 실행 모드 결정

아래 기준을 확인한다:

- **태스크 수 ≥ 5** 또는 **모든 태스크 의존 = `-`** → **에이전트 위임 모드** (아래 위임 절차 따름)
- 그 외 → **직접 실행 모드** (각 태스크마다 A→B→C 루프)

---

#### 에이전트 위임 모드 절차

**1. 위임 전 준비**: 태스크 목록 전체를 검토해 불명확한 부분을 사용자에게 질문하고 해소한다.

**2. 에이전트 프롬프트 작성**: 각 태스크마다 아래 내용을 포함한다.

> 다음 태스크를 구현하고 커밋하세요.
>
> **태스크**: [태스크 제목]
> **대상 파일**: [파일 목록]
> **변경 내용**: [상세 설명]
> **커밋 컨벤션**: [감지된 규칙 or 기본 Conventional Commits]
> **이전 태스크 완료 정보**: [의존 태스크 SHA 및 변경 요약 / 없으면 '없음']
>
> 규칙:
> - 불명확한 부분은 추측하지 말고 `[CLARIFICATION_NEEDED: 질문]` 형식으로 반환
> - `.env` 등 민감 파일 감지 시 커밋 중단 후 `[SECURITY_ALERT: 파일명]` 반환
> - 완료 후 마지막 줄에 `[SHA: abc1234]` 형식으로 커밋 SHA 출력

**3. 실행 방식**:
- 의존 = `-` 태스크들: 단일 메시지에 여러 Agent 호출로 병렬 실행
- 의존 있는 태스크: 앞 태스크 완료 후 순차 실행

**4. 반환값 처리**:
- `[CLARIFICATION_NEEDED]` → 사용자에게 질문 후 해당 에이전트 재실행
- `[SECURITY_ALERT]` → 경고 출력 후 전체 중단
- `[SHA: ...]` → SHA 기록 후 다음 태스크로

---

#### 직접 실행 모드

각 태스크마다 아래 A → B → C를 반복한다.

### A. 태스크 시작 알림

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
태스크 [N/전체] 시작: [태스크 제목]
대상 파일: [파일명 목록]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### B. 구현

대상 파일을 먼저 Read로 읽은 뒤, 태스크 내용에 따라 Edit/Write로 변경한다.

- 기존 파일 수정: Read → Edit (최소한의 변경, 스타일은 기존 코드에 맞춤)
- 새 파일 생성: Write
- 구현 중 불명확한 부분이 생기면 즉시 사용자에게 질문하고 진행 (추측하지 않음)

### C. 커밋

**1. 커밋 메시지 생성**

0단계에서 감지된 컨벤션이 있으면 해당 규칙을 따른다.
감지된 컨벤션이 없으면 기본 규칙을 사용한다:

| 항목    | 기본 규칙                                             |
| ------- | ----------------------------------------------------- |
| 포맷    | `<type>(<scope>): <subject>`                          |
| 타입    | `feat` `fix` `docs` `style` `refactor` `test` `chore` |
| 언어    | 한국어                                                |
| subject | 50자 이내, 명령형, "무엇"이 아닌 "왜"                 |

**2. 보안 체크**

```bash
git diff --cached --name-only | grep -E '\.env|secrets|credentials'
```

`.env` 등 민감 파일 감지 시 **반드시 경고** 후 중단.

**3. 커밋 실행**

커밋 메시지를 정보용으로 한 줄 출력한 뒤 즉시 실행한다:

```
[커밋] feat(auth): 토큰 만료 시 자동 갱신 로직 추가
```

```bash
git add -A
git commit -m "$(cat <<'EOF'
<생성된 커밋 메시지>
EOF
)"
```

커밋 완료 후 SHA를 기록해두고 다음 태스크로 이동한다.

---

## 4단계: 최종 요약 출력

모든 태스크가 완료되면 아래 형식으로 요약한다:

```
╔══════════════════════════════════════════════════╗
║          /task-impl 완료 요약                    ║
╠══════════════════════════════════════════════════╣
║  태스크 1: [제목]   → ✅ abc1234                 ║
║  태스크 2: [제목]   → ✅ def5678                 ║
║  태스크 3: [제목]   → ✅ ghi9012                 ║
╠══════════════════════════════════════════════════╣
║  총 N개 태스크 완료 | N개 커밋 생성              ║
╚══════════════════════════════════════════════════╝

다음 단계: /dev-loop 로 셀프 리뷰 + PR 생성을 진행하세요.
```

태스크 중 일부가 건너뛰어지거나 실패한 경우:

- `⏭ 건너뜀` — 사용자가 명시적으로 건너뛴 경우
- `❌ 미완료` — 수동 작업이 필요해 중단된 경우 (이유 표시)

---

## 중단 / 재개 처리

구현 중 사용자가 "중단", "잠깐 멈춰", "나중에 계속" 등을 입력하면:

```
현재까지 완료된 태스크:
- ✅ 태스크 1: [제목] (abc1234)
- ✅ 태스크 2: [제목] (def5678)

남은 태스크:
- ⏳ 태스크 3: [제목]
- ⏳ 태스크 4: [제목]

재개하려면 "/task-impl 재개" 또는 다음 태스크를 직접 지시해 주세요.
```

---

## 주의사항

- 2단계 확인 후에는 각 태스크 구현 완료 즉시 자동으로 커밋한다 (태스크별 중간 확인 없음)
- 태스크 하나를 구현하는 중 다른 파일을 임의로 수정하지 않는다 (사이드이펙트 방지)
- 구현 중 불명확한 요구사항이 있으면 추측하지 않고 즉시 질문한다
- `.env`, 비밀키, 개인정보 파일이 감지되면 반드시 경고 후 중단한다
- `--no-verify` 등 훅 우회 플래그는 사용자가 명시적으로 요청할 때만 사용한다
- 각 태스크는 독립적으로 빌드 가능한 상태(컴파일 에러 없음)로 구현한다
