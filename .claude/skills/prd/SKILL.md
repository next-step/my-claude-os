# /prd — 요구사항 → PRD 단계 (① 요구사항)

[OS.md](../../../OS.md) 개발 흐름의 **① 요구사항 단계**만 담당하는 스킬이다.
(배경·원칙은 OS.md를 따른다. 여기서는 그 단계를 실제로 어떻게 굴리는지만 적는다.)

이 스킬의 입력은 **사람이 직접 작성한 요구사항 md**다. 한 줄짜리 의도가 아니라,
사람이 글로 정리한 요구사항을 바탕으로 PRD를 만든다 → 입력이 풍부해 PRD 품질이 올라간다.

전달된 인자: ``
- 인자가 있으면 → 개발할 **기능 설명**(슬러그 추정용)으로 사용한다.
- 인자가 없으면 → "어떤 기능의 요구사항을 정리할까요?"라고 한 줄로 묻고 시작한다.

이 스킬의 범위는 **① 요구사항(PRD 확정)까지**다. ② Plan부터는 `/plan`(예정)·`/feature`에서 한다.

---

## 진행 순서

### [준비] 기능 이름(slug) 정하기

기능 설명에서 짧은 파일명용 slug를 정한다(영문 kebab-case 권장).
예: "다크모드 토글" → `dark-mode-toggle`. 산출물 경로는 아래와 같다.

```
docs/requirements/<slug>.md   # 사람이 채우는 요구사항 (이 스킬의 입력)
docs/prd/<slug>.md            # AI가 만드는 PRD (이 스킬의 출력)
```

디렉토리가 없어도 `Write`가 자동 생성하므로 미리 만들 필요는 없다.

---

### ① 요구사항 템플릿 생성 — ★멈춤(사람이 채움)★

먼저 `docs/requirements/<slug>.md` 가 이미 있는지 `Bash(test:*)`로 확인한다.

- **이미 있으면** → 사람이 이미 작성한 것으로 보고, 새로 덮어쓰지 말고 ②로 갈 준비를 한다.
- **없으면** → 아래 **빈 템플릿**을 `Write`로 만들어, 사람이 채울 뼈대를 제공한다.

```markdown
# 요구사항 — <기능 이름>

> 이 파일은 **사람이 직접 채웁니다.** 채운 내용을 바탕으로 /prd 가 PRD를 만듭니다.
> 모르는 항목은 비워두어도 됩니다(PRD 단계에서 "확인이 필요한 점"으로 남깁니다).

## 한 줄 요약
(이 기능을 한 문장으로)

## 배경 / 문제
(왜 필요한가, 지금 무엇이 불편한가)

## 원하는 동작
(사용자가 무엇을 할 수 있어야 하나 — 가능하면 시나리오로)

## 제약 / 조건
(꼭 지켜야 할 것: 성능, 호환, 마감 등)

## 이번에 안 할 것 (범위 밖)
(혼동을 막기 위해 명시적으로 제외)

## 참고
(관련 링크, 기존 기능, 예시)
```

템플릿을 만들었으면 **사람이 채울 수 있도록 멈춘다.** 보고 형식:

```
📝 요구사항 템플릿을 만들었습니다 → docs/requirements/<slug>.md

이 파일을 열어 직접 채워 주세요. (채팅이 아니라 md 파일에 적습니다.)
다 채우셨으면 "다 썼어" / "PRD 진행"처럼 알려 주세요. → ②로 넘어갑니다.
```

> 왜 사람이 직접 쓰나: 요구사항을 글로 정리하는 행위 자체가 생각을 또렷하게 만든다.
> AI가 한 줄 의도로 PRD를 지어내는 것보다, 사람이 정리한 입력에서 출발하는 편이 훨씬 정확하다.

---

### ② PRD 생성 — PRD 서브에이전트 호출

사람이 진행 신호를 주면, **요구사항 md를 `Read`로 다시 읽어** 최신 내용을 확보한 뒤
`Task` 도구로 **PRD 작성 서브에이전트**를 띄운다.

- `subagent_type`: `general-purpose`
- `description`: `"PRD 작성"`
- `prompt`: 아래 내용을 채워서 전달한다.

```
당신은 PRD(제품 요구사항) 작성 전문가다. 코드는 아직 보지 마라.
아래 "사람이 작성한 요구사항"을 구조화된 PRD로 정리하라.
사람이 적은 내용을 임의로 바꾸지 말고, 비거나 모호한 부분만 "열린 질문"으로 모아라.

사람이 작성한 요구사항:
<요구사항 md 본문 그대로>

다음 형식의 마크다운으로만 답하라(설명 문장 없이 문서 본문만):
# PRD — <기능 이름>
## 문제 (왜 필요한가)
## 목표 (이 기능이 해결하는 것)
## 사용자 시나리오 (기대 동작)
## 범위 밖 (이번에 안 하는 것)
## 제약 / 조건
## 열린 질문 (사람 확인이 필요한 점)
[요구사항에서 비었거나 모호했던 점을 질문으로. 각 질문 아래 "**답변:** _(여기에 작성)_" 줄을 둔다.]
```

서브에이전트가 돌려준 PRD 본문을 **그대로** `docs/prd/<slug>.md` 에 `Write` 로 저장한다.

---

### ③ PRD 검토 루프 — ★게이트★ (잘못된 점 확인 → 수정 → 반복)

PRD를 만들었으면 바로 확정하지 않는다. **검토 → 수정 → 다시 검토**를 사람이 만족할 때까지 돈다.

**3-1. 검토 서브에이전트 호출 (AI가 AI의 PRD를 점검)**

`Task` 도구로 **공유 검토자 서브에이전트(`reviewer`)** 를 띄운다. 적대적 검토 행동은 reviewer
에이전트 정의([.claude/agents/reviewer.md](../../agents/reviewer.md))에 있으니, 여기서는 **기준·대상·판정 형식만** 넘긴다.

- `subagent_type`: `reviewer`
- `description`: `"PRD 검토"`
- `prompt`:

```
[기준] 사람이 작성한 요구사항:
<요구사항 md 본문>

[대상] 생성된 PRD:
<PRD md 본문>

[판정 형식]
## 판정: 큰 문제 없음 | 지적 있음
## 짚은 점
- (분류: 누락/모순/모호/오류) 내용 — 어떻게 고칠지
```

**3-2. 사람에게 보고하고 멈춘다**

PRD 요약 + 검토 서브에이전트가 짚은 점을 함께 보고한다.

```
✅ PRD 작성 완료 → docs/prd/<slug>.md

[PRD 요약]
- 문제: ...
- 목표: ...

[검토에서 짚은 점]
- (누락) ... → 이렇게 보완 제안
- (모호) ... → 이렇게 명확화 제안
(없으면: 큰 문제 없음)

여기는 요구사항 검토 게이트입니다.
- 고칠 점을 알려주시면 PRD를 수정합니다(이 루프를 반복).
- PRD의 "열린 질문"은 md 파일의 "답변:" 줄에 직접 채우셔도 됩니다.
- 확정되면 "확정" / "Plan 진행"이라고 알려 주세요. → ① 단계를 마칩니다.
```

**3-3. 수정 반영**

사람이 고칠 점을 주면 PRD를 고친다. 작은 수정은 `Edit`로, 큰 재구성은 ②의 서브에이전트를
다시 호출해 다시 만든다. 고친 뒤 다시 3-2로 돌아가 보고한다. **확정 신호가 올 때까지 반복.**

> 왜 루프인가: 요구사항은 한 번에 완성되지 않는다. "만들고 → 틀린 거 확인 → 고치고"를
> 반복하는 게 정상이다. 그 반복을 스킬이 구조로 만들어 매번 같은 품질로 돌게 한다.

---

### 마침

사람이 "확정"하면 이 스킬은 끝난다. **PRD를 임의로 ② Plan·③ 구현으로 넘기지 않는다.**
다음 단계는 `/plan`(예정) 또는 `/feature` 오케스트레이터가 이어받는다.

---

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

- [ ] `/prd "샘플"` 실행 시 `docs/requirements/<slug>.md` 템플릿이 생기고 **사람이 채우도록 멈춘다.**
- [ ] 요구사항 md가 이미 있으면 덮어쓰지 않고 그걸 입력으로 쓴다.
- [ ] ② PRD 생성이 **요구사항 md를 다시 읽어** 그 내용을 반영한다.
- [ ] ② PRD 생성과 ③ PRD 검토가 각각 **별도의 `Task` 서브에이전트**로 일어난다.
- [ ] ③ 게이트에서 멈추고, 사람이 "확정"할 때까지 수정 루프를 반복한다.
- [ ] 확정 전에는 Plan/구현으로 넘어가지 않는다.

---

## 안전 규칙

- **게이트를 넘지 않는다.** 사람의 확정 전에는 다음 단계로 가지 않는다.
- **사람이 쓴 요구사항을 임의로 바꾸지 않는다.** 비거나 모호한 것만 질문으로 남긴다.
- **모든 산출물은 저장소 안에** 둔다(`docs/requirements/`, `docs/prd/`). (CLAUDE.md 규칙 1)
- 서브에이전트가 빈 결과/오류를 돌려주면 그대로 알리고 멈춘다. 추측으로 채우지 않는다.
- 이 스킬은 **문서 작성**만 한다. 코드 파일을 수정하지 않는다.
