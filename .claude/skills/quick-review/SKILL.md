# 빠른 코드 리뷰 (quick-review)

새로 개발하지 않고, 이미 있는(또는 방금 바꾼) 코드를 **가볍게 검증**만 하는 스킬이다. `feature-dev`의 무거운 4단계 대신 **2단계**만 돈다.

```
① code-analyzer → ② code-reviewer
```

> 💡 **feature-dev와 뭐가 다른가?** feature-dev는 "만들고 + 고치는 루프"까지 도는 풀 파이프라인이고, quick-review는 **읽고 판정만** 한다. 코드를 수정하지 않으므로 빠르고 안전하다. 두 스킬은 `code-analyzer`와 `code-reviewer` 에이전트를 **공유**한다.

## 사용 시점
- "리뷰해줘", "이 코드 봐줘", "문제 없는지 확인해줘", "커밋/PR 전에 훑어줘" 같은 요청을 받았을 때
- 무언가를 새로 구현하는 게 아니라, 기존/변경 코드의 품질만 보고 싶을 때

## 절차

### ① 코드베이스 파악 (맥락)
- `code-analyzer` 서브에이전트를 호출한다. 리뷰 대상(변경된 파일 또는 사용자가 지목한 부분)을 전달한다.
- 목적은 리뷰어가 "이 프로젝트의 관습과 주변 맥락"을 근거로 판단하게 하는 것.
- 대상이 명확하고 작으면(예: 방금 바꾼 몇 줄) 이 단계를 축약하거나 건너뛰어도 된다. 다만 관습을 알아야 정확한 리뷰가 되므로, 낯선 영역이면 반드시 거친다.

### ② 리뷰
- `code-reviewer` 서브에이전트를 호출한다. **①의 맥락 + 리뷰 대상(git diff 또는 지목 파일)**을 전달한다.
- code-reviewer가 정확성·안정성·보안·관습·테스트 관점으로 점검하고 **판정(통과/수정필요)**과 심각도별 지적을 낸다.

### ③ 결과 정리
- code-reviewer의 출력을 사용자에게 그대로 잘 정리해 보여준다.
- **판정**, 🔴 반드시 수정 / 🟡 개선 권장 / 🟢 사소 를 구분해 전달한다.
- 이 스킬은 **고치지 않는다.** 사용자가 "고쳐줘"라고 하면 그때 `feature-dev`나 직접 수정으로 넘어간다.

## 주의사항
- **읽기 전용 스킬이다.** 코드를 수정하지 않는다. 수정이 필요하면 사용자에게 확인 후 별도로 진행한다.
- 리뷰 대상이 불명확하면(무엇을 리뷰할지) 먼저 사용자에게 범위를 확인한다. 기본값은 "현재 커밋 안 된 변경(git diff)".
- code-reviewer의 지적은 근거와 함께 전달하되, 심각도를 부풀리지 않는다.
