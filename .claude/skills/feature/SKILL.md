# /feature — 기능 개발 오케스트레이터 (요구사항/PRD → Plan → 게이트)

[OS.md](../../../OS.md) 개발 흐름의 **①②③을 한 번에 굴리는 상위 오케스트레이터** 스킬이다.
(흐름의 배경·원칙은 OS.md를 따른다. 여기서는 그 절차를 실제로 어떻게 굴리는지만 적는다.)

스킬은 **단계별로 분리**되어 있다. 이 스킬은 직접 PRD를 만들지 않고 **각 단계 스킬/서브에이전트에
위임**한다:

- **① 요구사항·PRD** → `/prd` 스킬에 위임 (요구사항 md 작성 → PRD 생성 → 검토 루프)
- **② Plan** → Plan 서브에이전트를 `Task`로 호출 (기존 코드 재활용 조사 포함)
- **③ 사람 검토 게이트** → 이 스킬이 멈춰서 사람 승인을 받는다

> 단계만 따로 하고 싶으면 각 스킬을 직접 써도 된다(`/prd`만, 나중에 `/plan`만).
> `/feature`는 "처음부터 끝까지 한 번에" 가고 싶을 때 쓰는 상위 진입점이다.

전달된 인자: ``
- 인자가 있으면 → 개발할 **기능 설명**으로 사용한다.
- 인자가 없으면 → 사용자에게 "어떤 기능을 개발할까요?"라고 한 줄로 묻고 시작한다.

이 스킬의 범위는 **①②③까지**다. ④ 구현·⑤ 마무리는 게이트 통과 후 별도 단계/스킬에서 한다.

---

## 진행 순서

### ① 요구사항·PRD — `/prd` 스킬에 위임

`Skill` 도구로 **`/prd` 스킬을 호출**해 ① 단계 전체(요구사항 md 작성 → PRD 생성 → 검토 루프 →
확정)를 진행한다. 인자로 받은 기능 설명을 그대로 `/prd`에 넘긴다.

- `/prd`는 사람이 요구사항을 채우고 PRD를 확정할 때까지 **자체 게이트에서 멈춘다.**
  그 멈춤을 존중한다 — 사람이 "확정"하기 전에 ②로 넘어가지 않는다.
- `/prd`가 끝나면 `docs/prd/<slug>.md` 가 확정 상태로 존재한다. 그 slug를 ②에서 그대로 쓴다.

> 왜 위임하나: 요구사항 단계는 사람이 글을 쓰고 AI가 검토하는 **고유한 반복 루프**가 있다.
> 그걸 `/prd`가 전담하면, `/feature`는 "단계를 잇는" 일에만 집중해 깔끔해진다. (중복 제거)

---

### ② Plan 작성 — Plan 서브에이전트 호출 ("어떻게" + 기존 코드 재활용 조사)

`/prd`가 확정되면, 확정된 **PRD를 `Read`로 읽어** 본문을 확보한 뒤
`Task` 도구로 **Plan 설계 서브에이전트**를 띄운다.

- `subagent_type`: `Plan`
- `description`: `"구현 Plan 설계"`
- `prompt`: 아래 내용을 채워서 전달한다. **핵심: "기존 코드를 먼저 조사하라"고 명시**한다.
  (Plan 에이전트는 내부적으로 Explore 등으로 코드를 탐색할 수 있다.)

```
당신은 구현 Plan 설계자다. 아래 PRD를 어떻게 구현할지 설계하라.

[중요] 새 코드를 제안하기 전에, 이 저장소의 기존 코드/스킬/유틸을 먼저 조사해
재활용할 수 있는 것을 찾아라. (OS.md 원칙 2: 재활용 우선)

[중요] PRD에는 사람이 확정한 내용과, "열린 질문"에 사람이 채운 "답변:"이 들어 있다.
사람이 확정한 내용을 **최우선으로 반영**하라. 비어 있는 항목만 합리적으로 가정하고,
그 가정을 Plan에 명시하라. (사람의 결정을 임의로 뒤집지 마라.)

PRD (확정본):
<②에서 읽은 확정 PRD 본문>

다음 형식의 마크다운으로만 답하라:
# Plan — <기능 이름>
## 재활용할 기존 코드 (조사 결과: 파일/스킬과 어떻게 쓸지)
## 손댈 파일 (신규/수정)
## 구현 단계 (순서대로)
## 테스트 전략 (단위/통합 — OS.md 원칙 3, 필수)
## 위험 요소 / 롤백
```

서브에이전트가 돌려준 Plan 본문을 **그대로** `docs/plan/<slug>.md` 에 `Write` 로 저장한다.

> 학습 하이라이트: **오케스트레이터가 Plan 서브를 부르고, Plan 서브가 다시 코드 조사를 위해
> 탐색하는 "중첩 협력"** 이 일어난다. AI가 AI를 부르고, 각자 다른 컨텍스트의 결과만 위로 올라온다.

---

### ③ 사람 검토 게이트 ★반드시 멈춤★

PRD(확정본)와 Plan을 만들었으면, **요약해서 사람에게 보고하고 STOP 한다.**

보고 형식:

```
✅ ① PRD 확정 → docs/prd/<slug>.md
✅ ② Plan 작성 완료 → docs/plan/<slug>.md

[Plan 요약]
- 재활용: ...
- 손댈 파일: ...
- 테스트: ...
- 위험: ...

여기는 검토 게이트입니다. Plan을 확인해 주세요.
- 수정이 필요하면 어디를 고칠지 알려주세요(②로 되돌아갑니다).
- 승인하시면 ④ 구현 단계로 넘어갑니다.
```

**이 게이트를 넘어 임의로 구현(코드 작성)으로 진행하지 않는다.** (OS.md 원칙 1 — 가장 중요한 안전장치)
사용자가 명시적으로 "승인", "구현 시작"이라고 말하기 전에는 ④로 가지 않는다.

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

스킬은 코드가 아니라 지시문이라 단위테스트 대상이 아니다. 대신 아래 체크리스트로 검증한다:

- [ ] `/feature "샘플 기능"` 실행 시 ① 단계가 **`/prd` 스킬 호출**로 진행된다(직접 PRD를 짜지 않는다).
- [ ] `/prd`의 게이트(요구사항 작성·PRD 확정)에서 **멈추고**, 확정 전 ②로 넘어가지 않는다.
- [ ] ② Plan 단계는 진행 전 **확정 PRD를 읽어** 그 내용을 반영한다.
- [ ] ② Plan 단계가 **별도의 `Task` 서브에이전트 호출**로 일어나고, 기존 코드 재활용 조사가 반영된다.
- [ ] 최종 산출물로 `docs/prd/<slug>.md` 와 `docs/plan/<slug>.md` 가 존재한다.
- [ ] ③ 게이트에서 **멈추고**, 사용자 승인 없이 구현으로 넘어가지 않는다.

---

## 안전 규칙

- **게이트를 넘지 않는다.** `/prd`의 게이트도, ③ 검토 게이트도 사람 승인 전에는 넘지 않는다.
- **모든 산출물은 저장소 안에** 둔다(`docs/requirements/`, `docs/prd/`, `docs/plan/`). (CLAUDE.md 규칙 1)
- 서브에이전트가 빈 결과/오류를 돌려주면 그대로 사용자에게 알리고 멈춘다. 추측으로 채우지 않는다.
- 이 스킬은 **읽기·문서 작성·하위 스킬 호출**만 한다. 코드 파일을 수정하지 않는다(그건 ④ 구현의 몫).
