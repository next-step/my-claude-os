# /feature — 기능 개발 오케스트레이터 (PRD → Plan → 게이트)

이 스킬은 [OS.md](../../../OS.md)의 개발 흐름을 **자동으로 굴리는 메인 에이전트(오케스트레이터)** 다.
직접 PRD·Plan을 쓰지 않고, **각 단계를 전담하는 서브에이전트를 `Task` 도구로 호출**해 협력시킨다.
그리고 OS.md의 가장 중요한 안전장치인 **③ 사람 검토 게이트에서 반드시 멈춘다.**

> 설계 원칙: 오케스트레이터는 "조율"만 한다. 실제 사고(PRD 작성, 설계, 코드 조사)는
> 각자 격리된 컨텍스트를 가진 서브에이전트에게 맡기고, 메인은 **결론만** 받아 다음 단계로 넘긴다.
> 이렇게 하면 메인의 컨텍스트가 깨끗하게 유지되고, 단계별 책임이 분리된다.

전달된 인자: ``
- 인자가 있으면 → 개발할 **기능 설명**으로 사용한다.
- 인자가 없으면 → 사용자에게 "어떤 기능을 개발할까요?"라고 한 줄로 묻고 시작한다.

이 스킬의 범위는 **①②③까지**다. ④ 구현·⑤ 마무리는 게이트 통과 후 별도 단계/스킬에서 한다.

---

## 진행 순서

### 0단계: 기능 이름(slug) 정하기

기능 설명에서 짧은 파일명용 slug를 정한다(한글 가능하나 영문 kebab-case 권장).
예: "다크모드 토글" → `dark-mode-toggle`. 산출물 경로는 아래와 같다.

```
docs/prd/<slug>.md     # ① PRD
docs/plan/<slug>.md    # ② Plan
```

`docs/prd/`, `docs/plan/` 디렉토리가 없으면 Write가 자동 생성하므로 미리 만들 필요는 없다.

---

### ① 단계: PRD 서브에이전트 호출 — "무엇을, 왜"

`Task` 도구로 **PRD 작성 서브에이전트**를 띄운다.

- `subagent_type`: `general-purpose`
- `description`: `"PRD 작성"`
- `prompt`: 아래 내용을 채워서 전달한다.

```
당신은 PRD(제품 요구사항) 작성 전문가다. 코드는 아직 보지 마라.
아래 기능에 대해 "무엇을 / 왜 / 기대 동작"만 정리하라.

기능 설명: <인자로 받은 기능 설명>

다음 형식의 마크다운으로만 답하라(설명 문장 없이 문서 본문만):
# PRD — <기능 이름>
## 문제 (왜 필요한가)
## 목표 (이 기능이 해결하는 것)
## 사용자 시나리오 (기대 동작)
## 범위 밖 (이번에 안 하는 것)
## 열린 질문 (사람에게 확인이 필요한 점)
[이 섹션은 사람이 파일에서 직접 답을 채울 수 있게 "답변 틀"을 포함해야 한다.]
맨 위에 다음 안내 문구를 그대로 넣어라:
> ⚠️ 각 질문의 "**답변:**" 줄을 사람이 이 파일에 직접 채웁니다.
> (채팅이 아니라 이 md 파일에 적습니다.) 비워두면 ② Plan이 합리적으로 가정합니다.

그리고 질문은 번호 목록으로 쓰되, **각 질문 바로 아래에 사람이 채울 빈 답변 줄을 반드시 둔다.** 예시 형식:
1. <질문 내용>
   - **답변:** _(여기에 작성)_
2. <질문 내용>
   - **답변:** _(여기에 작성)_
```

서브에이전트가 돌려준 PRD 본문을 **그대로** `docs/prd/<slug>.md` 에 `Write` 로 저장한다.

> 왜 코드를 안 보게 하나: PRD는 "문제 합의" 단계다. 구현 디테일에 끌려가지 않도록
> 일부러 시야를 요구사항에만 한정한다. (OS.md 원칙 1)

---

### ①.5 단계: 열린 질문 답변 게이트 — ★멈춤★

PRD를 저장했으면, **열린 질문에 사람이 답할 수 있도록 한 번 멈춘다.**
이 답변은 채팅이 아니라 **PRD md 파일 안의 "답변:" 줄에 사람이 직접 적는다.**

보고 형식:

```
✅ ① PRD 작성 완료 → docs/prd/<slug>.md

이 PRD에는 사람이 확인해야 할 "열린 질문"이 있습니다.
docs/prd/<slug>.md 파일을 열어, 각 질문 아래 "답변:" 줄을 직접 채워 주세요.
(비워두는 질문은 ② Plan 단계에서 제가 합리적으로 가정하고, 그 가정을 Plan에 명시합니다.)

답변을 다 적으셨으면 "다음", "Plan 진행"처럼 알려주세요. → ②로 넘어갑니다.
```

사용자가 진행하라고 하면, **PRD 파일을 `Read`로 다시 읽어 사람이 채운 답변까지 포함한 최신 PRD 본문**을 확보한 뒤 ②로 넘어간다.

> 왜 멈추나: 열린 질문은 "사람만 답할 수 있는 것"이다. 여기서 답을 받지 않고
> Plan으로 가면 추측이 굳어버린다. 답을 문서에 남기면 결정의 근거가 저장소에 함께 보존된다.

---

### ② 단계: Plan 서브에이전트 호출 — "어떻게" (+ 기존 코드 재활용 조사)

`Task` 도구로 **Plan 설계 서브에이전트**를 띄운다.

- `subagent_type`: `Plan`
- `description`: `"구현 Plan 설계"`
- `prompt`: 아래 내용을 채워서 전달한다. **핵심: 이 서브에이전트에게 "기존 코드를 먼저
  조사하라"고 명시**한다. (Plan 에이전트는 내부적으로 Explore 등으로 코드를 탐색할 수 있다.)

```
당신은 구현 Plan 설계자다. 아래 PRD를 어떻게 구현할지 설계하라.

[중요] 새 코드를 제안하기 전에, 이 저장소의 기존 코드/스킬/유틸을 먼저 조사해
재활용할 수 있는 것을 찾아라. (OS.md 원칙 2: 재활용 우선)

[중요] 아래 PRD의 "열린 질문"에는 사람이 직접 채운 "답변:"이 들어 있을 수 있다.
사람이 채운 답변을 **최우선으로 반영**하라. 답변이 비어 있는 항목만 합리적으로
가정하고, 그 가정을 Plan에 명시하라. (사람의 답변을 임의로 뒤집지 마라.)

PRD (사람이 열린 질문에 답변을 채운 최신본):
<①.5에서 다시 읽은, 답변이 채워진 PRD 본문>

다음 형식의 마크다운으로만 답하라:
# Plan — <기능 이름>
## 재활용할 기존 코드 (조사 결과: 파일/스킬과 어떻게 쓸지)
## 손댈 파일 (신규/수정)
## 구현 단계 (순서대로)
## 테스트 전략 (단위/통합 — OS.md 원칙 3, 필수)
## 위험 요소 / 롤백
```

서브에이전트가 돌려준 Plan 본문을 **그대로** `docs/plan/<slug>.md` 에 `Write` 로 저장한다.

> 여기가 이 스킬의 학습 하이라이트다: **메인(오케스트레이터)이 Plan 서브를 부르고,
> Plan 서브가 다시 코드 조사를 위해 탐색하는 "중첩 협력"** 이 일어난다.
> AI가 AI를 부르고, 각자 다른 컨텍스트에서 일한 결과만 위로 올라온다.

---

### ③ 단계: 사람 검토 게이트 — ★반드시 멈춤★

PRD와 Plan을 만들었으면, **요약해서 사람에게 보고하고 STOP 한다.**

보고 형식:

```
✅ ① PRD 작성 완료 → docs/prd/<slug>.md
✅ ② Plan 작성 완료 → docs/plan/<slug>.md

[PRD 요약]
- 문제: ...
- 목표: ...

[Plan 요약]
- 재활용: ...
- 손댈 파일: ...
- 테스트: ...
- 위험: ...

여기는 검토 게이트입니다. PRD/Plan을 확인해 주세요.
- 수정이 필요하면 어디를 고칠지 알려주세요(①②로 되돌아갑니다).
- 승인하시면 ④ 구현 단계로 넘어갑니다.
```

**이 게이트를 넘어 임의로 구현(코드 작성)으로 진행하지 않는다.** (OS.md 원칙 1 — 가장 중요한 안전장치)
사용자가 명시적으로 "승인", "구현 시작"이라고 말하기 전에는 ④로 가지 않는다.

---

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

스킬은 코드가 아니라 지시문이라 단위테스트 대상이 아니다. 대신 아래 체크리스트로 검증한다:

- [ ] `/feature "샘플 기능"` 실행 시 `docs/prd/<slug>.md` 와 `docs/plan/<slug>.md` 두 파일이 생성된다.
- [ ] ① PRD 단계와 ② Plan 단계가 각각 **별도의 `Task` 서브에이전트 호출**로 일어난다.
- [ ] ① PRD의 "열린 질문"에 **사람이 파일에서 직접 채울 "답변:" 빈 줄(틀)** 이 들어 있다.
- [ ] ①.5 답변 게이트에서 **멈추고**, 사용자가 진행 신호를 줄 때까지 ②로 넘어가지 않는다.
- [ ] ② Plan 단계는 진행 전 **PRD를 다시 읽어** 사람이 채운 답변을 반영한다(빈 항목만 가정).
- [ ] ② Plan 단계에서 기존 코드 조사(재활용)가 실제로 수행되어 Plan에 반영된다.
- [ ] ③ 게이트에서 **멈추고**, 사용자 승인 없이 구현으로 넘어가지 않는다.

---

## 안전 규칙

- **게이트를 넘지 않는다.** 사람의 승인 전에는 절대 ④ 구현으로 가지 않는다.
- **모든 산출물은 저장소 안에** 둔다(`docs/prd/`, `docs/plan/`). (CLAUDE.md 규칙 1)
- 서브에이전트가 빈 결과/오류를 돌려주면 그대로 사용자에게 알리고 멈춘다. 추측으로 채우지 않는다.
- 이 스킬은 **읽기·문서 작성**만 한다. 코드 파일을 수정하지 않는다(그건 ④ 구현의 몫).
