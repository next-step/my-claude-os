# /diagram — 검증된 다이어그램 (HTML → 이미지)

도식을 **디자인된 HTML/CSS로 만들고**, **헤드리스 Chrome으로 PNG 스크린샷**을 떠서 **직접 눈으로 보고**,
의도와 다르면 고쳐 다시 찍는 루프를 **맞을 때까지** 돈다. 최종 산출물은 **검증된 PNG 이미지**.

> 왜 HTML인가: mermaid 자동 레이아웃은 빠르지만 투박하다("허접"). HTML/CSS는 **색·카드·범례·배치**를
> 마음대로 디자인해 깔끔하게 뽑힌다. (스타일 참고: `docs/diagrams/os-flow.html`, `docs/feature-flow.html`)
> `/impl ↔ /verify` 루프를 시각물에 적용 — 만들고 → 보고 판정 → 고치고. "안 보고 넘기기" 금지.

전달된 인자: ``
- **무엇을** 그릴지(설명)와, 선택적으로 **출력 경로**. 없으면 `docs/diagrams/<slug>.html`(+`.png`).

## 진행 순서

### 1. HTML 작성
- 도식을 **HTML/CSS로 디자인**해 `<out>.html` 로 `Write`. 깔끔하게:
  - 노드=둥근 카드, 종류별 색 구분(사람/서브에이전트/훅/게이트 등), 화살표·라벨, 하단 범례.
  - body에 고정 너비(예: `width:1000px`)를 주면 스크린샷 크기를 잡기 쉽다.
  - 스타일이 막막하면 기존 `docs/diagrams/os-flow.html`·`docs/feature-flow.html`의 CSS를 참고/재사용.

#### 설계 원칙 — "흐름은 흐름답게" (납작함 금지) ⭐
> 가장 흔한 실패: **결정·분기·루프를 칩 한 줄로 납작하게 나열**해서, 그림인데 "표"처럼 보이는 것.
> 도식의 값어치는 **"무엇이 무엇으로 갈라지고 되돌아오는지"가 화살표로 한눈에 보이는 것**이다.

- **결정(분기)은 갈라지는 그림으로.** "A냐 B냐 C냐"는 한 노드에서 **여러 갈래로 fan-out** 시킨다.
  3갈래를 가로로 나열한 칩 3개로 끝내지 말고, **결정 노드 → 각 갈래(라벨 붙은 화살표) → 결과**로 그린다.
- **루프는 되돌아오는 화살표로.** "재시도/재구현/수정 반복"은 **뒤로 돌아가는 화살표(↺)** 로 그린다
  (점선+색으로 본류와 구분). "반복"이라고 글자로만 쓰지 않는다.
- **빠져나감(에스컬레이션·종료)은 본류에서 벗어나는 화살표로.** 통과→다음단계, 탈출→위 단계 등 **방향**이 보이게.
- **종류=색, 의미=라벨, 범례 필수.** 색이 무엇을 뜻하는지 범례로 못박는다.
- **도구 선택:** 단순 선형(A→B→C)은 CSS 카드로 충분하다. 하지만 **분기·루프백·합류가 있으면 인라인 `<svg>`**
  (`<marker>` 화살촉 + `<path>`)로 그린다 — 화살표 방향·곡선·되돌아옴을 정확히 통제할 수 있다.
  (예: `docs/diagrams/os-flow.html`의 ③ 구현 블록 — 디스패치 fan-out + 판정 분기 + 구현결함 루프백.)
- **한눈 우선:** 한 화면에서 시선이 **시작→끝으로 자연스럽게 흐르고**, 곁가지(분기·루프)가 본류와 즉시 구분돼야 한다.

### 2. 이미지로 뽑기 (screenshot)
헬퍼 스크립트로 PNG를 만든다.

```bash
bash .claude/scripts/screenshot-html.sh <out>.html <out>.png [width] [height]
```
- 스크린샷은 `width×height`(viewport)만큼 캡처한다. **내용이 더 길면 아래가 잘린다** → 3에서 확인해 height를 키운다.
- (첫 실행은 Chrome 기동에 몇 초.)

### 3. 확인 (verify) — ★핵심·생략 금지★ — "처음 보는 사람" 기준
**`Read`로 방금 만든 `<out>.png`를 직접 본다.** 단, 만든 사람(의도를 아는 나)이 아니라
**"이 그림을 처음 보는 사람"의 눈으로** 판단한다. 핵심 질문은 하나: **설명 없이, 한눈에 이해되는가?**

**A. 한눈 판단 (먼저·가장 중요)** — 사람이 봤을 때:
- **5초 안에 전체 흐름(시작 → 끝)이 읽히나?** 시선이 자연스럽게 흐르나, 어디부터 볼지 헤매나?
- **결정/분기가 "갈라지는 그림"으로 보이나?** 아니면 그냥 칩이 줄줄이 나열돼 "표처럼 납작"한가?
- **루프·되돌아감·에스컬레이션 같은 비선형 흐름이 화살표로 드러나나?** (글자로만 "반복"이라 쓰여 있진 않나?)
- **색·범례만 보고 종류 구분이 즉시 되나?** 무슨 색이 뭔지 헷갈리지 않나?
- **곁가지(분기·루프)가 본류와 한눈에 구분되나?**
> 위 중 하나라도 "아니오"면 — 잘림이 없어도 **실패**다. 1의 설계 원칙으로 돌아가 **흐름답게 다시 그린다.**

**B. 기술 점검 (그 다음)** — 한눈 판단을 통과했으면:
- 아래가 잘리지 않았나(범례까지 다 보이나)? 겹침·삐져나옴·글자 깨짐 없나?
- 노드·라벨·색이 의도와 맞나? 화살표 방향·연결이 맞나?

### 4. 분기
- **A(한눈)·B(기술) 둘 다 맞으면** → 종료(성공). 5로.
- **하나라도 틀리면** → HTML/CSS를 고쳐 **2로**. 특히 **A에서 "납작하다"면 분기·루프를 SVG 흐름도로 다시 그린다**
  (잘렸으면 height를 키워 다시 스크린샷). 최대 `4`회.

### 5. 보고
- 최종 **이미지(`<out>.png`)** 와 **소스(`<out>.html`)** 경로, 돈 라운드 수를 보고한다.
- 커밋할지는 사용자가 정한다.

> 빠른 초안만 필요하면 mermaid 헬퍼(`.claude/scripts/render-mermaid.sh`)도 있다(투박하지만 빠름).
> 기본은 위 HTML 경로 — 예쁘게 뽑고 검증한다.

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

- [ ] HTML/CSS로 디자인해 **스크린샷(PNG)** 으로 뽑는다.
- [ ] **분기·루프가 있으면 납작한 칩 나열이 아니라 흐름도(분기 fan-out·루프백 화살표, 필요시 SVG)** 로 그린다.
- [ ] PNG를 **`Read`로 직접 보고**, **"처음 보는 사람" 기준으로 "한눈에 이해되나"를 먼저** 판정한다(잘림·노드 점검은 그 다음).
- [ ] 한눈 판단에서 "납작하다/안 읽힌다"면 잘림이 없어도 **실패로 보고 다시 그린다.**
- [ ] 틀리거나 잘리면 HTML을 고쳐 **다시 스크린샷·확인**하는 루프가 돈다.
- [ ] 최대 횟수로 무한 루프를 막는다.
- [ ] 최종 이미지·소스 경로를 보고한다.

---

## 안전 규칙

- **"안 보고 통과" 금지.** 반드시 PNG를 `Read`로 본 뒤 "맞다"고 한다.
- **무한 루프 금지.** 최대 횟수 가드 필수.
- 스크린샷 실패(Chrome 오류 등)는 원인을 **그대로** 보고하고, 추측으로 "됐다"고 하지 않는다.
