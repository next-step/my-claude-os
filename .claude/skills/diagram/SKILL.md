# /diagram — 검증된 다이어그램 만들기 (렌더 ↔ 확인 루프)

도식을 **소스만 쓰고 끝내지 않는다.** PNG로 **렌더해서 직접 눈으로 보고**, 의도와 다르면 고쳐
다시 그리는 루프를 **맞을 때까지** 돈다. (예전 약점 = "안 보고 넘기기"를 구조로 막는다.)

> `/impl ↔ /verify` 루프를 **시각물**에 적용한 것. 만들고(구현) → 보고 판정(검증) → 고치고(재구현).
> 다루는 형식: **mermaid**(flowchart·sequence·class 등 — 렌더·검증이 깔끔한 종류).

전달된 인자: ``
- **무엇을** 그릴지(자연어 설명 또는 mermaid 소스)와, 선택적으로 **출력 경로**.
- 출력 경로가 없으면 → `docs/diagrams/<slug>.png` 를 제안한다(미리보기만 원하면 scratchpad).

## 진행 순서

### 1. mermaid 소스 작성
- 설명을 mermaid로 옮긴다(또는 받은 소스를 그대로 쓴다). `<out>.mmd` 로 `Write` 저장.
- 한글 라벨·`<br/>` 줄바꿈·화살표 라벨(`-->|텍스트|`) 사용 가능.

### 2. 렌더 (render)
헬퍼 스크립트로 PNG를 만든다.

```bash
bash .claude/scripts/render-mermaid.sh <out>.mmd <out>.png
```

- 렌더가 **실패하면**(문법 오류 등) 메시지를 보고 소스를 고쳐 다시 2를 한다.
- (첫 실행은 npx가 mermaid-cli를 받느라 수십 초 걸릴 수 있다.)

### 3. 확인 (verify) — ★핵심·생략 금지★
**`Read`로 방금 만든 `<out>.png`를 직접 본다.** 의도와 하나씩 대조한다:

- 노드·박스가 다 있나? 라벨 글자가 안 깨지고 안 잘렸나?
- 화살표 **방향·연결**이 맞나? 루프/분기 화살표가 의도대로인가?
- 레이아웃이 읽을 만한가(겹침·삐져나옴 없나)?

### 4. 분기
- **맞으면** → 루프 종료(성공). 5로 간다.
- **틀리면** → mermaid 소스를 고쳐 **2로 돌아간다**(다시 렌더·확인).
- **루프 안전장치**: 최대 `4`회. 도달하면 멈추고 "여기까지 됐고 남은 문제는 ~"라고 보고(무한 시도 금지).

### 5. 보고
- 최종 **이미지 경로**와 **소스(.mmd)**, 돈 라운드 수를 보고한다.
- 저장소에 둘지(커밋)·PR에 쓸지는 사용자가 정한다(이 스킬은 만들기·검증까지).

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

- [ ] 소스만 쓰고 끝내지 않고 **PNG로 렌더**한다.
- [ ] 렌더된 PNG를 **`Read`로 직접 보고** 의도와 대조해 판정한다(생략 금지).
- [ ] 틀리면 소스를 고쳐 **다시 렌더·확인**하는 루프가 돈다.
- [ ] 최대 횟수로 무한 루프를 막는다.
- [ ] 최종 이미지·소스 경로를 보고한다.

---

## 안전 규칙

- **"안 보고 통과" 금지.** 반드시 PNG를 `Read`로 본 뒤에 "맞다"고 한다.
- **무한 루프 금지.** 최대 횟수 가드 필수.
- 렌더 실패는 원인(문법 오류 등)을 **그대로** 보고하고, 추측으로 "됐다"고 하지 않는다.
- HTML 기반 도식(스윔레인 등)은 이 스킬 범위 밖이다(렌더·검증 경로가 다름).
