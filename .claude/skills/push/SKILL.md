# /push — 로컬 커밋을 원격에 반영

이미 커밋된 로컬 변경을 **GitHub(원격)에 푸시**하는 스킬입니다.

> 역할 분리: `/commit`은 로컬 기록까지만, `/push`는 외부 공개를 담당한다.
> push는 남들이 보게 되고 되돌리기 어려운 작업이므로, **무엇을 올리는지 먼저 보여주고** 진행한다.

전달된 인자: `$ARGUMENTS`
- 보통 비워서 호출한다 (현재 브랜치를 업스트림에 푸시).
- 특정 브랜치/리모트를 지정하고 싶을 때만 인자로 받는다.

---

## 진행 순서

### 1단계: 푸시 대상 파악 (병렬로 한 번에 실행)

아래 명령들을 **한 메시지에서 동시에** 실행해 "무엇을, 어디로" 올리는지 본다.

```bash
git status -sb                          # 현재 브랜치 + 업스트림 대비 ahead/behind
git log @{u}..HEAD --oneline 2>/dev/null # 원격에 아직 없는, 이번에 올라갈 커밋들
git remote -v                           # 어떤 원격에 올라가는가
```

확인할 것:
- **올릴 커밋이 있는가?** ahead 0이면 "원격에 이미 최신입니다" 알리고 종료.
- 업스트림이 설정돼 있는가? (없으면 `git log @{u}..` 가 에러 → 새 브랜치로 간주, 4단계 참고)
- behind 상태인가? (원격이 더 앞서 있으면 그냥 push하면 rejected → 3단계에서 처리)

### 2단계: 올라갈 내용 보고 (확인 단계)

1단계에서 본 **이번에 푸시될 커밋 목록과 대상 브랜치**를 사용자에게 한눈에 보여준다.

- 사용자가 처음부터 "푸시해줘"라고 명시적으로 호출했으므로 의도는 이미 분명하다 → 바로 3단계로 진행해도 된다.
- 다만 올라갈 커밋이 예상보다 많거나(여러 개), 보호 브랜치(`main`/`master`)거나, 민감해 보이면 한 번 짚고 넘어간다.

### 3단계: 푸시

```bash
git push                              # 업스트림이 이미 설정된 경우
```

- **rejected (behind) 인 경우**: 원격에 새 커밋이 있다는 뜻. 원인을 그대로 알리고,
  `git pull --rebase` 후 재시도할지 사용자에게 물어본다. **임의로 `--force`를 쓰지 않는다.**
- 인증 오류 등 다른 실패도 메시지를 그대로 전달하고 멈춘다.
- 강제 푸시(`--force` / `--force-with-lease`)는 사용자가 명시적으로 요청할 때만 사용한다.

### 4단계: 업스트림이 없는 새 브랜치

`git push`가 업스트림 미설정으로 막히면, 현재 브랜치를 origin에 연결하며 푸시한다.

```bash
git push -u origin <현재-브랜치>
```

### 5단계: 결과 보고

- 푸시된 브랜치와 원격, 올라간 커밋 수/해시를 보고한다.
- 원격이 GitHub면 PR을 만들지 물어볼 수 있다(선택).

---

## 안전 규칙

- push는 외부 공개다. **무엇이 올라가는지 먼저 보여준 뒤** 진행한다.
- `rejected`가 떠도 절대 임의로 force 하지 않는다. 원인을 알리고 사용자 판단을 받는다.
- 비밀키·토큰·`.env` 가 포함된 커밋이 올라가려 하면 멈추고 알린다. (한 번 push되면 원격 히스토리에 남는다)
