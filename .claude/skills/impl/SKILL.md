# /impl — 빌드 디스패처 (구현 ⇄ /verify 루프 제어)

[OS.md](../../OS.md) 개발 흐름의 **③ 구현 단계**를 **조율(제어)하는 한 디스패처**다.
입력은 **승인된 Plan**(`docs/plan/<slug>.md`)과 PRD, 출력은 **실제 코드 + 테스트**다.

> 역할 분리 — **판단은 /verify, 결정은 /impl.** 이 스킬은 직접 검증하지 않는다.
> 구현 일꾼(서브에이전트)과 `/verify` 일꾼에게 상황 따라 던지고(dispatch), `/verify`가 낸 **판정대로 분기**한다.
> 검증이 자라도(PR Checks·CI 등) `/verify`에만 들어가므로 이 스킬은 얇게 유지된다.

전달된 인자: ``
- 인자가 있으면 → 기능 설명/slug로 사용한다.
- 인자가 없으면 → "어떤 기능을 구현할까요?"라고 한 줄로 묻고 시작한다.

---

## 진행 순서

### [준비] Plan·PRD 확인 + 승인 확인

slug를 정하고 `Bash(test:*)`로 입력을 확인한다.

```
docs/prd/<slug>.md     # 있어야 함
docs/plan/<slug>.md    # 있어야 함 (= 설계 완료)
```

- **Plan이 없으면** → "먼저 `/plan`으로 설계하고 검토 게이트를 통과하세요"라고 안내하고 **멈춘다.**
- **있으면** → Plan·PRD를 `Read`로 읽고, **사람이 ② 승인 게이트에서 Plan을 승인했는지** 확인한다.
  명시적 승인("승인", "구현 시작") 없이 임의로 구현에 들어가지 않는다. (OS.md 원칙 1)

---

### 1. 구현 디스패치 — ★오케스트레이터 결정★

이 스킬은 오케스트레이터다. **"무엇을 구현할지" 뿐 아니라 "어떻게 디스패치할지"도 매 라운드 결정**한다.
(이게 핵심 — "어떤 일꾼을 어떻게 부를지"는 정적 규칙이 아니라 **이번 일의 모양에 맞춘 결정**이다.)

**1-a. 디스패치 방식 결정** — 이번 라운드 작업의 모양을 보고 고른다:

| 일의 모양 | 디스패치 | 왜 |
|-----------|----------|----|
| **큰 신규·광범위 구현** (여러 파일·새 동작) | **새 구현 에이전트** (`Task`, `general-purpose`) | 메인 컨텍스트 보호, 병렬·격리 |
| **작고 좁은 보완** (1~2파일·독립적·맥락 적음) | **메인에서 인라인** 또는 **새 좁은 에이전트**(필요한 것만 담아) | resume 안 함 |
| **직전 에이전트의 누적 맥락이 정말 필요한 연속 작업** | **같은 에이전트 resume** | 맥락 재구축 비용 > 재로딩 비용일 때만 |

> ⚠️ **resume 비용 주의**: 에이전트를 resume 하면 그 에이전트의 **전체 트랜스크립트가 통째로 재로딩**된다.
> 첫 구현으로 거대해진 에이전트를 **작은 보완 때문에 resume 하면, 1줄 고치는 데 수십~수백 k 토큰**을 쓴다.
> **원칙: resume 재로딩 비용이 변경 크기를 넘으면, resume 말고 좁게 새로 던지거나 인라인으로.**

**1-b. 결정한 방식으로 디스패치** — 무엇을 넘기든 공통:
- 승인된 Plan의 "구현 단계·손댈 파일·테스트 전략" + PRD 수용 기준,
- **단위/통합 테스트도 함께 작성**(OS.md 원칙 3), 기존 파일을 깨지 말 것.
- 에이전트면 `description: "구현"`. 재구현(구현결함 분기)이면 **검증자가 짚은 미충족 항목만** 고치게 한다.

---

### 2. 검증 강도 결정 — ★디스패처 판단★

`/verify`를 부르기 전에, **변경 위험도를 보고 강도를 정한다.**

| 강도 | 언제 |
|------|------|
| **light** (테스트만) | 동작 보존 리팩터링·기계적 변경(import·포매팅·rename)·문서/설정·**잘 덮인** 좁은 버그픽스 — 그리고 영향 영역 테스트가 잘 덮일 때 |
| **full** (테스트+적대적) | 새 기능·동작 변경·보안/인증·외부 입출력 — 또는 커버리지가 약할 때 |

- **애매하면 full**(안전한 쪽).
- "공개 API·시그니처 변경 / 보안·인증 / 성능·동시성 / 외부 입출력·스키마 변경"은 소규모여도 **full**.
- (이번 신규 구현처럼 새 동작이 많으면 full, 동작 보존 리팩터링이면 light.)

---

### 3. 검증 — `/verify` 호출

`Skill` 도구로 **`/verify`를 호출**한다(무엇을·강도 전달). `/verify`가 **판정(verdict)**을 돌려준다.
**이 스킬은 검증을 직접 하지 않는다** — 테스트 실행·기준 대조는 전부 `/verify`의 몫.

---

### 4. 분기 — ★판정대로★ (이 스킬의 핵심 = 결정)

`/verify`의 3갈래 판정에 따라:

- **통과** → 루프 종료(성공). 5로 간다.
- **구현결함** → 미충족 항목을 들고 **1(구현)로 돌아가** 고친 뒤 다시 2~3.
- **설계문제** → **루프를 탈출**한다. 구현을 더 돌리지 말고, "이건 구현이 아니라 설계 문제"라며
  **`/plan`(또는 `/prd`)으로 에스컬레이션**할 것을 사람에게 보고하고 멈춘다.

**루프 안전장치**
- **최대 `N`회**(기본 5) 도달 시 멈추고 남은 미충족과 함께 보고.
- **진척 없음 감지**: 같은 미충족이 2회 연속이면 멈춰 "막혔다"고 보고(무한 핑퐁 방지).
- 매 라운드 한 줄 진행 보고(예: `라운드 2/5 · 강도 full · 테스트 12/14 · 미충족 2`).

---

### 5. 보고 게이트 — ★멈춤★

루프가 끝나면 사람에게 보고하고 **멈춘다.**

```
✅ ③ 구현 완료 (검증 통과) — 라운드 R · 강도 light|full
또는  ↩ 설계문제로 에스컬레이션 — /plan 으로 (사유: ...)
또는  ⚠️ 상한/막힘 — 남은 미충족: ...

[구현 요약] 추가/수정 파일 · 테스트 N/M · /verify 최종 판정
다음은 ④ 마무리입니다. 확인 후 /commit · /push 로 진행하세요. (.gitignore에 비밀·DB 점검)
```

- 코드를 만들었으니 사람이 확인하도록 멈춘다. **임의로 커밋·푸시하지 않는다(④의 몫).**

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

- [ ] Plan이 없거나 승인 전이면 구현에 들어가지 않는다.
- [ ] **디스패치 방식을 일의 크기로 결정**한다(큰 신규=새 에이전트 / 작은 보완=인라인·좁은 에이전트 / 맥락필요=resume). 작은 보완에 큰 에이전트를 resume 하지 않는다.
- [ ] **검증은 직접 하지 않고 `/verify`를 호출**한다(판단 위임).
- [ ] 변경 위험도를 보고 **강도(light/full)를 정해** /verify에 전달한다.
- [ ] `/verify` 판정 **3갈래대로 분기**한다(통과 종료 / 구현결함 재구현 / 설계문제 에스컬레이션).
- [ ] 최대 횟수·진척 없음 안전장치로 **무한 루프에 빠지지 않는다.**
- [ ] 끝나면 사람에게 보고하고 멈춘다(자동 커밋·푸시 금지).

## 안전 규칙

- **판단은 /verify, 결정은 /impl.** 검증 로직을 이 스킬에 도로 넣지 않는다(얇게 유지). 디스패치 라우팅도 **결정 테이블**일 뿐 — 무거운 메타-에이전트를 만들지 않는다.
- **에이전트 규모를 변경 크기에 맞춘다.** 작은 보완에 큰 에이전트 resume 금지(전체 트랜스크립트 재로딩 = 낭비). resume 비용이 변경 크기를 넘으면 인라인/좁은 새 에이전트.
- **셀프 채점 금지.** 구현 일꾼이 자기 결과를 검증하지 않는다(검증은 /verify).
- **무한 루프 금지.** 최대 횟수·진척 없음 가드 필수. 설계문제는 루프 대신 에스컬레이션.
- **승인 전 구현 금지, 자동 커밋 금지.** 비밀·로컬 DB(.env, *.db)는 커밋 대상 아님.
- 서브에이전트/하위 스킬이 빈 결과·오류를 돌려주면 그대로 알리고 멈춘다.
