# /impl — 구현 ↔ 검증 루프 (④ 구현)

[OS.md](../../../OS.md) 개발 흐름의 **④ 구현 단계**를 담당하는 스킬이다.
입력은 **승인된 Plan**(`docs/plan/<slug>.md`)과 PRD다. 출력은 **실제 코드 + 테스트**다.

> 핵심 설계 — **스킬은 1개, 역할은 2개.** 구현과 검증은 서로 다른 일이지만, "검증을 통과할
> 때까지 반복"하는 **루프는 주인이 하나여야** 한다. 그래서 이 스킬이 루프를 소유하고, 그 안에서
> **구현 서브에이전트**와 **(매 라운드 새로 띄우는) 독립 검증 서브에이전트**로 역할을 나눈다.
> 검증자를 매번 신선하게 띄우는 이유: **셀프 채점 방지.** 구현한 쪽이 자기 결과를 채점하면
> "다 됐다"고 착각한다. 독립된 눈이 Plan·PRD 기준으로 적대적으로 대조해야 진짜 만족을 안다.

전달된 인자: ``
- 인자가 있으면 → 기능 설명/slug로 사용한다.
- 인자가 없으면 → "어떤 기능을 구현할까요?"라고 한 줄로 묻고 시작한다.

---

## 진행 순서

### [준비] Plan·PRD 확인 + 승인 확인

slug를 정하고 `Bash(test:*)`로 입력을 확인한다.

```
docs/prd/<slug>.md     # 있어야 함
docs/plan/<slug>.md    # 있어야 함 (= 설계 완료)
```

- **Plan이 없으면** → "먼저 `/plan`으로 설계하고 검토 게이트를 통과하세요"라고 안내하고 **멈춘다.**
- **있으면** → Plan과 PRD를 `Read`로 읽는다. 그리고 **사람이 ③ 게이트에서 Plan을 승인했는지**
  확인한다. 명시적 승인("승인", "구현 시작") 없이 임의로 구현에 들어가지 않는다. (OS.md 원칙 1)

---

### 1. 구현 — 구현 서브에이전트

`Task` 도구로 **구현 서브에이전트**를 띄워, 승인된 Plan의 "구현 단계"를 따르게 한다.

- `subagent_type`: `general-purpose`
- `description`: `"구현"`
- `prompt`에 담을 것:
  - Plan 본문(특히 "손댈 파일", "구현 단계", "테스트 전략")과 PRD의 수용 기준.
  - **단위/통합 테스트도 함께 작성**하라는 지시. (OS.md 원칙 3 — 테스트 없는 구현은 미완료)
  - 코드는 Plan의 디렉토리 구조를 따르고, 기존 파일을 깨지 않게 하라는 지시.

> 2회차 이후(검증 실패로 되돌아온 경우)에는, 같은 구현 에이전트를 **이어서 호출**해 이전 작업
> 맥락을 유지한 채 **검증자가 짚은 미충족 항목만** 고치게 한다(처음부터 다시 짜지 않는다).

---

### 2. 검증 루프 — ★이 스킬의 핵심★

아래를 **검증이 만족할 때까지, 단 최대 `N`회(기본 5회)까지** 반복한다.

**2-1. 독립 검증 서브에이전트 (매 라운드 새로 띄움)**

`Task` 도구로 **검증 서브에이전트**를 띄운다. **구현 에이전트와 다른, 신선한 컨텍스트**다.

- `subagent_type`: `general-purpose`
- `description`: `"구현 검증"`
- `prompt`에 담을 것:
  - "당신은 적대적 검증자다. 통과를 의심하라."
  - **객관 신호**: 테스트를 직접 실행하라(예: `pytest`). 통과/실패와 출력을 확인하라.
  - **기준 대조**: Plan의 수용 기준·테스트 전략, PRD의 시나리오를 **하나씩** 코드와 대조하라.
    빠진 기능·틀린 동작·테스트가 실제로 그 기준을 커버하지 못하는 곳·엣지케이스 누락을 찾아라.
  - 다음 형식으로만 보고: `## 판정: 만족 | 불만족` + `## 미충족 항목`(불만족이면 구체적으로,
    각 항목에 "무엇이/왜/어떻게 고칠지"). 없는 문제를 지어내지 마라.

**2-2. 판정**

- **테스트 통과 AND 검증자 "만족"** → 루프 종료(성공). 3단계로 간다.
- 그 외 → 검증자의 **미충족 항목**을 들고 **1번(구현)으로 돌아가** 고친 뒤 다시 2-1.

**2-3. 루프 안전장치 (무한 루프·낭비 방지)**

- **최대 `N`회**(기본 5)에 도달하면 멈추고, **남은 미충족 항목과 함께 사람에게 보고**한다.
- **진척 없음 감지**: 같은 미충족 항목이 2회 연속 그대로면, 더 돌리지 말고 멈춰 사람에게
  "여기서 막혔다"고 보고한다(에이전트끼리 무한 핑퐁 방지).
- 매 라운드 한 줄로 진행 상황을 알린다(예: `라운드 2/5 — 테스트 12/14 통과, 미충족 2건`).

---

### 3. 보고 게이트 — ★멈춤★

루프가 끝나면(성공이든 상한 도달이든) 사람에게 보고하고 **멈춘다.**

```
✅ ④ 구현 완료 (검증 통과) — 라운드 R회
또는
⚠️ ④ 구현 중단 (상한/막힘) — 남은 미충족: ...

[구현 요약]
- 추가/수정 파일: ...
- 테스트 결과: 통과 X / 전체 Y
- 검증자 최종 판정: 만족 / 불만족(사유)

다음은 ⑤ 마무리입니다. 결과를 확인하시고, 괜찮으면 /commit · /push 로 진행하세요.
(비밀·로컬 DB 파일이 커밋에 섞이지 않는지 확인 — .gitignore 점검)
```

- **코드를 만들었으니 사람이 결과를 확인하도록 멈춘다.** 임의로 커밋·푸시하지 않는다(⑤의 몫).

## 수용 기준 (이 스킬이 제대로 동작하는지 검증)

- [ ] Plan이 없으면 안내하고 멈춘다(Plan 없이 구현 시작 안 함).
- [ ] 사람의 Plan 승인 없이 구현에 들어가지 않는다.
- [ ] 구현과 검증이 **서로 다른 서브에이전트**로 일어난다(검증은 매 라운드 신선하게).
- [ ] 검증자는 **테스트를 실제로 실행**하고 Plan·PRD 기준과 대조해 만족/불만족을 판정한다.
- [ ] 불만족이면 미충족 항목을 구현에 피드백해 **루프**가 돈다.
- [ ] 최대 횟수·진척 없음 안전장치로 **무한 루프에 빠지지 않는다.**
- [ ] 끝나면 사람에게 보고하고 멈춘다(자동 커밋·푸시 금지).

---

## 안전 규칙

- **테스트 없이 "완료"라고 하지 않는다.** (OS.md 원칙 3)
- **셀프 채점 금지.** 구현한 에이전트가 자기 결과를 검증하지 않는다(검증은 독립 에이전트).
- **무한 루프 금지.** 최대 횟수·진척 없음 가드를 반드시 둔다.
- **비밀·로컬 산출물 보호.** `JWT_SECRET`·`.env`·`*.db` 등은 커밋 대상이 아니다(Plan의 gitignore 반영).
- **승인 전 구현 금지, 자동 커밋 금지.** 구현 시작은 사람 승인 후, 마무리(⑤)도 사람이 한다.
- 서브에이전트가 빈 결과/오류를 돌려주면 그대로 알리고 멈춘다. 추측으로 채우지 않는다.
