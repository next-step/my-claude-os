# Orchestrate — 기획자 ↔ 개발자 협업 루프

서브에이전트는 서로를 직접 호출하지 못한다. 그래서 **메인 Claude(너)가 유일한 오케스트레이터**다.
이 스킬은 네가 `product-planner`, `backend-developer`, `frontend-developer`를 **정해진 순서로 호출하고, 그들의 의견을 서로 중개**해 하나의 기능을 합의된 형태로 완성하는 절차다.

대상은 "기능 한 단위"다(예: 공고 피드, 회사 리서치 화면, 수집 파이프라인). 단순 질문 하나면 이 스킬을 쓰지 말고 해당 에이전트 1개만 호출한다.

## 시작 전
- 사용자가 만들려는 **기능/목표를 한 문장으로 확정**한다. 모호하면 1~2개만 짧게 되묻는다.
- 프로젝트 루트 `OS.md`가 단일 진실 출처임을 기억한다. 모든 에이전트가 이걸 전제로 일한다.

## 절차

### 1단계 — 기획 (planner)
`product-planner`를 호출해 다음을 받는다: 전체 구조 안에서 이 기능의 위치, 필요한 스택/도구, **백엔드·프론트엔드 작업 분배**, 그리고 둘 사이의 **계약(API 형태·데이터 스키마 합의점)**.
- 기획자에게 "백엔드와 프론트가 병렬로 일할 수 있도록 인터페이스를 명확히 해달라"고 요구한다.
- 결과를 네가 짧게 요약해 사용자에게 보여준다(중간 가시성 확보).

### 2단계 — 개발 (backend + frontend 병렬)
기획자가 정한 틀을 **그대로 전달**하며 `backend-developer`와 `frontend-developer`를 **한 메시지에서 동시에**(병렬) 호출한다.
- 각 개발자에게 명확히 지시한다: "이 기획 틀 안에서 구현하되, **막히거나 더 나은 길이 보이면 제약과 대안을 반드시 보고하라**"(각 에이전트 프롬프트의 '기획자와의 관계' 규칙대로).
- 두 결과를 받아, 각각 ① 만든 것 ② **기획에 대한 이견/제약이 있는지**를 분리해 정리한다.

### 3단계 — 중재 (제약이 올라오면 planner로 되돌림)
2단계에서 개발자가 제약·이견을 냈는지 확인한다.
- **이견이 없으면** → 4단계로.
- **이견이 있으면** → 그 제약을 **구조화해서 `product-planner`에게 다시 전달**한다. 전달할 때 반드시 포함: 어느 개발자가 / 무엇이 왜 막히는지 / 제안한 대안 / 영향받는 기획 범위.
  - 기획자는 (프롬프트대로) 방어하지 말고 판단한다 → 기획 수정 / 기존 유지(근거 제시) 중 하나를 돌려준다.
  - **핵심 가치·1차 범위·수집 폴백 전략**에 영향 주는 변경이면, 기획자가 사용자 확인을 요청할 것이다 → 그 트레이드오프를 **사용자에게 그대로 전달하고 답을 받는다.**
  - 기획이 수정됐으면 → 바뀐 부분만 해당 개발자에게 다시 전달해 반영시킨다(2단계로 부분 복귀). 합의될 때까지 반복하되, **최대 2~3라운드**를 넘기면 멈추고 사용자에게 현재 쟁점을 정리해 결정을 요청한다(무한 핑퐁 방지).

### 4단계 — 종합
합의된 결과를 하나로 모아 사용자에게 보고한다.
- 무엇을 만들었는지(백엔드/프론트 각각), 기획이 도중에 어떻게 바뀌었는지(있었다면), 다음 단계.
- 기획·스택의 **확정된 결정**이 OS.md에 남아야 하면, 기획자가 갱신하도록 했는지 확인한다(안 됐으면 갱신 요청).

## 원칙
1. **너는 중재자이지 결정권자가 아니다.** 기획 판단은 planner, 구현 판단은 개발자에게 맡기고, 너는 메시지를 **정확히·구조화해서** 나른다.
2. **각 단계 결과를 사용자에게 짧게 가시화**한다. 통째로 삼키고 마지막에만 보고하지 않는다.
3. **가치에 영향 주는 결정은 반드시 사용자에게.** 에이전트끼리 합의했어도 핵심 가치/범위가 흔들리면 사용자 확인이 우선이다.
4. **루프에 상한을 둔다.** 같은 쟁점으로 2~3라운드를 넘기면 사람에게 결정을 넘긴다.
5. 병렬로 돌릴 수 있는 호출(2단계 backend+frontend)은 한 메시지에서 동시에 보낸다.
