# Plan 스킬

기획 요소(인자)가 있으면 **지시 모드**, 없으면 **자율 모드**로 동작한다.
어떤 모드든 기획자 에이전트 → 디자이너 에이전트 → 개발자 에이전트 순으로 흐른다.

---

## 모드 판별

스킬 호출 시 인자(args)가 있는지 확인한다.

| 상황 | 모드 |
|------|------|
| `/plan XP 시스템 추가해줘` | 지시 모드 |
| `기획자한테 스트릭 기능 넘겨줘` | 지시 모드 |
| `/plan` (인자 없음) | 자율 모드 |
| `기획 검토해줘` | 자율 모드 |

---

## Step 1: 기획자 에이전트 실행

Agent 도구로 `planner` 에이전트를 실행한다.

### [지시 모드] 프롬프트 템플릿

```
아래 기획 요소를 검토하고 PRD를 작성해줘.

기획 요소: {사용자가 전달한 내용}

작업 순서:
1. habit-tracker/docs/planning/POLICY.md 를 읽고 이 기획 요소가 현재 정책과 어떻게 연결되는지 파악해
2. 정책 추가/수정이 필요하면 POLICY.md를 직접 수정하고 변경 이력을 기록해
3. habit-tracker/docs/planning/prd-[기능명].md 형식으로 PRD를 작성해
4. PRD 완성 후, 아래 두 가지를 명확하게 출력해:
   a. [디자이너 지시] — 디자이너 에이전트에게 전달할 디자인 브리프 (전체 내용)
   b. [개발자 지시] — 백엔드/프론트 에이전트에게 전달할 작업 목록 (전체 내용)
   c. [정책 변경 여부] — POLICY.md 수정 여부와 변경 섹션
```

### [자율 모드] 프롬프트 템플릿

```
현재 앱 정책과 로드맵을 검토하고, 다음에 추가하면 좋을 기능을 스스로 골라서 기획해줘.

작업 순서:
1. habit-tracker/docs/planning/POLICY.md 를 읽고 Phase별 로드맵 현황을 파악해
2. habit-tracker/docs/api/endpoints.md 와 구현 파일들을 확인해서 현재 코드 상태를 파악해
3. 로드맵 우선순위와 게이미피케이션 가치를 기준으로 지금 추가할 기능 1개를 선택해
4. 선택 이유를 한 문장으로 설명한 뒤, 해당 기능의 PRD를 habit-tracker/docs/planning/prd-[기능명].md 로 작성해
5. PRD 완성 후, 아래 두 가지를 명확하게 출력해:
   a. [디자이너 지시] — 디자이너 에이전트에게 전달할 디자인 브리프 (전체 내용)
   b. [개발자 지시] — 백엔드/프론트 에이전트에게 전달할 작업 목록 (전체 내용)
   c. [정책 변경 여부] — POLICY.md 수정 여부와 변경 섹션
```

---

## Step 2: 기획자 결과 파싱

기획자 에이전트 결과에서 세 블록을 추출한다:

- `[디자이너 지시]` 블록 → Step 3에서 디자이너 에이전트로 전달
- `[개발자 지시]` 블록 → Step 4에서 백엔드/프론트 에이전트로 전달
- `[정책 변경 여부]` 블록 → 사용자에게 요약 보고 시 포함

---

## Step 3: 디자이너 에이전트 실행

기획자의 `[디자이너 지시]` 블록을 그대로 포함해 `designer` 에이전트를 실행한다.

```
기획자로부터 아래 디자인 브리프를 받았어. 화면 명세서를 작성해줘.

{[디자이너 지시] 블록 전체}

작업 완료 후:
- 화면 명세서 파일 경로
- 주요 컴포넌트 목록
- 백엔드/프론트가 주의할 디자인 결정사항
을 출력해줘.
```

---

## Step 4: 개발자 에이전트 실행

디자이너 결과가 나온 뒤, 기획자의 `[개발자 지시]` 블록을 기반으로 개발자 에이전트를 실행한다.

**백엔드와 프론트엔드 에이전트 중 실제로 변경이 필요한 쪽만 실행한다.**
기획자 지시에 "API 추가 없음" 또는 "기존 API 그대로 사용" 이면 백엔드는 건너뛴다.

### 백엔드 에이전트 프롬프트

```
기획자로부터 아래 개발 지시를 받았어. 백엔드 구현을 진행해줘.

{[개발자 지시] 블록 중 Backend 관련 내용}

디자인 명세서: {Step 3에서 나온 파일 경로}

TDD 원칙에 따라 테스트 먼저 작성하고, 완료 후 보고해줘.
```

### 프론트엔드 에이전트 프롬프트

```
기획자로부터 아래 개발 지시를 받았어. 프론트엔드 구현을 진행해줘.

{[개발자 지시] 블록 중 Frontend 관련 내용}

디자인 명세서: {Step 3에서 나온 파일 경로}
API 계약서: habit-tracker/docs/api/endpoints.md

완료 후 변경된 템플릿 파일 경로와 주요 인터랙션을 보고해줘.
```

---

## Step 5: qa-engineer 에이전트 실행

프론트엔드 구현이 완료된 뒤 `qa-engineer` 에이전트를 실행한다.

> **공유 에이전트**: `qa-engineer`는 `/qa` 스킬에서도 독립적으로 호출된다.
> 이 단계에서는 기능명을 그대로 전달해 자동으로 범위를 잡는다.

```
아래 기능에 대해 QA를 진행해줘.

기능명: {기획자가 결정한 기능명}

작업 순서:
1. 관련 구현 파일 탐색 (백엔드·프론트 에이전트가 커밋한 파일 기준)
2. habit-tracker/docs/planning/prd-[기능명].md 의 수용 기준(AC) 확인
3. habit-tracker/docs/design/specs/ 의 UX 체크리스트 확인
4. 코드 정적 분석으로 버그 선제 발굴 (엣지 케이스, 경계값, null, 동시성)
5. 테스트 케이스 작성 → habit-tracker/docs/qa/test-cases/[기능명].md
6. 버그 발견 시 버그 리포트 작성 → habit-tracker/docs/qa/bug-reports/[기능명]-bugs.md
7. 작업 완료 후 직접 커밋
```

모든 에이전트 실행이 끝나면 사용자에게 아래 형식으로 보고한다.

```

모드: [지시 모드 / 자율 모드]

─────────────────────────────

─────────────────────────────
PRD: habit-tracker/docs/planning/prd-[기능명].md
정책 변경: [POLICY.md #섹션 수정 / 변경 없음]

─────────────────────────────
🎨 디자이너
─────────────────────────────
화면 명세서: [파일 경로]
주요 컴포넌트: [목록]

─────────────────────────────

─────────────────────────────
[구현 파일 목록 / 건너뜀]

─────────────────────────────
🖥️  프론트엔드
─────────────────────────────
[구현 템플릿 목록 / 건너뜀]
```

## 실행 원칙

- Step 3(디자이너) → Step 4(개발자) → Step 5(QA) 순서는 **반드시 직렬**로 실행한다.
  앞 단계 산출물이 다음 단계의 입력이기 때문이다.
- 에이전트가 오류를 반환하면 사용자에게 즉시 알리고 다음 단계로 넘어가지 않는다.
- 기획자가 정책 변경을 했다면, 최종 보고에서 변경된 정책 내용을 명시적으로 강조한다.
