# QA 스킬

기능명(인자)이 있으면 **지정 모드**, 없으면 **자동 감지 모드**로 동작한다.
qa-engineer 에이전트를 실행해 테스트 케이스 작성 → 버그 발굴 → 버그 리포트까지 처리한다.

> **공유 에이전트**: `qa-engineer`는 이 스킬과 `/plan` 스킬 양쪽에서 호출된다.
> `/plan` 마지막 단계에서 개발 완료 직후 QA가 자동 실행되고,
> `/qa`로는 독립적으로 원하는 기능만 골라서 QA할 수 있다.

## 모드 판별

| 상황 | 모드 |
|------|------|
| `/qa XP 시스템` | 지정 모드 |
| `QA해줘 스트릭 기능` | 지정 모드 |
| `/qa` (인자 없음) | 자동 감지 모드 |
| `QA해줘` | 자동 감지 모드 |

## Step 1: 범위 결정

### [지정 모드]
인자로 받은 기능명을 그대로 qa-engineer에게 전달한다.

### [자동 감지 모드]
qa-engineer 실행 전 아래 명령으로 최근 변경 범위를 파악한다:

```bash
# 최근 커밋에서 변경된 파일 목록 확인
git log --oneline -5
git diff HEAD~1 --name-only
```

변경된 파일 경로에서 기능명을 추론한다:
- `domain/Xp*.java` → "XP 시스템"
- `domain/Streak*.java` → "스트릭 시스템"
- `templates/index.html` → "홈 화면"
- 여러 기능이 섞인 경우 → 가장 최근 커밋 메시지를 기능명으로 사용

---

## Step 2: qa-engineer 에이전트 실행

### [지정 모드] 프롬프트 템플릿

```
아래 기능에 대해 QA를 진행해줘.

기능명: {인자로 받은 기능명}

작업 순서:
1. 관련 구현 파일 탐색
   - habit-tracker/src/main/java 에서 관련 도메인/서비스/컨트롤러 찾기
   - habit-tracker/src/main/resources/templates 에서 관련 화면 찾기
2. habit-tracker/docs/planning/prd-[기능명].md 가 있으면 수용 기준(AC) 확인
3. habit-tracker/docs/design/specs/ 에서 UX 체크리스트 확인
4. 코드 정적 분석으로 버그 선제 발굴 (엣지 케이스, 경계값, null, 동시성)
5. 테스트 케이스 작성 → habit-tracker/docs/qa/test-cases/[기능명].md
6. 버그 발견 시 버그 리포트 작성 → habit-tracker/docs/qa/bug-reports/[기능명]-bugs.md
7. 작업 완료 후 직접 커밋
```

### [자동 감지 모드] 프롬프트 템플릿

```
최근 변경된 기능을 분석하고 QA를 진행해줘.

분석 대상 파일:
{Step 1에서 파악한 변경 파일 목록}

추정 기능명: {추론한 기능명}

작업 순서:
1. 위 파일들을 읽고 무엇이 구현됐는지 파악
2. habit-tracker/docs/planning/ 에서 관련 PRD 찾아 수용 기준(AC) 확인
3. habit-tracker/docs/design/specs/ 에서 UX 체크리스트 확인
4. 코드 정적 분석으로 버그 선제 발굴 (엣지 케이스, 경계값, null, 동시성)
5. 테스트 케이스 작성 → habit-tracker/docs/qa/test-cases/[기능명].md
6. 버그 발견 시 버그 리포트 작성 → habit-tracker/docs/qa/bug-reports/[기능명]-bugs.md
7. 작업 완료 후 직접 커밋
```

## Step 3: 결과 보고

qa-engineer 에이전트 완료 후 사용자에게 아래 형식으로 보고한다.

```
✅ QA 완료: [기능명]

모드: [지정 모드 / 자동 감지 모드]

─────────────────────────────
🧪 테스트 케이스
─────────────────────────────
파일: docs/qa/test-cases/[기능명].md
총 [N]개 — 정상 [N]개 / 엣지 케이스 [N]개

─────────────────────────────
🐛 버그 리포트
─────────────────────────────
[버그 없음 / 파일: docs/qa/bug-reports/[기능명]-bugs.md]
심각도별: 치명적 [N]건 / 높음 [N]건 / 낮음 [N]건

─────────────────────────────
다음 단계
─────────────────────────────
[버그 있음] → "백엔드/프론트 개발자로 버그 수정해줘"
[버그 없음] → "커밋해줘" 또는 다음 기능 개발 진행
```

---

## 실행 원칙

- qa-engineer는 코드를 **수정하지 않는다** — 발견만 하고 리포트를 남긴다
- 버그가 없을 때도 테스트 케이스 문서는 반드시 작성한다
- `/plan` 파이프라인 마지막에서 호출될 때는 Step 1을 생략하고 기능명을 그대로 전달받는다
