# Interview 스킬 (의도 추출)

모호한 생각을 **질문으로 구체화**한다. "어떻게 할까"를 결정 가능한 선택으로 바꿔,
다음 단계(설계·구현)가 딛고 설 구체적 의도를 만든다.

## 언제 쓰나

- 여러 유효한 기술·구현 방법 중 하나를 골라야 할 때 (work-principles가 여기로 보냄).
- 유지보수에서 티켓·요구의 **범위가 모호**할 때 (maintenance-mode가 여기로 보냄).
- 막연한 아이디어를 뭘·왜·어디까지 만들지로 구체화할 때.
- 요구사항을 뽑아 설계·구현으로 넘겨야 할 때 (신규 기능 기획 등).

## 무엇을 묻나 (의도 축)

| 축 | 캐는 질문 | 드러내는 것 |
|---|---|---|
| 왜·진짜 문제 | "이게 되면 뭐가 좋아져요?" | 요청한 해법 뒤의 목적 |
| 성공 기준 | "뭘 보면 됐다고 해요?" | 목표를 측정 가능하게 |
| 범위·영향 | "어디까지 하고 뭐가 영향받아요?" | 경계 |
| 제약 | "언제까지, 어떤 한계 안에서요?" | 시간·리소스·정책 |
| 우려·리스크 | "뭐가 걱정돼요? 뭐가 깨지면 안 돼요?" | 금지·숨은 요구 |
| 우선순위 | "다 못하면 뭐부터요?" | 트레이드오프·택1 |

파급 큰 축부터. **"왜"를 맨 먼저** — 표면 요청("X 만들어줘")을 곧이곧대로 받지 않는다.

## 방법 (반드시 지킬 것)

1. **한 번에 하나씩.** 질문을 쏟지 않는다. 한 질문 → 답 → 반영 → 다음.
2. **발산 먼저, 수렴 나중.** 축을 모를 땐 열린 질문으로 캔다("뭐가 걱정돼요?" — 선택지로
   못 가두니 자유 서술로 받음). 축이 서면 배타적 선택지+트레이드오프+추천으로 택일한다.
3. **답을 되비춰 확인.** 확정된 걸 "제가 이해하기론 X, 맞아요?"로 한 줄 되짚고 넘어간다.
   추상적이면 "예를 들면?"으로 구체화한다.
4. **멈출 때를 안다.** 핵심 모호함이 풀리면 끝낸다. 사소한 건 합리적 기본값으로 채우고
   그 사실을 알린다. (work-principles "가급적 묻고 시킨 것만"과 정합)

> 수렴 질문은 AskUserQuestion(선택지)으로, 발산 질문은 자유 서술로 받는다. 진짜 독립적인
> 축만 한 번에 묶고, 아니면 하나씩.

## 절차

1. 모호한 입력을 받아, 안 정해진 것을 위 축으로 뽑는다.
2. 파급 큰 축부터 하나씩 묻는다(발산 → 수렴).
3. 각 답을 반영하고, 필요하면 축을 더하거나 지운다.
4. 핵심이 다 서면 아래로 정리해 반환한다.

## 산출물 (구체화 결과)

```markdown
## 구체화 결과 — <주제>
- 목표: (무엇을 왜)
- 범위: (이번에 하는 것 / 안 하는 것)
- 결정: (축별로 택한 것 + 한 줄 이유)
- 미정·가정: (기본값으로 채운 것, 더 확인할 것)
- 다음 단계: (설계·구현으로 넘길 것)
```

> 이 결과(구체화된 의도·결정)가 다음 단계 — 택1 확정·design-doc·구현 — 의 입력이 된다.
