# /commit — 변경사항 커밋 (푸시는 확인 후)

로컬 변경사항을 의미 있는 단위로 **커밋**하는 스킬입니다.
커밋은 "로컬에 기록"하는 가벼운 작업이라 이 스킬의 본 역할이고,
**푸시는 외부(GitHub)에 공개하는 무거운 작업**이라 커밋 이후에 한 번 더 확인하고 진행합니다.

> 설계 원칙: 이름(`commit`)과 동작을 일치시킨다. `/commit`은 기본적으로 **로컬에서 멈춘다.**
> push는 되돌리기 어려운 외부 반영이므로, 사용자의 명시적 동의가 있을 때만 한다.

전달된 인자: `$ARGUMENTS`
- 인자가 있으면 → 커밋 메시지로 사용 (사용자 의도 우선)
- 인자가 없으면 → 변경 내용을 분석해 메시지를 직접 작성
- 인자에 "푸시"/"올려줘"/"push" 같은 말이 포함되면 → 사용자가 푸시까지 원한 것이므로 5단계에서 따로 묻지 않고 진행

---

## 진행 순서

### 1단계: 현재 상태 파악 (반드시 병렬로 한 번에 실행)

아래 3개 명령을 **한 메시지에서 동시에** 실행해 전체 그림을 본다.

```bash
git status              # 무엇이 변경/스테이징 되었는가
git diff HEAD           # 실제 변경 내용 (staged + unstaged)
git log --oneline -5    # 기존 커밋 메시지 스타일(언어/형식) 참고
```

확인할 것:
- 커밋할 변경이 있는가? (없으면 "커밋할 변경사항이 없습니다" 알리고 종료)
- 현재 브랜치가 어디인가? (`main`/`master`면 아래 2단계 주의)
- 기존 커밋들이 한국어인가 영어인가, 접두사(`docs:`, `feat:` 등)를 쓰는가

### 2단계: 브랜치 확인 (보호 장치)

현재 브랜치가 `main` 또는 `master`이고 사용자가 명시적으로 "main에 커밋해"라고 하지 않았다면,
가볍게 한 번 확인한다. (실수로 보호 브랜치에 작업하는 것 방지)

> 이 프로젝트는 학습용이라 main 작업이 일반적일 수 있다. 사용자가 main을 쓰고 있던 흐름이면
> 굳이 막지 말고 가볍게 한 번만 확인한다.

### 3단계: 스테이징

- 의도한 변경만 담는다. 관련 없는 파일이 섞여 있으면 사용자에게 알린다.
- 보통 `git add -A` 로 전체를 담되, `git status`에서 본 무관한 파일(예: 빌드 산출물, `.idea` 등)이 있으면 골라 담는다.
- `.gitignore`에 들어가야 할 파일이 스테이징되려 하면 짚어준다.

### 4단계: 커밋 (← 이 스킬의 핵심)

- 인자로 메시지를 받았으면 그대로 사용한다.
- 아니면 **변경 내용(diff)을 근거로** 메시지를 작성한다. "왜" 바뀌었는지가 드러나게.
- **기존 커밋 스타일을 따른다** (1단계의 `git log`에서 본 언어·접두사 형식).
- 커밋 메시지 끝에 아래 서명을 붙인다:

```
Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
```

여기까지가 `/commit`의 기본 완료 지점이다. **로컬 커밋이 끝났으면 일단 멈춘다.**

### 5단계: 푸시 — 확인 후에만

커밋 결과(해시·메시지)를 보고한 뒤, **푸시할지 사용자에게 묻는다.**

- 사용자가 처음부터 "푸시까지", "올려줘", "push"라고 했으면 → 묻지 않고 바로 진행
- 그 외에는 → "방금 커밋을 GitHub에 푸시할까요?"라고 한 번 확인하고, 동의할 때만 진행

```bash
git push                              # 업스트림이 이미 설정된 경우
git push -u origin <현재-브랜치>       # 새 브랜치라 업스트림이 없는 경우
```

- 푸시 실패 시(예: `rejected`, 인증 오류) 원인을 그대로 사용자에게 전달하고 멈춘다.
- 강제 푸시(`--force`)는 사용자가 명시적으로 요청하지 않는 한 절대 쓰지 않는다.

### 6단계: 결과 보고

- 커밋 해시, 메시지를 보고한다. (푸시했다면 푸시된 브랜치도 함께)
- 원격이 GitHub면 PR을 만들지 물어볼 수 있다(선택).

---

## 안전 규칙

- **커밋은 로컬, 푸시는 외부.** 푸시는 되돌리기 어려운 외부 반영이므로 사용자 동의가 있을 때만 한다.
- 비밀키·토큰·`.env` 같은 민감 파일이 스테이징에 잡히면 멈추고 알린다.
- 사용자가 보지 못한 변경을 임의로 커밋하지 않는다. 무관한 변경이 섞였으면 먼저 확인한다.
