# Git Commit

변경사항을 검토하고, 관련 파일을 스테이징한 뒤, 명확한 커밋 메시지로 커밋을 만든다.
**push는 하지 않는다.** 커밋 생성까지만 수행하고, 이후 push는 사용자가 직접 한다.

## 절차

1. **현재 상태 파악** — 아래 명령을 한 번에(병렬로) 실행해 무엇이 바뀌었는지 본다.
   - `git status` — 추적/미추적 파일 확인
   - `git diff` — 스테이징되지 않은 변경 내용
   - `git diff --staged` — 이미 스테이징된 변경 내용
   - `git log --oneline -10` — 이 저장소의 커밋 메시지 스타일 참고

2. **스테이징** — 이번 커밋에 포함할 변경을 선택해 `git add`로 스테이징한다.
   - 사용자가 특정 파일만 지정하면 그것만 추가한다.
   - 지정이 없으면, 하나의 논리적 단위로 묶이는 변경을 추가한다.
   - 관련 없는 변경이 섞여 있으면 사용자에게 어떻게 나눌지 확인한다.
   - `.env`, 비밀키, 대용량 빌드 산출물 등 커밋하면 안 되는 파일이 보이면 멈추고 알린다.

3. **메시지 작성** — 무엇을 했는지가 아니라 **왜** 했는지가 드러나게 쓴다.
   - `git log`에서 본 기존 스타일(언어, 형식, prefix 등)을 따른다.
   - 제목 줄은 간결하게(약 50자 이내) 명령형으로. 본문이 필요하면 한 줄 띄우고 적는다.
   - 이 저장소 커밋이 한국어면 한국어로, 영어면 영어로 맞춘다.

4. **커밋 실행** — heredoc으로 메시지를 전달해 줄바꿈을 보존한다.

   ```bash
   git commit -m "$(cat <<'EOF'
   제목 줄

   필요하면 본문.
   EOF
   )"
   ```

5. **결과 확인** — `git status`로 커밋이 정상 생성됐는지 확인하고, 커밋 해시와 한 줄 요약을 사용자에게 보고한다. push가 필요하면 사용자가 직접 하도록 안내한다.

## 주의

- `git add -A`나 `git add .`를 무심코 쓰지 말 것 — 의도치 않은 파일이 들어갈 수 있다. 무엇이 스테이징되는지 항상 확인한다.
- 이미 push된 커밋을 다시 쓰는 `--amend`, `rebase` 등은 사용자가 명시적으로 요청할 때만 한다.
- 변경사항이 전혀 없으면 커밋하지 말고 그대로 알린다.
- 커밋 메시지에 사용자가 요청하지 않은 서명/공동작성자 표기를 임의로 추가하지 않는다.
