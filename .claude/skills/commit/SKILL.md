# Commit

스테이징된(또는 변경된) 코드를 분석해 의미 있는 단위로 커밋한다.

## 절차

1. **현재 상태 파악** — 다음을 병렬로 실행한다.
   - `git status` — 추적/미추적 파일 확인
   - `git diff HEAD` — 변경 내용 확인 (스테이징 + 미스테이징)
   - `git log --oneline -10` — 이 레포의 커밋 메시지 스타일(언어, 형식) 확인

2. **스테이징** — 사용자가 일부 파일만 지정하지 않았다면, 관련 변경을 `git add` 한다.
   - 논리적으로 별개인 변경이 섞여 있으면 분리해서 여러 번 커밋할지 제안한다.
   - `.env`, 비밀키, 대용량 산출물 등 커밋하면 안 되는 파일이 포함됐는지 확인하고, 있으면 멈추고 알린다.

3. **메시지 작성** — Conventional Commits 형식을 따른다.
   ```
   <type>: <제목 (한글, 50자 이내, 명령형)>

   <본문 - 무엇을 왜 바꿨는지. 선택>
   ```
   - type: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`
   - 기존 레포 커밋이 다른 스타일(예: 영어, prefix 없음)이면 **그 스타일을 따른다**.
   - 제목은 "무엇을 했다"가 아니라 "무엇을 한다" 형태의 명령형으로.

4. **커밋 실행** — HEREDOC으로 메시지를 전달한다.
   ```bash
   git commit -m "$(cat <<'EOF'
   feat: 사용자 로그인 기능 추가

   - JWT 기반 인증 미들웨어 구현
   - 로그인 실패 시 에러 응답 처리
   EOF
   )"
   ```

5. **결과 확인** — `git status`로 커밋이 성공했는지 확인하고 결과를 짧게 보고한다.

## 규칙

- 사용자가 명시적으로 요청하지 않으면 **`git push`는 하지 않는다.**
- 기본 브랜치(main/master)에 직접 커밋하려는 상황이면, 브랜치를 새로 팔지 물어본다.
- 커밋 메시지에 작성자 서명(Co-Authored-By 등)은 사용자가 요청할 때만 추가한다.
- pre-commit 훅이 실패하면 멈추고 원인을 보고한다. 훅을 우회(`--no-verify`)하지 않는다.
