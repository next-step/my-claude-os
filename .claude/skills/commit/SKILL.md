# commit — 규칙을 지키는 커밋 & 푸시 스킬

이 프로젝트의 커밋 규칙을 자동으로 지키며 변경사항을 커밋하고 GitHub에 푸시한다.

## 반드시 지킬 커밋 규칙

1. **Conventional Commits 접두사 사용**: 제목은 `<type>: <설명>` 형식으로 작성한다.
   - 자주 쓰는 type: `feat`(기능), `fix`(버그 수정), `docs`(문서), `refactor`(리팩터링),
     `style`(포맷팅), `test`(테스트), `chore`(빌드/설정 등 잡일)
   - 변경 성격에 맞는 type을 고른다. 이 프로젝트는 문서 작업이 많으므로 대부분 `docs:`가 된다.
2. **`Co-Authored-By` 트레일러 제외**: 커밋 메시지에 Claude 공동작성자 트레일러를 절대 넣지 않는다.
   (전역 기본 동작과 반대이니 주의.)
3. **메시지는 한국어로**, 무엇을·왜를 간결하게. 기존 커밋 히스토리의 톤을 따른다.

## 실행 절차

### 1단계 — 현재 상태 파악
다음을 **병렬로** 실행해 무엇이 바뀌었는지 본다.
```bash
git status
git diff            # unstaged 변경
git diff --staged   # 이미 staged된 변경
git log --oneline -5 # 기존 메시지 톤 참고
```

### 2단계 — 스테이징
- 이미 staged된 변경만 커밋할지, 전체를 올릴지 애매하면 사용자에게 확인한다.
- 보통은 작업한 변경 전체를 `git add`로 스테이징한다. 단, 의도치 않은 파일
  (빌드 산출물, 비밀키, 임시파일 등)이 섞여 있지 않은지 status로 먼저 검토한다.

### 3단계 — 커밋 메시지 작성
- 위 "커밋 규칙"에 따라 제목을 정한다. 변경이 여러 갈래면 본문에 bullet로 요약한다.
- 변경 내용이 서로 무관한 여러 작업이면, **나눠서 커밋할지** 사용자에게 제안한다.

### 4단계 — 커밋
`Co-Authored-By`가 들어가지 않도록 `-m`으로 직접 메시지를 지정해 커밋한다.
```bash
git commit -m "docs: 예시 메시지"
```
> 주의: 절대 `Co-Authored-By: ...` 줄을 추가하지 않는다.

### 5단계 — 푸시
현재 브랜치를 `origin`에 푸시한다. `upstream`(원본 저장소)에는 푸시하지 않는다.
```bash
git push origin HEAD
```
- 현재 브랜치에 upstream 트래킹이 없으면 `git push -u origin HEAD`로 설정한다.
- 기본(main) 브랜치에 직접 커밋하려는 상황이면, 먼저 작업 브랜치를 만들지 사용자에게 확인한다.

### 6단계 — 결과 보고
커밋 해시와 푸시된 브랜치를 한 줄로 알려준다. 푸시 실패(인증/충돌 등) 시
출력을 그대로 공유하고 해결 방법을 제안한다.

## 하지 말 것
- `Co-Authored-By` 트레일러 추가 금지.
- 사용자가 요청하지 않았는데 `upstream`으로 푸시하거나 PR을 만들지 않는다.
- `git push --force` 등 파괴적 동작은 명시적 요청 없이는 하지 않는다.
