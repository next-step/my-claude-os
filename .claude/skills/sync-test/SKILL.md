# /sync-test 스킬 — 테스트 최신화 오케스트레이터

`.claude/` 의 **현재 실제 상태**를 스캔해서, 낡은 `tests/` 스위트가 실제 구현을
계속 검증하도록 갱신한다. 스킬·훅·데이터 파일을 추가/삭제한 뒤 실행하면,
새로 생긴 "테스트 안 되는 표면"을 찾아 테스트를 채우고, 사라진 대상에 대한
죽은 테스트를 걷어낸다.

## 사용법

```
/sync-test          ← 커버리지 빈틈을 찾아 테스트 갱신안을 만들고 보여준 뒤 적용
```

---

## 테스트 구조 (전제)

이 OS의 테스트는 3계층이다. 계층마다 "자동 적응 / 손유지" 정도가 다르다.

| 계층 | 파일 | 손으로 유지해야 하는 부분 | sync-test가 보는 것 |
|------|------|--------------------------|---------------------|
| L1 lint | `tests/lint.sh` | `check_json` 줄 (데이터 파일별 필수 키) | 새 `data/*.json` 이 검사 목록에 있나 |
| L2 unit | `tests/unit.test.js` | 케이스 테이블 (결정적 훅) | 테스트 없는 결정적 훅이 있나 |
| L3 smoke | `tests/smoke.sh` | 어떤 스킬을 end-to-end 도는가 | 스모크 안 도는 스킬이 있나 |

> L1의 **나머지**(frontmatter·링크 무결성·JS/셸 문법)는 glob 기반이라 새 스킬·훅이
> 생기면 **자동으로 검사 대상에 포함**된다. 그래서 sync-test가 굳이 손댈 필요 없다.
> 손유지 지점은 위 3곳뿐이다 — 거기만 정확히 겨냥한다.

---

## 실행 절차

> **설계 포인트 — 정본(source of truth)을 읽는다**
> 테스트는 한 번 짜두면 코드가 바뀌어도 그대로 남아 금세 현실과 어긋난다.
> 이 스킬은 테스트의 *옛 내용을 믿지 않고*, 항상 실제 파일시스템을 스캔해
> "지금 무엇을 테스트해야 하는가"를 다시 길어 올린다. ([[sync-readme]]가
> README를 실제 상태에 맞추는 것과 같은 철학.)
>
> **오케스트레이터 패턴 포인트**
> ① 사실 수집(Bash/Read)과 ② 테스트 작성(test-engineer 위임)과 ③ 검증(테스트 실행)을
> 분리한다. 작성한 컨텍스트가 스스로 합격을 선언하지 않고, 별도 실행으로 증거를 남긴다.

### Step 1: 테스트 가능 표면 스캔 (사실 수집)

지금 프로젝트에 무엇이 있는지 모은다.

```bash
echo "=== [A] 스킬 목록 ===" && \
for d in .claude/skills/*/; do
  b=$(basename "$d"); [ "$b" = "_shared" ] && continue
  [ -f "$d/SKILL.md" ] && echo "- $b"
done && \
echo "" && echo "=== [B] 훅 파일 (JS=결정적 단위테스트 후보 / SH=통합) ===" && \
ls -1 .claude/hooks/*.js .claude/hooks/*.sh 2>/dev/null && \
echo "" && echo "=== [C] 데이터 파일 + 소비처가 읽는 키 ===" && \
for j in .claude/data/*.json; do
  [ -e "$j" ] || continue
  echo "- $j  →  코드가 jq로 읽는 키:"
  grep -rhoE "jq -r '?\.[a-zA-Z_]+'?" .claude --include='*.sh' 2>/dev/null \
    | grep -oE '\.[a-zA-Z_]+' | sort -u | sed 's/^/    /'
done
```

> 데이터 파일은 **소비처(셸 스크립트)가 실제로 읽는 키**가 곧 "필수 키"다.
> 토큰 값 자체는 절대 읽지도 옮기지도 않는다 — 키 이름만 본다.

### Step 2: 현재 커버리지 스캔 + 빈틈 계산

테스트가 지금 **무엇을 검사하고 있는지**를 읽어, Step 1과 비교한다.

```bash
echo "=== L1: lint이 check_json 하는 데이터 파일 ===" && \
grep -oE 'check_json [^ ]+' tests/lint.sh && \
echo "" && echo "=== L2: unit.test.js가 다루는 훅 ===" && \
grep -oE '\.claude/hooks/[a-zA-Z-]+\.js|hooks", "[a-z-]+\.js"|"[a-z-]+\.js"' tests/unit.test.js | sort -u && \
echo "" && echo "=== L3: smoke.sh가 도는 스킬 ===" && \
grep -oE '/capture|/[a-z-]+' tests/smoke.sh | sort -u
```

이제 **빈틈(gap)** 을 계산한다. (모델이 Step 1 ↔ Step 2를 대조)

- **L1 빈틈**: Step 1[C]에 있는데 `check_json` 줄이 없는 `data/*.json` → 검사 누락
- **L2 빈틈**: Step 1[B]의 `*.js` 훅 중 `unit.test.js`가 안 다루는 것 → 단위테스트 누락
  - `*.sh` 훅(remind-cron, telegram-listener 등)은 부작용·외부의존이 커서 단위테스트가
    어렵다. **빈틈으로 보고만** 하고 자동 추가하지 않는다. (사용자 판단)
- **L3 빈틈**: Step 1[A]의 스킬 중 smoke가 안 도는 것 → 통합테스트 누락 (**보고만**, L3는 비쌈)
- **죽은 테스트**: 이미 사라진 스킬/훅/데이터를 아직 검사하는 테스트 → 제거 대상

### Step 3: 빈틈 보고 + 사용자 합의

계산한 빈틈을 표로 보여주고, **자동으로 채울 것 / 보고만 할 것**을 구분해 제시한다.

- **자동으로 채움 (결정적이라 안전)**:
  - L1 `check_json` 한 줄 추가/삭제
  - L2 새 결정적 JS 훅에 대한 케이스 테이블 또는 테스트 파일 추가
  - 죽은 테스트 제거
- **보고만 (사용자 결정)**:
  - L3 새 스킬 스모크 추가 (claude 실행 비용·Notion 부작용이 크므로)
  - `*.sh` 훅 통합테스트

빈틈이 하나도 없으면 "테스트가 현재 구현과 일치합니다 ✅" 출력 후 종료.

### Step 4: test-engineer 에이전트에 갱신 위임

"자동으로 채움" 항목을 oh-my-claudecode의 `test-engineer` 에이전트에 위임한다.
Agent 도구를 호출하며 아래를 전달한다.

- **목표**: Step 3에서 합의한 빈틈만 채우도록 `tests/` 갱신
- **입력 데이터**: Step 1~2 스캔 결과 + 채울 빈틈 목록
- **지켜야 할 제약**:
  - 기존 테스트의 **구조·말투·한국어 주석 톤을 유지** (전면 재작성 금지, 빈틈만 메움)
  - L2 새 훅 테스트는 **stdin→stdout 계약**을 케이스 테이블로 (기존 `detect-todo` 방식 그대로)
  - **L3를 새로 작성할 때의 철칙** (이 OS에서 실제로 데인 함정):
    1. 테스트용 입력에 `__TEST__` 같은 **의도 누설 마커 금지** — 똑똑한 LLM이
       "테스트구나" 하고 실제 부작용(저장)을 건너뛴다. 평범한 입력으로 위장한다.
    2. LLM이 입력을 **정규화**하므로(접미사 숫자 등 제거), 내가 넣은 고유 문자열이
       그대로 저장된다고 가정하지 않는다. **실행 전/후 상태 diff(id 집합 등)** 로 식별한다.
    3. 부작용을 만들었으면 반드시 **정리(아카이브/삭제)** 해서 실데이터 오염을 막는다.
    4. 자격증명·외부 CLI가 없으면 실패가 아니라 **SKIP**.
  - 비밀값(토큰·ID)은 테스트 코드·로그에 절대 쓰지 않는다

### Step 5: 검증 (별도 실행으로 증거 남기기)

작성한 컨텍스트가 스스로 합격을 선언하지 않는다. 결정적 계층을 실제로 돌려 확인한다.

```bash
./run-tests.sh l1 l2
```

- 통과하면: `git diff tests/` 로 변경분을 사용자에게 보여준다.
- 실패하면: 출력의 `✗` 항목을 test-engineer에 되돌려 **통과할 때까지 반복**한다.
- L3를 새로 추가했다면, 사용자에게 `./run-tests.sh l3` 수동 실행을 안내한다(비용·부작용 고지).

커밋은 사용자가 원할 때만 한다. (이 OS의 커밋은 사용자 결정 사항)

## 설계 노트 — 왜 이렇게 만들었나

- **AI 협업 학습 포인트**: 테스트 최신화는 세 가지 다른 일(빈틈 탐지 + 테스트 작성 +
  실행 검증)의 합성이다. 한 컨텍스트에서 다 하면 "내가 짠 테스트가 통과한다"는
  자기검증의 함정에 빠진다. 그래서 **작성(test-engineer)** 과 **검증(run-tests 실행)** 을
  다른 레인으로 쪼갰다.
- **빈틈만 겨냥하는 이유**: L1의 대부분은 glob 기반이라 자동 적응한다. 사람이 손대야
  하는 지점(`check_json`·L2 케이스·L3 스킬)만 정확히 좁혀야, 매번 테스트를 통째로
  다시 쓰지 않고 "추가분만 따라오게" 만들 수 있다.
- **L3는 신중하게**: 통합 스모크는 claude를 실제로 띄우고 외부(Notion)에 부작용을
  남긴다. 그래서 자동 확장하지 않고 **보고 + 사용자 합의** 후에만 추가한다.
  새 스킬을 추가할 때마다 `/sync-test` 한 번이면 결정적 커버리지가 따라오고,
  통합 커버리지는 의식적으로 결정하도록 한 것이 목적이다.
