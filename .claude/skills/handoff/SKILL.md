# Handoff — 개발 위임 표준 양식

기획자(product-planner)가 정한 틀을 개발자에게 넘길 때, 매번 빠짐없이 같은 형식으로 전달해
위임 품질을 일정하게 유지한다. 빠진 항목(완료 기준 없음·의존성 누락)이 나중의 재작업·오해를 만든다.

/ orchestrate 스킬의 2단계(backend·frontend 병렬 호출) 직전에 이 양식으로 작업을 찍어내면,
각 개발자가 "무엇을, 어떤 계약 위에서, 어디까지, 무엇이 끝난 뒤에" 하는지 명확해진다.

## 언제 쓰나
- 기획이 확정되어(OS.md 갱신 완료) 개발 작업을 개발자에게 넘기기 직전.
- 한 기능을 backend/frontend로 쪼개 병렬로 맡길 때 각자의 핸드오프를 따로 만들 때.

## 핸드오프 양식 (각 개발자당 1장)

```
## Handoff → {backend-developer | frontend-developer}
작업명: <한 줄>

### 1. 범위 (이번에 만들 것)
- 포함: <명확히 만들 것 목록>
- 제외: <지금 하지 않을 것 — M2+로 미루는 것 등. 과설계 방지>

### 2. 참조 계약 (단일 진실 출처)
- OS.md <해당 장/절>(예: 12.3 스키마, 12.5 API).
- 의존하는 타입/엔드포인트: <Job, JobDTO, /api/jobs ...>
- 계약을 바꿔야 할 것 같으면 → 임의 변경 금지. 기획자에게 올려 OS.md 먼저 갱신.

### 3. 완료 기준 (Definition of Done)
- <검증 가능한 조건. "X 화면이 Mock 데이터로 끝까지 돈다" 식으로 관찰 가능하게>
- 빠지면 안 되는 것: <빈 상태/에러/로딩, 폴백 처리 등 — 행복경로만 = 미완성>

### 4. 의존성 / 순서
- 선행: <이게 끝나야 시작 가능한 것. 예: backend 타입+Mock 공개>
- 병렬 가능: <동시에 진행해도 되는 것>
- 후행: <이 작업이 끝나야 가능한 다음 단계>

### 5. 보고 요청
- 만든 것 요약 + (있다면) 기획에 대한 제약/이견 → 기획자에게 되돌릴 수 있게.
```

## 절차
1. **현재 확정 계약을 확인**한다(OS.md 12장). 핸드오프의 "참조 계약"이 실제 OS.md와 일치해야 한다.
2. 넘길 작업을 backend 몫 / frontend 몫으로 나눈다(OS.md 12.7 작업 분배 기준).
3. 각 몫을 위 양식으로 채운다. **완료 기준과 의존성**을 특히 비워두지 않는다(가장 자주 누락되는 두 항목).
4. /orchestrate 사용 중이라면, 채운 핸드오프 내용을 각 개발자 호출 프롬프트의 본문으로 그대로 전달한다.

## 원칙
- **완료 기준은 관찰 가능하게.** "잘 만든다"가 아니라 "온보딩→피드→북마크가 Mock으로 끝까지 돈다"처럼 확인 가능한 문장으로.
- **제외 항목을 반드시 적는다.** 무엇을 '지금 안 하는지'를 명시해야 개발자가 M1 범위를 과설계하지 않는다.
- **계약은 OS.md가 진실.** 핸드오프가 OS.md와 어긋나면 OS.md를 먼저 맞춘다(기획자 권한).
