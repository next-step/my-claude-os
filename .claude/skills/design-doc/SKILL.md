# Design-doc 스킬 (설계 문서)

구현하기 전에 **"무엇을 왜 어떻게 고칠지"를 글로 합의한다.** 코드부터 짜면
방향이 틀려도 늦게 안다. 문서가 그 위험을 앞당겨 막는다.

## 원칙

- **최소 범위.** 티켓이 시킨 것만. 그 외 개선은 "범위 밖"에 기록만 한다.
- **대안을 적는다.** 후보가 둘 이상이면 택일 이유를 남긴다.
- **테스트를 설계에 포함.** 검증 방법 없는 수정안은 미완성이다.

## 절차

1. `investigate`·`analyze` 노트를 입력으로 받는다.
2. 아래 양식으로 문서를 작성한다.
3. `docs/design/<티켓번호>.md`로 저장한다 (폴더 없으면 만든다).
4. **사용자에게 보여주고 합의를 받는다. OK 전엔 구현으로 넘어가지 않는다.**

## 설계 문서 양식

```markdown
# 수정 설계 — <티켓번호> <제목>

## 문제
- (무엇이 잘못/필요한가)

## 원인 / 배경
- (커밋 히스토리에서 읽은, 왜 지금처럼 됐나)

## 수정 방안
- (택한 방안 — 파일:라인 단위로 무엇을 바꾸나)
- 대안: (검토했으나 안 택한 것 + 이유)

## 영향 / 비기능
- 영향 범위: (깨질 수 있는 곳)
- 성능 등: (비기능 영향 / 없으면 "해당 없음")

## 테스트 계획
- (회귀/신규 테스트로 무엇을 어떻게 검증)

## 범위 밖 (이번엔 안 건드림)
- (눈에 띈 개선거리)
```

> 합의되면 maintenance-mode가 구현·QA 단계로 넘긴다.
