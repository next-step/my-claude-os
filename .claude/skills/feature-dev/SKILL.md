# 기능 개발 파이프라인 (feature-dev)

기능 하나를 **4단계**로 개발하는 오케스트레이터 스킬이다. 각 단계는 전용 서브에이전트(`.claude/agents/`)에게 맡기고, 이 스킬은 그 사이를 잇고 **검증 루프**를 돌린다.

```
① code-analyzer → ② code-writer → ③ code-reviewer ⇄ code-writer (통과까지 루프) → ④ doc-writer
```

> 💡 **왜 이렇게 나누나?** 한 에이전트가 전부 하면 맥락이 뒤섞이고, 자기가 짠 코드를 자기가 리뷰하는 "확증 편향"이 생긴다. 단계를 나누고 **리뷰어를 구현자와 분리**하면, 독립적인 시선으로 품질을 검증할 수 있다. 이게 이 파이프라인의 핵심이다.

## 사용 시점
- "~기능 만들어줘/개발해줘/구현해줘" 같은 새 기능·변경 요청을 받았을 때

## 절차

### ① 코드베이스 파악
- `code-analyzer` 서브에이전트를 호출한다. 사용자 요구사항을 함께 전달한다.
- 결과: 관련 파일 지도, 프로젝트 관습, 변경 진입점, 주의점.

> 💡 **왜 먼저?** 어디를 어떻게 고칠지는 코드베이스의 기존 규칙에 달렸다. 파악 없이 짜면 프로젝트와 안 맞는 코드가 나온다.

### ② 분석·개발·테스트
- `code-writer` 서브에이전트를 호출한다. **①의 지도 + 사용자 요구사항**을 함께 전달한다.
- code-writer가 요구 분석 → 테스트 작성 → 구현 → 테스트 통과까지 진행한다.
- 결과: 변경 파일 목록, 테스트 결과(green), 구현 노트.

### ③ 검증 루프 (핵심)
아래를 **판정이 "통과"가 될 때까지, 최대 3회** 반복한다.

1. `code-reviewer` 서브에이전트를 호출한다. 변경된 코드(git diff)를 리뷰하게 한다.
2. code-reviewer의 **판정**을 본다.
   - **통과** → 루프 종료, ④로 간다.
   - **수정필요** → code-reviewer의 🔴 지적 사항을 그대로 `code-writer` 서브에이전트에 넘겨 고치게 한다. 그 후 1번으로 돌아가 다시 리뷰.
3. 3회를 돌아도 "통과"가 안 나오면, 남은 지적을 사용자에게 요약해 보고하고 어떻게 할지 묻는다. (무한 루프 금지)

> 💡 **왜 루프인가?** 한 번 고친다고 다 해결되지 않는다. "리뷰 → 수정 → 재리뷰"를 반복해야 품질이 수렴한다. 사람이 매번 개입하지 않아도 되도록, 반복 상한(3회)과 종료 조건(판정=통과)을 명확히 둔다.
>
> 💡 **왜 code-reviewer가 직접 안 고치나?** 지적하는 사람과 고치는 사람을 분리해야 리뷰가 객관적으로 유지된다. 그래서 수정은 항상 code-writer가 한다.

### ④ 문서화
- 검증을 통과한 뒤 `doc-writer` 서브에이전트를 호출한다.
- 변경 내용을 프로젝트 스타일에 맞춰 문서(README/사용법/CHANGELOG 등)로 남긴다.

### ⑤ 최종 보고
사용자에게 아래를 정리해 보고한다.
- 무엇을 만들었는지(요구사항 대비)
- 바뀐 파일과 테스트 결과
- 검증 루프 결과(몇 회 만에 통과했는지, 남은 🟡/🟢 있으면 언급)
- 문서화한 내용
- 커밋이 필요하면 `git-commit` 스킬을 안내한다.

## 오케스트레이션 팁
- 각 서브에이전트는 **한 번에 하나씩** 순서대로 호출한다. 앞 단계의 출력을 다음 단계 입력으로 명시적으로 넘긴다.
- 서브에이전트는 대화 맥락을 공유하지 않으므로, 필요한 정보(요구사항, 지도, 지적 사항)를 프롬프트에 담아 전달해야 한다.
- ①에서 요구사항이 너무 모호하면, 개발을 시작하기 전에 사용자에게 먼저 확인한다.

## 주의사항
- **검증 루프는 반드시 종료 조건이 있어야 한다.** 판정=통과 또는 3회 도달. 그 외에는 멈추지 않는다.
- code-reviewer가 "통과"를 줬는데 테스트가 실패 상태면, 그건 모순이니 다시 확인한다.
- 범위를 벗어난 큰 변경이 필요해지면 멈추고 사용자에게 알린다.
