# Auto Commit

코드 변경사항을 분석해서 의미 있는 커밋 메시지를 한국어로 자동 생성하고 커밋까지 수행합니다.

인자로 브랜치명을 받으면 해당 브랜치에, 없으면 현재 브랜치에 커밋합니다.

---

## 1단계: 브랜치 확인 및 변경사항 파악

먼저 커밋 대상 브랜치를 결정합니다:

```bash
# 현재 브랜치 확인
git branch --show-current
```

- 인자로 브랜치명이 주어진 경우: `git checkout <브랜치명>` 으로 전환 후 진행
- 인자가 없는 경우: 현재 브랜치 그대로 진행

그 다음 변경사항을 파악합니다:

```bash
git status
git diff HEAD
git diff --cached
```

staged 파일과 unstaged 파일을 모두 확인합니다. 변경사항이 없으면 사용자에게 알리고 종료합니다.

---

## 2단계: 변경사항 분석

diff 내용을 읽고 다음을 파악합니다:

- 어떤 파일이 추가 / 수정 / 삭제되었는가
- 코드에서 실제로 무엇이 바뀌었는가 (함수 추가, 버그 수정, 설정 변경 등)
- 변경의 **목적**이 무엇인가 — 이것이 커밋 메시지의 핵심이 됩니다

---

## 3단계: 커밋 메시지 생성

[Conventional Commits](https://www.conventionalcommits.org/) 형식으로 작성합니다.

```
<type>(<scope>): <subject>

<body>  ← 선택사항. 왜 변경했는지 설명할 때만 추가
```

### 타입 선택 기준

| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서(README, 주석 등) 변경 |
| `style` | 기능 변화 없는 포맷 변경 (들여쓰기, 세미콜론 등) |
| `refactor` | 기능 변화 없는 코드 구조 개선 |
| `test` | 테스트 코드 추가 / 수정 |
| `chore` | 빌드 설정, 패키지, 기타 유지보수 |

### 좋은 subject 작성법

- 50자 이내로 간결하게
- 명령형 동사로 시작 — "추가", "수정", "제거", "개선" 등
- "무엇을" 했는지보다 **"왜" 했는지**를 담을 것
- 변경된 파일명 나열 금지 — 의도와 맥락을 담을 것
- 항상 한국어로 작성

### 예시

변경사항: `Button.tsx`에 disabled 상태 추가, `Button.css`에 회색 스타일 추가

```
feat(Button): disabled 상태 및 스타일 추가
```

---

## 4단계: 사용자에게 확인 요청

생성한 메시지를 다음 형식으로 보여줍니다:

```
생성된 커밋 메시지:
──────────────────────────────────────
feat: 사용자 로그인 기능 추가

JWT 기반 인증으로 보안을 강화하고
세션 유지 기능을 구현했습니다.
──────────────────────────────────────

이 메시지로 커밋할까요?
수정하려면 원하는 메시지를 입력해 주세요.
```

사용자가 수정을 요청하면 수정된 메시지로 다시 보여주고 재확인합니다.

---

## 5단계: 커밋 실행

사용자가 확인하면 다음 순서로 실행합니다:

```bash
# 1. 변경사항 스테이징 (이미 staged된 파일이 있으면 그것만 사용)
git add -A

# 2. 커밋
git commit -m "$(cat <<'EOF'
<확인된 커밋 메시지>
EOF
)"

# 3. 최종 상태 확인
git status
```

커밋 완료 후 어떤 파일이 포함되었는지 간략히 알려줍니다.

---

## 주의사항

- `.env`, 비밀키, 개인정보가 포함된 파일이 감지되면 **반드시 경고**하고 사용자 확인 후 진행
- 사용자의 명시적 승인 없이 자동으로 커밋하지 말 것
- `--no-verify` 등 훅 우회 플래그는 사용자가 명시적으로 요청할 때만 사용
