스킬 ↔ 서브에이전트 지도

어느 스킬이 어떤 서브에이전트를 — 왜 — 띄우는가. 보라색 = 내가 만든 커스텀(공유), 파란색 = 플랫폼 빌트인.

/feature런처
서브에이전트 안 띄움 — 다른 스킬을 Skill 도구로 호출만
/prd① 요구사항·PRD
general-purpose · PRD 작성
이유: 컨텍스트 격리
reviewer공유 · PRD 검토
이유: 독립(셀프채점 방지)
/plan② 설계
Explore · 재활용 조사(병렬 N개)
이유: 격리 + 병렬
Plan · 설계
이유: 격리 + 전문화
/impl③ 구현(디스패처)
general-purpose · 구현(코드+테스트)
이유: 격리(파일 많이 씀)
+ /verify 는 Skill 호출(서브에이전트 아님)
/verify검증 일꾼
reviewer공유 · (full) 기준↔코드 검증
이유: 독립(적대적 판정)
light 모드는 테스트만(Bash) — 서브에이전트 없음
🔗 공유 서브에이전트 = reviewer — /prd(요구사항↔PRD)와 /verify(기준↔코드)가 같은 reviewer를 호출한다. 행동(적대적 검토)은 .claude/agents/reviewer.md 한 곳에, 재료(기준·대상)만 각 스킬이 넘김.
※ general-purpose 도 /prd·/impl 둘이 쓰지만, 이건 "범용 빌트인"이라 의미 있는 공유 역할은 reviewer 쪽.
빌트인 (general-purpose · Explore · Plan) 커스텀·공유 (reviewer)
위임 이유 4가지 — ① 컨텍스트 격리(결론만 받기) · ② 독립(셀프채점 방지) · ③ 병렬 · ④ 전문화·시야 제한. 없으면 인라인으로(위임도 토큰 비용이니까).
