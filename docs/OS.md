# 내 OS 설계: 주식/투자 OS

> 4주 동안 만들 "나만의 클로드 코드 OS" 설계 문서. 진행하며 계속 업데이트하는 살아있는 문서다.
> 배경 지식은 [onboarding.md](./onboarding.md) 참고.

## 주제 & 동기

- **주제**: 주식/투자 OS
- **최종 목표**: 알아서 매매까지 해주는 **자동매매 에이전트**
- **동기**: 투자 판단과 실행을 AI에게 위임해, 내 투자 성향에 맞게 자동으로 굴러가는 체계를 갖고 싶다.

## 핵심 원칙: 동작하는 것부터, 신뢰는 단계적으로

자동매매는 **종착지이지 출발점이 아니다.** 온보딩의 권장사항("완벽하지 않아도 동작하는 OS를 먼저, 이후 지속 업그레이드")과 현실적 위험(실제 돈·실매매 API·규제)을 고려해, **사람이 승인하는 조언가(1단계)부터** 시작해 신뢰를 쌓으며 자동화 수준을 올린다.

## 단계별 로드맵

**이번 4주 강의의 범위는 1단계 완성**이다. 2·3단계는 강의가 끝난 뒤 혼자 얹어볼 **미래 로드맵**으로만 남겨둔다.

| 단계 | 사람의 역할 | AI의 역할 | 위험 | 상태 |
|------|------------|----------|------|------|
| **1단계 (MVP)** | 투자 성향 제공 + **매매 최종 승인** | 분석·조언·주문서 초안 작성 | 낮음 (모의/수동) | 🟢 4주 강의 목표 |
| **2단계** | 전략·한도 설정, 예외만 개입 | 조건 충족 시 주문 제안, 승인 후 실행 | 중간 | ⚪ 강의 후 미래 |
| **3단계 (목표)** | 가드레일·리뷰만 | 탐색→판단→**자동 매매**→결과 학습 | 높음 | ⚪ 강의 후 미래 |

# 1단계 MVP — 전체 워크플로우

1단계의 핵심은 **세 개의 스킬이 하나의 루프**를 이루는 것이다. 온보딩 공통 골격 `의도 공유 → 실행 → 피드백 → 반복`을 그대로 스킬에 매핑했다.

```
[사람]  "종목 추천해줘"                                    ← 의도 공유
   ↓
[recommend-stocks]  코스피 전체 → 필터링 → 후보 ~10개      ┐
   ↓                                                       │ 실행
[analyze-company]   후보 10개를 각각 정밀 분석             ┘
   ↓                  (진입가 / 기대치 / 손절가 산출)
[사람]  추천 리포트 검토 후 매매 승인/거절 (실주문은 사람)  ← 의사결정
   ↓
[기록]  추천 스킬·분석 스킬이 각자 근거를 회고용으로 저장
   ↓
[회고 스킬]  주 1회, 그동안의 추천·분석이 실제로 어찌 됐는지 대조 → 개선
   └──────────────────────── ↻ 반복 (기준 튜닝) ────────────────────────┘
```

## 스킬 구성과 책임 분담

| 스킬 | 트리거 | 역할 | 기록 책임 |
|------|--------|------|-----------|
| `recommend-stocks` | "종목 추천해줘" | 코스피 전체에서 안정성 우선으로 ~10개로 좁힘 | **선정 결정** (필터 기준·선택/탈락 종목) |
| `analyze-company` | 추천 스킬이 호출 **또는** 사용자가 직접 | 회사 1개 정밀 분석 + 매매 가격대 산출 | **분석 1건** (분석 기준·진입/기대/손절) |
| `회고 스킬` (예정) | 주 1회, 사용자 | 그동안의 추천·분석 결과를 실제와 대조해 개선 | (기록을 읽어 평가만 append) |

**핵심 설계 결정 — 기록 책임을 둘로 쪼갠다.** *왜:* 진입/목표/손절 같은 분석 수치는 `analyze-company`가 산출하므로 그 기록도 거기서 남겨야 책임이 한 곳에 모인다. `recommend-stocks`는 "왜 이 10개를 골랐나(필터·탈락 표본)"만 책임지고, 개별 분석 수치는 `analyze-company` 기록을 참조한다. → 중복 저장이 없고, 단독 분석도 똑같이 기록된다.

## analyze-company 단독 사용

`analyze-company`는 추천 루프의 부품이면서 **그 자체로도 독립 스킬**이다.
예: 이번 추천에 삼성전자가 안 들어왔는데 궁금하면 → "삼성전자 분석해줘"로 분석만 돌린다. 이때도 분석 기록은 동일하게 남아 회고 대상이 된다.
*왜 이렇게:* 추천(스크리닝)과 분석(개별 심층)은 관심사가 다르다. 분석을 독립시켜 두면 추천 밖에서도 재사용되고, 추천 스킬은 "분석을 호출하는 조립자"로 단순해진다.

# 스킬 1: `recommend-stocks` (종목 추천)

코스피 ~1000개에서 **싼 필터로 먼저 줄이고**, 살아남은 소수만 **무겁게 정밀 분석**하는 깔때기.

```
코스피 전체(~1000)
  ↓ 1차 필터  screen_kospi.py (네이버 시총 표: 시총·거래대금·PER·ROE)   — 싼 하드컷
 ~50
  ↓ 2차 필터  score_stocks.py (개별 재무: 부채비율·흑자추세) → 안정성 점수로 순위
 ~10
  ↓ 정밀 분석  analyze-company 를 10개 각각에 실행
최종 추천 = 종목별 [진입가 · 목표(기대치) · 손절가 · 안정성 점수],  안정성 우선 정렬
```

- **왜 2겹으로 거르나:** 1000개 전부의 개별 페이지를 긁는 건 느리고 과하다. 싼 일괄 컬럼으로 먼저 줄이고(1차), 줄어든 것만 깊게 본다(2차).
- **왜 무거운 `analyze-company`는 마지막에:** 종목당 서브에이전트 3개가 웹검색을 돈다. 1000개에 돌리면 불가능(시간·비용). ~10개로 좁힌 뒤에만 실행.
- **안정성 우선 정렬:** 상승 기대치가 높다고 위로 올리지 않는다. "A: 상승15%·불안정 < B: 상승8%·안정"이면 B를 더 추천. → 안정성 점수 = 정렬 기준, 상승 기대치 = 산출물(동점 보조).

## 필터 기준

**1차 하드컷** (`screen_kospi.py`): 시총 ≥ 5,000억 · 일 거래대금 ≥ 10억 · 0 < PER ≤ 25 · ROE ≥ 5% · 우선주·스팩 제외.
**2차 안정성 스코어링** (`score_stocks.py`, 0~100점): 부채비율 ≤ 150% & 최근 영업이익 흑자(하드컷) 후 — 재무건전성 30 · 이익안정성 25 · 주가변동성 15 · 배당 10 · ROE 10 · PER 10으로 점수화해 상위 ~10개. (안정성 계열 80 : 성장/밸류 20 = 안정성 우선.) 변동성은 `ohlcv.py` 재사용.

> 임계값·배점은 **튜닝 대상**이다. 회고로 "필터가 좋은 종목을 놓쳤나"를 보고 조정한다. 변경 시 스크립트의 상수와 이 문서를 함께 고친다.

## 데이터 소스 결정 (정형)

- **1·2차 필터 = 네이버 금융 스크래핑.** *왜:* 네이버는 FnGuide 재무까지 보여줘 **시세·시총·PER·ROE + 요약 재무(부채비율·흑자추세)** 를 한 곳에서 제공. 키 불필요, 당장 동작.
- **pykrx 는 탈락.** *왜:* 시장 전체 일괄 조회 함수가 모두 `data.krx.co.kr`(403/`LOGOUT`)에 막혀 빈 데이터로 크래시(2026-06-30 직접 테스트 확인). 정작 1차 필터에 필요한 그 기능이 막혀 부적합.
- **DART 는 도입 보류 → 정밀분석 단계로.** *왜:* 필터는 네이버로 충분. DART의 강점(공시 "사건"·수치 공신력)은 **한 종목을 깊게 볼 때** 판단을 바꾸므로, `analyze-company`에 서브에이전트를 붙이는 방식으로 후행 도입.

## 기록 (회고용 — 선정 결정 스냅샷)

`save_run.py`가 이번 실행의 **입력(필터 기준·단계별 개수·탈락 표본)과 출력(추천 종목 목록·각 종목이 가리키는 분석 기록)**을 `data/recommendations/YYYY-MM-DD-run.md`에 append-only로 저장한다. 같은 날 재실행은 `-2`,`-3`…으로 과거를 덮지 않는다.
- **구현됨:** 진입/목표/손절은 여기 복제하지 않고 `picks[].analysis`에 분석 기록 ref만 담는다. 본문 표를 그릴 때 그 분석 파일의 frontmatter를 **역참조**해 진입/목표/손절을 끌어온다(기록 없으면 "참조불가"). → frontmatter엔 숫자 중복이 없어 진실원천은 분석 기록 한 곳.

# 스킬 2: `analyze-company` (회사 분석)

회사 이름 하나를 받아, 여러 축을 모아 분석 리포트와 매매 가격대를 만든다.
**정성 정보는 서브 에이전트 병렬 웹검색으로, 정밀 수치는 결정론적 스크립트로, 종합은 메인이** 한다. (축을 늘릴 때 정성은 서브에이전트, 정량은 스크립트를 하나씩 더 붙이면 된다.)

## 분석에 쓰는 데이터 (정성 3축 + 정밀 정량 3종)

**정성 — 서브 에이전트 웹검색** (내러티브·해석용)

| 축 | 서브 에이전트 | 수집 내용 |
|----|---------------|-----------|
| 기술적 | `stock-trend-researcher` | 최근 1년 주가 흐름 (지지/저항·컨센서스는 색깔로만 — 정밀값은 스크립트가 확정) |
| 심리 | `company-news-researcher` | 회사 관련 최근 뉴스 |
| 거시 | `market-news-researcher` | 대한민국 경제(거시) 뉴스 |

**정밀 정량 — 결정론적 스크립트** (네이버 직접 파싱, 진입/목표/손절의 근거 숫자)

| 스크립트 | 산출 | 쓰임 |
|----------|------|------|
| `quote.py` | 현재가·52주 고저·거래량 | 진입/손절 **앵커** |
| `ohlcv.py` | 지지/저항(터치 강도)·변동성(연율·ATR) | 진입=지지, 손절=지지하단/변동성 |
| `fundamentals.py` | 재무(매출·영업이익·ROE·부채·PER/PBR·배당) + 컨센서스 목표가 | 고/저평가 해석, 목표가 근거 |

- **왜 둘로 나누나:** 웹검색은 출처마다 값이 달라 정밀 수치엔 위험(기아 검증에서 현재가 24% 오차 → 진입가가 현재가보다 높게 나옴). **정성=웹검색, 정밀 수치=스크립트**로 못 박았다. 스크립트는 네이버만 써서 API 키 불필요.

## 매매 가격대 산출 (진입 / 목표 / 손절)

근거 숫자는 스크립트 값. 앵커는 `quote.py` 현재가. 재료가 없으면 "산출 불가"로 적는다(지어내지 않음).
- **진입가**: 현재가 + `ohlcv.py` 지지선. 현재가가 지지선 부근이면 현재가, 한참 위면 가까운 지지선까지 기다리는 가격. (진입가 > 현재가면 오류 → 재검토.)
- **손절가**: 진입가 아래 −5~8%. `ohlcv.py` 가까운 지지선 하단/변동성(atr_pct)으로 폭 결정.
- **목표(기대치)**: `fundamentals.py` 컨센서스 목표주가 있으면 그 기준, 없으면 `ohlcv.py` 저항선으로 보수적으로. 현재가 대비 기대수익률(%) 함께.

## 기록 (회고용 — 분석 1건 스냅샷)

분석을 마칠 때마다 **무엇을·어떤 기준으로 분석했는지 + 진입/목표/손절 + 근거**를 회고용으로 저장한다(단독 실행이든 추천 안에서 돌든 동일). 회고 스킬이 나중에 현재가와 대조해 status를 평가한다.
- **구현됨:** `analyze-company/scripts/save_analysis.py` — `data/analyses/YYYY-MM-DD-회사.md`에 frontmatter(정량, 회고가 status 갱신) + 본문(근거)으로 append-only 저장. 출력으로 `ref`(상대경로)를 돌려줘 추천 기록이 참조하게 한다. 진입/목표/손절·status의 **단일 진실원천**이 이 파일이다.

# 스킬 3: 회고 스킬 (예정)

**주 1회**, 사용자가 직접 돌려 그동안의 추천·분석이 실제로 어떻게 흘러갔는지 대조하고 기준을 개선한다. 루프의 "피드백" 단계.

- **읽는 것**: `recommend-stocks` 선정 기록 + `analyze-company` 분석 기록.
- **하는 것**:
  1. 각 분석의 진입/목표/손절을 **현재가와 대조** → status 판정(`hit_target | stopped | watching`).
  2. **필터 기준이 타당했나** — 탈락시킨 종목 중 크게 오른 게 있나? 고른 종목이 실제로 안정적이었나?
  3. 결과를 바탕으로 **임계값·배점·분석 축을 조정**(다음 추천에 반영).
- **기록 원칙**: 과거 추천·분석은 **수정 금지(append-only)**. 회고는 기존 기록에 "평가"만 덧붙인다. *왜:* 그때의 판단을 박제해야 "기준이 틀렸는지"를 정직하게 복기할 수 있다.

# 가드레일 (안전 원칙)

- 1단계에서는 **실제 주문을 AI가 직접 실행하지 않는다.** (사람 승인 필수)
- 손실 한도·종목당 비중 등 **한도를 명시**하고 그 안에서만 조언.
- 모든 매매 조언에 **근거를 남겨** 나중에 복기 가능하게 한다.
- **출처 없는 수치는 싣지 않는다.** 못 구한 값은 "확인 불가"로 표기.
- 이 단계의 스킬은 **분석·조언까지만.** (실주문은 사람 몫)

# 확장 후보 데이터 (우선순위순, 강의 후/필요 시 추가)

투자 분석의 고전적 5축은 **기술적 / 펀더멘털 / 수급 / 심리 / 거시**. 현재 `analyze-company`는 기술적·심리·거시를 다루고, **펀더멘털·수급**이 비어 있다. 채울 후보(서브에이전트 추가 방식):

| 순위 | 데이터 | 왜 필요한가 |
|------|--------|------------|
| 1 | **재무제표/펀더멘털** (매출·영업이익·ROE, PER·PBR·배당) | 주가는 "시장의 평가", 재무는 "그럴 만한가". 대조해야 고/저평가 판단이 선다. |
| 2 | **공시 (DART)** (실적·증자·자사주·배당·내부자거래) | 무료+구조화+공신력 갖춘 1차 자료. 뉴스보다 신뢰도 높음. |
| 3 | **수급** (외국인/기관/개인 매매) | 한국 단기 주가의 핵심 동인. |
| 4 | **동종업계/경쟁사 비교** | 단독 PER·성장률은 의미가 약함. 업계 평균이 기준점. |
| 5 | (선택) **컨센서스/목표주가**, **구체 거시 수치**(환율·금리·유가) | 외부 시각·정량 거시 보강. |

> 선택 기준: **"이 데이터가 매수/매도/보류 판단을 바꾸는가?"** — 아니면 넣지 않는다.
